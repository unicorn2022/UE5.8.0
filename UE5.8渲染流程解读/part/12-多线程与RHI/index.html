多线程与RHI

GameThread / RenderThread / RHIThread 三线程模型、TaskGraph 调度、RHI 命令队列与平台抽象。

GameThread RenderThread RHIThread TaskGraph FRHICommandList AsyncCompute

1. 管线中的位置

UE5.8 的多线程渲染架构将游戏逻辑、渲染命令生成、GPU 命令提交三个核心环节分配到独立线程，通过 TaskGraph 系统实现细粒度的任务调度和并行执行。

渲染管线的三条关键线程构成了一个线程流水线：GameThread 负责主逻辑和可见性提交，RenderThread 负责 RDG Pass 的执行和 DrawCall 生成，RHIThread 负责将 RHI 命令翻译为 D3D12/Vulkan API 调用并提交到 GPU 驱动。此外，AsyncCompute 队列可以独立于 Graphics 管线并行执行 Compute Shader 工作（如 Nanite 裁剪、Lumen 光照评估），进一步利用 GPU 的并行能力。

阶段线程交互说明
InitViews (L1902-2465) GameThread → RenderThread 可见性计算在 GameThread 提交，部分 GPU Scene 更新（InstanceCulling、Primitive 变换）在 RenderThread 异步执行。GameThread 通过 ENQUEUE_RENDER_COMMAND 将更新请求推送到 RenderThread 的命令队列。
FinishGatherDynamicMeshElements (L2241) RenderThread 动态网格元素收集完成，为每个可见 Section 生成 FMeshDrawCommand（Shader 绑定 + PSO 状态 + SortKey）。此阶段是 RenderThread 上 CPU 开销最大的环节之一。
PrePass + Nanite (L2476-2534) RenderThread + RHIThread 并行 RenderThread 注册深度预通道和 Nanite 裁剪 Pass，RHIThread 异步执行上一帧提交的 GPU 命令。两条线程在此阶段形成流水线重叠。
BasePass (L3098) RenderThread → RHIThread G-Buffer 渲染命令通过 RDG 注册后，随 GraphBuilder.Execute() 一次性提交到 RHI 命令队列，RHIThread 开始消费。
GraphBuilder.Execute() (L4423) RenderThread → RHIThread RDG 编译所有注册的 Pass 为 RHI 命令列表，批量提交到 RHIThread。RHIThread 逐条翻译为平台 API 调用（D3D12 ExecuteCommandLists / Vulkan vkQueueSubmit）。
关键理解：线程流水线化是 UE 渲染性能的核心设计。RenderThread 处理第 N 帧的 Pass 注册时，RHIThread 可能仍在执行第 N-1 帧的 GPU 命令提交。这种帧间重叠使得 CPU 端渲染开销不会直接累加到帧时间上，理想情况下吞吐量可接近单线程的 3 倍。但这也引入了 2-3 帧的输入端延迟（Input Latency），需要在响应性和吞吐量之间权衡。

2. 三线程模型

GameThread 负责游戏逻辑和渲染状态设置，RenderThread 负责渲染命令的生成和 RDG Pass 调度，RHIThread 负责将 RHI 命令提交到 GPU 驱动。三线程以异步流水线方式运行，减少 GPU 空闲等待。

2.1 线程职责边界

线程职责关键入口线程类型
GameThread 游戏逻辑、物理模拟、动画更新、可见性计算提交。通过 ENQUEUE_RENDER_COMMAND 宏向 RenderThread 发送命令，通过 FlushRenderingCommands 强制同步等待。 ENQUEUE_RENDER_COMMAND
FlushRenderingCommands		 游戏主线程（固定）
RenderThread RDG 构建与执行、DrawCall 提交、Shader 参数绑定。执行 FDeferredShadingSceneRenderer::Render() 的完整九阶段流程，是渲染管线的核心调度线程。 FDeferredShadingSceneRenderer::Render() Named Thread
ENamedThreads::ActualRenderingThread
RHIThread RHI 命令到 D3D12/Vulkan API 的翻译与提交。消费 RenderThread 生成的 FRHICommandList，调用平台底层 API（如 ID3D12CommandQueue::ExecuteCommandLists）。 FRHICommandListExecutor::Execute() Named Thread

2.2 线程流水线时序

流水线解读：理想情况下三条线程形成 3 级流水线——GameThread 处理第 N+1 帧的 InitViews 时，RenderThread 正在执行第 N 帧的 RDG Pass，RHIThread 还在提交第 N-1 帧的 GPU 命令。同步点 FRenderCommandFence（GameThread 等待 RenderThread 完成）和 FRHICommandListFence（RenderThread 等待 RHIThread 完成）仅在需要读取渲染结果时才触发，正常流水线状态下不产生阻塞。

2.3 Latency 与并行度

三线程流水线的核心权衡在于吞吐量与延迟：

  • 理想吞吐量：三线程完全并行时，CPU 端渲染吞吐量接近单线程的 3x。GameThread 提交完 InitViews 后即可开始下一帧的逻辑，无需等待 RenderThread 或 GPU 完成。
  • 输入端延迟：从 GameThread 提交渲染命令到 GPU 实际显示画面，理想情况下有 2-3 帧延迟。这是因为 RenderThread 处理当前帧时，RHIThread 还在提交上一帧，GPU 在更后面执行。对于 VR 等延迟敏感场景，可通过 r.RHIThread.Enable=0 关闭 RHIThread 来减少一帧延迟。
  • 实际瓶颈：InitViews 阶段占 RenderThread 时间的 40-60%，是最大的串行瓶颈。InitViews 中的可见性计算虽然部分并行化（通过 TaskGraph 拆分为多个并行子任务），但 FinishGatherDynamicMeshElements 阶段需要等待所有并行任务完成后才能继续，形成同步点。
  • 关闭 RHIThread 的代价：r.RHIThread.Enable=0 后 RenderThread 兼任 RHI 命令提交，延迟减少约一帧，但 RenderThread 每帧执行时间增加 15-30%（因为需要直接调用 D3D12/Vulkan API），总体吞吐量下降。

3. TaskGraph 调度

TaskGraph 是 UE5 的通用任务图调度系统，渲染模块大量使用 NamedThread 和 TaskGraph Task 来组织渲染工作，支持前置任务（Prereqs）和异步回调。

3.1 FTaskGraphInterface 核心概念

TaskGraph 是 UE 底层的任务调度系统，所有线程（包括 GameThread、RenderThread、RHIThread）都由 TaskGraph 管理。每个线程维护一个 FTaskThreadBase 实例，拥有自己的任务队列（Lock-free MPSC 队列）。任务以 TGraphTask 模板的形式提交，支持前置任务依赖链和完成事件回调。

任务的典型递交方式：

// TaskGraph 任务的标准创建和派发模式 FGraphEventRef PrerequisiteEvent = TGraphTask<FMyPreTask>::CreateTask().ConstructAndDispatchWhenReady(Args...); // 带前置依赖的主任务——只有 PrerequisiteEvent 完成后才开始执行 FGraphEventRef MainTaskEvent = TGraphTask<FMyMainTask>::CreateTask(PrerequisiteEvent) .ConstructAndDispatchWhenReady(Args...); // 等待任务完成（阻塞当前线程） FTaskGraphInterface::WaitUntilTasksComplete(MainTaskEvent);

TaskGraph 中的线程分为两类：

  • Named Thread（命名线程）：具有固定名称和专用任务队列的线程，如 ENamedThreads::ActualRenderingThread（RenderThread）、ENamedThreads::RHIThread、ENamedThreads::AudioThread 等。命名线程不会被销毁，优先级固定，它们的任务队列只能由对应线程消费。
  • Worker Thread（工作线程）：由 ENamedThreads::AnyThread 标记的线程池线程。没有固定名称和专用队列，可以动态创建和销毁。TaskGraph 将 AnyThread 任务分发到全局线程池，由空闲 Worker 执行。适合 CPU 密集的并行计算（如可见性计算、遮挡剔除）。

3.3 渲染线程绑定

核心枚举
ENamedThreads::ActualRenderingThread
渲染专用命名线程，固定优先级（略低于 GameThread），拥有专用任务队列。不是 Worker Thread——有固定优先级和专用队列，不会被线程池回收。GameThread 通过 ENQUEUE_RENDER_COMMAND 将任务推送到此线程的队列，TaskGraph 在每帧开始时批量处理。RenderThread 的任务优先级低于 GameThread，防止渲染工作阻塞游戏逻辑。
核心枚举
ENamedThreads::RHIThread
RHI 命令执行线程，优先级低于 RenderThread。消费 RenderThread 产生的 FRHICommandList 队列，逐条翻译为平台 API 调用。RHIThread 的任务模式与 RenderThread 不同——它在 FRHICommandListExecutor::Execute() 中循环消费命令，而不是等待 TaskGraph 分发离散任务。关闭 RHIThread（r.RHIThread.Enable=0）后，RenderThread 在提交 Pass 时直接调用平台 API。
同步机制
FGraphEventRef
任务完成的事件信号。每个 TGraphTask 创建时返回一个 FGraphEventRef（智能指针封装的 FGraphEvent）。可通过 FTaskGraphInterface::WaitUntilTasksComplete(EventArray) 阻塞等待一组事件全部完成，或通过 FGraphEvent::AddSubsequent() 将后续任务链入依赖图。引用计数管理生命周期，最后一个引用释放时自动销毁。

4. FRenderCommandFence 与同步原语

跨线程同步是保证渲染正确性的关键。FRenderCommandFence、RHI Fence 和 GPU Fence 等同步原语确保 GameThread 对渲染资源的访问不会与 GPU 执行产生数据竞争。

4.1 核心同步类型

GameThread → RenderThread
FRenderCommandFence
GameThread 等待 RenderThread 完成所有已提交渲染命令的核心同步原语。内部实现为引用计数 + FGraphEventRef：GameThread 调用 BeginFence() 时记录 RenderThread 队列尾部位置；调用 Wait() 时阻塞直到 RenderThread 消费完该位置之前的所有命令。

使用场景：确保下一帧游戏逻辑执行前渲染状态同步（如 UObject 的渲染代理更新）；关卡流送时确保已卸载关卡的渲染资源已完全释放（避免 UObject 析构时 RenderThread 仍在引用）。
RenderThread → RHIThread
FRHICommandListFence
RenderThread 等待 RHIThread 完成 GPU 命令提交的同步原语。当 RHIThread 启用时，RenderThread 提交的命令队列可能尚未被 RHIThread 消费，FRHICommandListFence 确保 RHIThread 已执行完指定命令列表中的所有命令。

使用场景：Buffer Readback 后确保 GPU 写入完成再读取 CPU 端数据（Staging Buffer 映射）；渲染资源（Texture/Buffer）释放前确认 RHI 线程不再持有对底层 D3D12/Vulkan 资源的引用。
GPU 端同步
FGPUFenceRHIRef
GPU 端同步原语，实现跨 Command Queue 或跨帧的 GPU-GPU 同步。基于 D3D12 Fence / Vulkan Timeline Semaphore 实现，由 GPU 直接信号通知，不经过 CPU 线程。

使用场景：AsyncCompute 完成后 Graphics 队列读取 Compute 输出结果（如 Lumen 光照评估结果供 BasePass 使用）；多 GPU AFR（Alternate Frame Rendering）帧间同步——确保前一 GPU 完成帧渲染后下一 GPU 开始渲染。

4.2 FlushRenderingCommands 详解

FlushRenderingCommands() 是 GameThread 阻塞等待 RenderThread + RHIThread 完成所有已提交命令的全局同步点。它内部依次执行：发送 FRenderCommandFence 并 Wait → 如果需要，再发送 FRHICommandListFence 并 Wait，确保两个线程都处理完所有已排队命令。

  • 截帧/截图：调用 FlushRenderingCommands() 确保渲染管线完全刷新后，再通过 Readback 获取最终帧缓冲内容。如果不同步，Readback 可能读到上一帧或未完成的帧数据。
  • 资源释放（World 卸载）：切换关卡或卸载 World 时，需确保渲染线程不再持有对即将销毁的 UObject（如 UTexture、UMaterial）的引用。否则 RenderThread 访问已释放内存导致 Crash。
  • 渲染到纹理 Readback：将场景渲染到 RenderTarget 后，如果 GameThread 需要立即使用该 RenderTarget 的内容（如保存到文件、传给 UI），必须先 Flush 确保内容已写入。
⚠ 性能警告：过度使用 FlushRenderingCommands 是渲染性能的常见杀手——它完全破坏线程流水线并行，导致 GameThread 停顿等待所有渲染工作完成。平均每次调用增加 2-8ms 额外延迟（取决于队列深度和 GPU 负载）。

替代方案：异步 Readback + Fence 轮询（GameThread 定期检查 Fence 状态，不阻塞主循环）；延迟资源释放到下一帧（通过 BeginCleanup() 注册延迟析构回调，等待 RenderThread 自然完成引用）。
// FRenderCommandFence 创建与等待的标准模式 class FMyRenderResource { FRenderCommandFence ReleaseFence; void BeginRelease() { // 步骤 1：在 RenderThread 上释放 GPU 资源 ENQUEUE_RENDER_COMMAND(ReleaseMyResource)( [this](FRHICommandListImmediate& RHICmdList) { // 释放 RHI 资源（Texture/Buffer/SRV 等） TextureRHI.SafeRelease(); BufferRHI.SafeRelease(); }); // 步骤 2：在 RenderThread 队列末尾插入 Fence ReleaseFence.BeginFence(); } bool IsReady() const { // 非阻塞检查：Fence 是否已被 RenderThread 通过 return ReleaseFence.IsFenceComplete(); } void WaitForRelease() { // 阻塞等待：GameThread 停顿直到 RenderThread 完成释放 // 仅在必须确认资源已释放时调用（如关卡卸载完成回调） ReleaseFence.Wait(); } };

5. RHI 命令队列

FRHICommandList 将高层渲染操作序列化为平台无关的命令流，RHI 线程批量提交到 GPU 驱动。命令队列支持 Immediate 和 Deferred 两种模式，直接影响渲染吞吐量和延迟。

5.1 FRHICommandList 结构

FRHICommandList 是 RHI 层的核心抽象——它将 RenderThread 发出的渲染指令（如 SetRenderTarget、DrawPrimitive、SetShaderParameter）逐一序列化为 FRHICommand* 派生类对象，存储在命令内存缓冲区中。RHIThread（或 Immediate 模式下的 RenderThread）消费这些命令并翻译为平台 API 调用。

1. 录制（Record）
用户代码调用 RHICmdList.SetRenderTarget()、RHICmdList.DrawPrimitive()、RHICmdList.SetShaderParameter() 等 RHI 接口函数。这些函数内部不直接调用 D3D12/Vulkan API，而是通过 RHICMD_* 宏生成对应的 FRHICommand* 对象，将参数序列化到命令内存缓冲区（FRHICommandList::Bypass() 或 Deferred 模式下的线性分配器 FMemStack）。
2. 提交（Submit）
RenderThread 完成当前帧的所有 Pass 注册后，通过 FRHICommandListExecutor::Submit()（或 RDG 的 GraphBuilder.Execute() 自动触发）将命令队列标记为就绪状态。在 Deferred 模式下，Submit 只设置信号量/事件，不实际执行命令——RHIThread 随后异步消费。在 Immediate 模式下，Submit 直接在当前线程执行所有命令。
3. 执行（Execute）
RHIThread（或 Immediate 模式下的 RenderThread）遍历命令队列中的每个 FRHICommand 对象，调用其 Execute() 虚函数。每个 Execute() 实现将参数翻译为对应的 D3D12 ID3D12GraphicsCommandList API 调用或 Vulkan VkCommandBuffer 函数调用，最终通过驱动提交到 GPU Command Queue。

5.2 Immediate vs Deferred 上下文

Immediate 模式
r.RHIThread.Enable=0 时生效。RHI 命令直接在当前线程（通常 RenderThread 兼任）执行，提交即执行。不经过中间队列缓冲，RHICmdList.DrawPrimitive() 调用直接触发 D3D12/Vulkan API 调用。

▲ 优点：零延迟——渲染命令立即提交到 GPU 驱动，减少一帧输入端延迟（适合 VR 场景）。

▼ 缺点：无流水线并行——RenderThread 在调用 D3D12/Vulkan API 时阻塞，无法与 RHIThread 形成流水线。RenderThread 每帧时间增加 15-30%。
Deferred 模式（默认）
r.RHIThread.Enable=1（默认值）时生效。命令先录制到线性分配器的内存缓冲区中，RHIThread 异步消费和执行。RenderThread 注册完 Pass 后在 GraphBuilder.Execute() 处批量提交，然后立即返回处理下一帧。

▲ 优点：RenderThread 与 RHIThread 完全流水线化——RenderThread 处理第 N+1 帧的 Pass 时，RHIThread 仍在提交第 N 帧的 GPU 命令。吞吐量显著提升。

▼ 缺点：增加 1 帧延迟——从 RenderThread 提交到 GPU 实际开始执行存在固定的一帧间隔。

5.3 FRHICommand 宏生成机制

每个 RHI 函数调用对应一个 FRHICommand* 派生类。UE 通过 RHICMD_* 系列宏自动生成两类回调：（1）录制回调——在调用线程上执行，将参数存储到命令对象中；（2）执行回调——在 RHIThread（或 Immediate 模式）上执行，调用平台 API。

// ===== 步骤 1：RHI 接口声明（RHICommandList.h） ===== void FRHICommandList::DrawPrimitive(uint32 BaseVertexIndex, uint32 NumPrimitives, uint32 NumInstances) { // 条件编译：Deferred 模式 vs Immediate 模式 #if RHI_COMMAND_LIST_DEBUG // Immediate 模式（简化版） GetContext().RHIDrawPrimitive(BaseVertexIndex, NumPrimitives, NumInstances); #else // Deferred 模式（录制命令） ALLOC_COMMAND(FRHICommandDrawPrimitive)(BaseVertexIndex, NumPrimitives, NumInstances); #endif } // ===== 步骤 2：宏展开后生成的命令类（RHICommandListCommandExec.inl） ===== struct FRHICommandDrawPrimitive : public FRHICommand<FRHICommandDrawPrimitive> { // --- 录制端参数存储 --- uint32 BaseVertexIndex; uint32 NumPrimitives; uint32 NumInstances; // --- 执行端（在 RHIThread 上调用）--- RHI_API void Execute(FRHICommandListBase& CmdList) override { // 多态派发到平台实现：FD3D12DynamicRHI / FVulkanDynamicRHI 等 RHISTAT(DrawPrimitive); INTERNAL_DECORATOR_CONTEXT(CmdList).RHIDrawPrimitive( BaseVertexIndex, NumPrimitives, NumInstances); } };

这套宏生成机制的优势在于：渲染代码只需写一次 RHICmdList.DrawPrimitive(...)，编译器根据 r.RHIThread.Enable 的值自动选择 Immediate 或 Deferred 路径。所有 FRHICommand 派生类的命令内存从 FMemStack 线性分配器中分配，无碎片、无锁竞争，单帧结束后整体回退。

6. FDynamicRHI 与平台抽象

FDynamicRHI 是 RHI 的平台抽象层接口，提供 D3D12、Vulkan、Metal 等后端适配。上层渲染代码通过统一接口操作 GPU 资源，无需关心平台差异。

6.1 FDynamicRHI 接口体系

FDynamicRHI 定义了约 300+ 纯虚函数，覆盖完整的 GPU 操作接口。它是一个抽象基类，由平台特化子类（FD3D12DynamicRHI、FVulkanDynamicRHI、FMetalDynamicRHI 等）实现。所有上层渲染代码通过 GDynamicRHI 全局指针（在 DynamicRHI.h 中声明）间接调用，无需关心后端是 D3D12 还是 Vulkan。

  • 资源创建：RHICreateTexture()、RHICreateBuffer()、RHICreateShader()、RHICreateRenderTarget()、RHICreateVertexDeclaration() — 创建 GPU 资源并返回平台无关句柄（FTextureRHIRef、FBufferRHIRef 等）。
  • 状态设置：RHISetRenderTarget()、RHISetPipelineState()、RHISetShaderParameter()、RHISetViewport()、RHISetBlendState()、RHISetDepthStencilState() — 设置 GPU 管线的各种状态，对应 D3D12 PSO 或 Vulkan Pipeline 的各组件。
  • 绘制：RHIDrawPrimitive()、RHIDrawIndexedPrimitive()、RHIDrawPrimitiveIndirect()、RHIDispatchComputeShader() — 实际触发 GPU 执行的绘制和计算命令。
  • 查询：RHICreateRenderQuery()（Occlusion Query）、RHICreateTimestampQuery()（GPU 时间戳）— GPU 端查询对象，用于遮挡剔除和 GPU Profiling。时间戳查询是 stat gpu 的数据源。
  • 资源锁定与传输：RHILockTexture() / RHIUnlockTexture()、RHIReadSurfaceData()、RHIUpdateTexture()、RHIUpdateBuffer() — CPU-GPU 数据传输，包括 Staging Buffer 映射和 Readback 路径。
  • 屏障管理：RHIBeginTransitions() / RHIEndTransitions()（UE5.x 新接口）— 批量资源状态转换，替代旧版的逐个资源屏障调用。内部映射到 D3D12 ResourceBarrier(count, barriers[]) 或 Vulkan vkCmdPipelineBarrier()。

6.2 D3D12RHI vs VulkanRHI 实现差异

特性D3D12RHIVulkanRHI
命令列表 ID3D12GraphicsCommandList — 硬件命令列表，录制后可提交到 Command Queue，支持 Bundle（可重用命令序列） VkCommandBuffer — 从 VkCommandPool 分配，支持 Primary/Secondary 两级命令缓冲
资源屏障 D3D12_RESOURCE_BARRIER — 三种类型：Transition、Aliasing、UAV。支持批量提交 ResourceBarrier(count, barriers[]) VkImageMemoryBarrier / VkBufferMemoryBarrier — 分别处理 Image Layout 转换和 Buffer 访问同步，通过 vkCmdPipelineBarrier() 提交
描述符 ID3D12DescriptorHeap (CBV/SRV/UAV Heap) — 描述符堆按类型分组，CBV/SRV/UAV 共享堆，Sampler 独立堆。支持 Descriptor Table（Root Signature 中的描述符范围） VkDescriptorPool + VkDescriptorSet — 描述符池按类型配额，DescriptorSet 通过 vkUpdateDescriptorSets() 绑定具体资源
根签名/管线布局 ID3D12RootSignature — 定义 Shader 可见的资源绑定布局（Root Constants + Root Descriptors + Descriptor Tables），绑定效率极高 VkPipelineLayout — 定义 DescriptorSetLayout + PushConstant 的范围，功能上等价于 Root Signature

绑定模型 Resource Binding Tier 2/3 (SM6.0+) — Tier 2 支持 CBV/SRV/UAV 描述符堆完全可见；Tier 3 支持无界描述符表 Standard Descriptor Set Binding — 通过 DescriptorSet 编号绑定资源，PushDescriptor 扩展允许直接写入描述符到 Command Buffer

Mesh Shader ID3D12MeshShader (SM6.6+) — 原生 Mesh Shader 管线，DispatchMesh() 替代传统 VS/GS/IA VK_EXT_mesh_shader — Vulkan 扩展，提供 vkCmdDrawMeshTasksEXT()，功能对等 D3D12

屏障批处理 ResourceBarrier(count, barriers[]) — 一次性提交多个资源屏障，驱动层面合并相邻屏障 vkCmdPipelineBarrier() — 一次调用可混合 Image Barrier、Buffer Barrier、Memory Barrier，支持多 Stage 和多 Access Mask 组合

6.3 资源创建路径

从 RDG 声明纹理到 D3D12 创建 Committed Resource 的完整调用链，展示 RHI 层如何在 RDG 和平台 API 之间桥接：

FRDGBuilder::CreateTexture(Desc, Name) → FDynamicRHI::RHICreateTexture()
Ray Step 1→RDG 调用 FRDGBuilder::CreateTexture(Desc, Name) — 声明纹理资源，记录到 RDG 资源表中，但不立即分配物理内存
Ray Step 2→Execute 阶段 TransientResourceAllocator 决定实际分配时机（Transient 资源按 Pass 间隔复用内存），调用 FDynamicRHI::RHICreateTexture()
Ray Step 3→FDynamicRHI 多态派发到 FD3D12DynamicRHI::RHICreateTexture() 或 FVulkanDynamicRHI::RHICreateTexture()
Ray Step 4→D3D12: CreateCommittedResource(D3D12_HEAP_TYPE_DEFAULT, ...) — 创建 Committed 资源 + 隐式 Heap 分配
Ray Step 5→Vulkan: vkCreateImage() + vkAllocateMemory() + vkBindImageMemory() — 三步骤显式分配
RHI 层的核心价值：所有上层渲染代码（FDeferredShadingSceneRenderer、FBasePassMeshProcessor、FLumenSceneLighting 等）无需关心后端是 D3D12 还是 Vulkan。一套 FDynamicRHI 接口覆盖所有 GPU 操作，平台差异被隔离在 FD3D12DynamicRHI / FVulkanDynamicRHI 的实现层。这也意味着 UE 新增平台支持只需实现一个 FDynamicRHI 子类，而非修改上层渲染管线的任何代码。

7. 管道状态对象（PSO）

PSO（Pipeline State Object）封装了 GPU 管线的完整状态——Shader 绑定、混合模式、光栅化状态、深度模板状态等。PSO 缓存机制避免了每帧重新创建管线状态的开销，直接影响 DrawCall 效率。

8. AsyncCompute 并行

AsyncCompute 允许 Compute Shader 工作与 Graphics 管线并行执行，典型应用包括 Nanite 裁剪、Lumen 光照评估、PostProcess 降噪等 Compute-heavy Pass 与 BasePass 并行。
