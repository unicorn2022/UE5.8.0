1. 管线中的位置

材质贯穿 UE5.8 渲染管线的多个核心阶段，从 BasePass 的 GBuffer 写入到延迟光照的 BRDF 计算，再到半透明前向着色和 Nanite 延迟材质着色。每一个渲染阶段都需要通过 FMaterialRenderProxy 和 FMaterial 获取材质属性。

在 FDeferredShadingSceneRenderer::Render() 中，材质相关的关键节点如下：

阶段 / 行号关键函数材质作用
Phase 3 — L2241 FinishGatherDynamicMeshElements 完成所有动态网格元素收集，生成 MeshDrawCommand 列表。每个命令包含材质排序键（SortKey）、PSO 初始状态、Shader 绑定。
Phase 4 — L2516 RenderNanite Nanite 光栅化阶段不直接使用材质，但为后续着色阶段准备 VisBuffer64（三角形 ID 编码），供材质查询使用。
Phase 4-5 间隙 — L2729 Nanite::BuildShadingCommands CPU 端提前构建 Nanite 着色命令，将 VisBuffer 像素按材质分组（Material Bin），为 Compute Shader 着色做准备。
Phase 5 — L3098 RenderBasePass BasePass 核心入口：传统几何通过 FMeshDrawCommand 硬件光栅化写入 GBuffer；Nanite 通过 Compute Shader 遍历 ShadingBin 写入 GBuffer。
Phase 5 — L3216 Substrate::AddSubstrateDBufferBasePass Substrate 框架下的 DBuffer 贴花材质投影到 GBuffer。
Phase 5 — L3223 Substrate::AddSubstrateMaterialClassificationPass Substrate 材质分类 Pass（像素级材质分类），为随机光照分类提供分类图。
Phase 6 — L3507 RenderLights 延迟光照计算阶段：读取 GBuffer 中的材质属性（法线、粗糙度、金属度、BaseColor），评估每个光源的 BRDF 贡献。
Phase 7 — L4057 RenderTranslucency 半透明渲染：使用前向渲染路径，按混合模式排序，支持折射扭曲和 OIT 乱序透明。
关键理解：材质不是单独的一个"Pass"，而是贯穿全管线的数据源。每条 MeshDrawCommand 绑定一组材质 Shader + Uniform Buffer，GPU 在每次 DrawCall 时根据绑定的材质资源计算像素着色结果。材质决定了每个像素写入 GBuffer 的 bit 级内容、BRDF 计算中的响应函数、混合模式下的颜色合成方式。

2.2 核心数据结构关系

UMaterial (GameThread)
材质蓝图资源，存储在 Material.cpp。包含材质表达式节点图（UMaterialExpression*），通过 FMaterialCompiler 翻译为 HLSL 代码。MaterialDomain 属性决定材质用途：Surface / DeferredDecal / LightFunction / Volume / PostProcess。
FMaterialRenderProxy (RenderThread)
材质渲染代理，位于 MaterialRenderProxy.cpp。GameThread 与 RenderThread 之间的桥接层。维护 UniformExpressionCache（每 FeatureLevel 一份），通过 UpdateUniformExpressionCacheIfNeeded 将材质参数填充到 GPU Uniform Buffer。支持 FColoredMaterialRenderProxy 等派生类。
FMaterial / FMaterialResource
编译后的运行时材质表示。包含 FMaterialShaderMap（材质类型→Shader 映射），通过 GetRenderingThreadShaderMap() 获取。提供 BRDF 查询接口（GetShadingModel、GetBlendMode 等）供渲染管线决策。
FMeshDrawCommand
渲染器内部的 DrawCall 封装，位于 MeshPassProcessor.cpp。包含 FMeshDrawCommandSortKey（排序键）、FGraphicsMinimalPipelineStateId（PSO 缓存）、Shader 绑定列表、顶点/索引缓冲区引用。Renderer 通过此结构提交最终的 GPU DrawCall。

2.3 材质域（MaterialDomain）分类

枚举值适用场景涉及的渲染 Pass
MD_Surface标准 3D 表面材质（不透明/半透明）BasePass (GBuffer)、Translucency Pass、Nanite Shading
MD_DeferredDecal延迟贴花材质DBuffer 渲染投影到 GBuffer
MD_LightFunction光照函数材质（光源遮罩/衰减）LightFunctionAtlas 渲染 → Clustered 采样
MD_Volume体积材质（雾、云等）体积渲染 Pass
MD_PostProcess后处理材质后处理管线（Bloom、Tonemap、色彩分级等）
MaterialDomain 定义位置：Engine/Source/Runtime/Engine/Public/MaterialDomain.h（UENUM 声明）。UMaterial::MaterialDomain 属性在蓝图编辑器中由用户设置，编译时决定材质的入口 Shader 类型。

3. 材质与 MeshDrawCommand

MeshDrawCommand 是渲染器内部的 DrawCall 原子封装，每条命令都绑定了一套完整的材质 Shader + 渲染状态。理解 MeshDrawCommand 的结构是理解材质渲染的关键。

3.1 FMeshDrawCommand 结构

组件 1
FMeshDrawCommandSortKey
64 位排序键，分为三种布局：
Opaque（不透明）：16-bit VertexShaderHash + 32-bit PixelShaderHash + 16-bit MeshPass
Translucent（半透明）：16-bit Priority + 16-bit MeshPass + 32-bit Distance（从远到近或从近到远）
Generic：32-bit VertexShaderHash + 32-bit PixelShaderHash
位于 MeshPassProcessor.h:1536。
组件 2
FGraphicsMinimalPipelineStateId
PSO（Pipeline State Object）最小描述符的唯一 ID。基于 RasterizerState + BlendState + DepthStencilState 哈希生成的索引，存储在全局 PersistentIdTable 中。同一 PSO 的 DrawCall 可以合并批处理，大幅减少 GPU 状态切换。
组件 3
Shader 绑定
Vertex Shader + Pixel Shader（或 Compute Shader）的 FRHIShader 引用 + Shader Resource 参数绑定表。每个命令绑定一个 MaterialUB（Uniform Buffer），包含材质常数、纹理 SRV、采样器状态等。
组件 4
几何数据
Index Buffer + Vertex Buffer 引用、图元数量、顶点工厂类型（FLocalVertexFactory 等）。通过 FVisibleMeshDrawCommand 包裹，在 InitViews 阶段确定可见性后提交到 Pass。

3.2 FinishGatherDynamicMeshElements 的作用

FinishGatherDynamicMeshElements 在可见性计算完成后调用（DeferredShadingRenderer.cpp L2241），负责：

InitViewTaskDatas.VisibilityTaskData->FinishGatherDynamicMeshElements(BasePassDepthStencilAccess, InstanceCullingManager, VirtualTextureUpdater.Get())
1. 遍历所有可见图元，对每个 Section 调用 FMeshPassProcessor::AddMeshBatch（BasePass / DepthPass / 等 EMeshPass）
2. 对每个 FMeshBatch，通过 BuildMeshDrawCommands 生成 FMeshDrawCommand：绑定材质 Shader、计算 SortKey、创建或查找 PSO ID
3. 将生成的 FMeshDrawCommand 添加到 Pass 对应的 FDynamicMeshDrawCommandStorage 中
4. 实例裁剪管理器（InstanceCullingManager）收集所有图元的 MeshDrawCommand，准备 GPU 端的实例裁剪
5. 虚拟纹理更新器（VirtualTextureUpdater）提交本帧需要的 VT 页面请求
设计要点：FinishGatherDynamicMeshElements 是 CPU 端渲染任务中最昂贵的一环。它需要为每个可见的材质 Section 调用 Shader 编译器生成的 Shader 绑定代码。UE5.8 通过 PSO 缓存（跳过已编译的 PSO）、StateBucket 合批（相同 SortKey + PSO 的 DrawCall 合并为 Instancing）、以及延迟队列（异步生成命令）来优化此阶段。

3.3 DrawCall 排序与批处理策略

生成的 FMeshDrawCommand 按 SortKey 排序后形成三个处理区间，与光照系统三桶分类类似：

Opaque（不透明）
按 VertexShaderHash + PixelShaderHash + MeshPass 排序。相同 Shader 的 DrawCall 聚集在一起，最大化 PSO 复用。不透明物体从前到后渲染（利用 Early-Z），不需要逐像素排序。
Masked（遮罩）
与 Opaque 相同的排序键布局，但需要执行 Alpha Test（discard）。在 Opaque 之后渲染以确保遮罩像素不会错误遮挡半透明物体。
Translucent（半透明）
按 Priority + MeshPass + Distance 排序。支持从远到近（BackToFront）或从近到远（FrontToBack）两种排序模式，取决于混合操作的类型（加法混合通常从远到近，乘法混合从近到远）。

4. BasePass 中的材质

BasePass 是材质属性的"写入"阶段。传统几何体通过 FMeshDrawCommand 硬件光栅化写入 GBuffer，Nanite 几何体通过 Compute Shader 着色路径写入同一组 GBuffer。

4.1 RenderBasePass 函数签名

// DeferredShadingRenderer.cpp L3094-L3098 RenderBasePass(*this, GraphBuilder, Views, SceneTextures, BasePassDepthStencilAccess, ForwardScreenSpaceShadowMaskTexture, InstanceCullingManager, bNaniteEnabled, Scene->NaniteShadingCommands[ENaniteMeshPass::BasePass], NaniteRasterResults);

在内部，BasePass 遍历所有 EMeshPass::BasePass 的可见 MeshDrawCommand（由 FinishGatherDynamicMeshElements 生成），执行两组独立的渲染：

RenderBasePass 内部双路径
路径 1（传统几何）：通过 FBasePassMeshProcessor 遍历可见 MeshDrawCommand，使用硬件光栅化管线（IA → VS → PS）。Pixel Shader 计算材质属性并写入 GBuffer。
路径 2（Nanite 几何）：通过 Scene->NaniteShadingCommands[ENaniteMeshPass::BasePass] 传入预构建的 Nanite 着色命令。Compute Shader 按 ShadingBin 分组执行，从 VisBuffer64 解码三角形 ID，计算材质属性并写入 GBuffer。

4.2 GBuffer 通道编码详解

GBuffer 是延迟渲染的核心数据结构，共五张 RT（A/B/C/D/E），加上 LightingChannelsTexture 和 SceneDepth 构成完整的材质数据输出。以下表格为 UE5.8 默认模式的编码格式：

通道格式存储内容位级编码说明
GBufferA R10G10B10A2 UNorm WorldNormal.xyz + PerObjectGBufferData RGB: 世界空间法线（编码为 R10G10B10）
A(2bit): PerObjectGBufferData（HasShadowCast 标志）
GBufferB R8G8B8A8 UNorm Metallic + Specular + Roughness + ShadingModelID R: Metallic（金属度 0-1）
G: Specular（高光度 0-1）
B: Roughness（粗糙度 0-1）
A: ShadingModelID（着色模型选择，4bit） + SelectiveOutputMask（4bit）
GBufferC R8G8B8A8 sRGB BaseColor.rgb + IndirectIrradiance RGB(sRGB): BaseColor（基础色）
A: IndirectIrradiance（间接辐照度，预烘焙时有效）
GBufferD R8G8B8A8 UNorm CustomData (Subsurface / ClearCoat / etc.) 内容取决于 ShadingModel：
Subsurface: SubsurfaceColor + Opacity
ClearCoat: ClearCoat + ClearCoatRoughness
Hair: 发丝散射/高光参数
Default: 未使用
GBufferE R8G8B8A8 UNorm PrecomputedShadowFactors + Emissive R: PrecomputedShadowFactor（预计算阴影因子）
GB: CustomData 扩展或未使用
A: Emissive（自发光强度）
Nanite 与传统路径兼容性：Nanite Compute Shader 写入的 GBuffer 格式与传统 Pixel Shader 写入的格式完全一致。延迟光照阶段无需区分几何体来源——GBuffer 统一采样器同时适用于两条路径。这是 Nanite 与现有管线无缝集成的核心设计保证。

4.3 材质属性与 ShadingModel 选择

GBufferB.A 中的 ShadingModelID 是决定延迟光照如何解释 GBuffer 数据的关键。每个 ShadingModel 在 BasePass Pixel Shader 中有独立的编码路径，在延迟光照阶段有对应的 BRDF 解码路径：

  • ShadingModelID 编码：BasePass 的 Pixel Shader 根据材质选择的 ShadingModel 将对应的属性（CustomData、Specular 等）写入 GBufferD/GBufferE
  • ShadingModelID 解码：延迟光照 BRDF 评估时，先读取 GBufferB.A 获取 ShadingModelID，再按对应路径解码 GBufferD/GBufferE 中的自定义数据
  • Substrate 扩展：启用 Substrate 后，GBuffer 布局可能不同（分类图优先），但原理相同——Substrate::AddSubstrateMaterialClassificationPass 在 BasePass 后添加材质分类 Pass

5. Nanite 中的材质

Nanite 采用延迟材质着色（Deferred Material Shading）方案：先将所有可见三角形 ID 记录到 VisBuffer，再将像素按材质分组（Material Bin），最后用 Compute Shader 批量计算材质属性。

5.2 提前执行优化

Nanite::BuildShadingCommands 在预通道后立即执行（L2729），而非等到 BasePass 阶段。源码注释明确阐述：

// DeferredShadingRenderer.cpp L2725-L2734 // Run Nanite compute commands early in the frame to allow some task overlap // on the CPU until the base pass runs. if (bNaniteEnabled && RendererOutput != ERendererOutput::DepthPrepassOnly) { Nanite::BuildShadingCommands(GraphBuilder, *Scene, ENaniteMeshPass::BasePass, Scene->NaniteShadingCommands[ENaniteMeshPass::BasePass]); if (bAnyLumenEnabled && RendererOutput == ERendererOutput::FinalSceneColor) { Nanite::BuildShadingCommands(GraphBuilder, *Scene, ENaniteMeshPass::LumenCardCapture, Scene->NaniteShadingCommands[ENaniteMeshPass::LumenCardCapture]); } }

这是一种 CPU 端任务重叠的经典优化：着色命令的构建（扫描 VisBuffer、分配 ShadingBin、记录材质参数）与后续的 HZB 构建、距离场准备等任务在 TaskGraph 中并行执行。

6. 半透明材质

半透明材质使用前向渲染路径（而非延迟渲染），直接采样半透明光照体积（Translucency Lighting Volume）获取光照信息。排序策略和混合模式是半透明渲染的核心性能决策点。

6.1 混合模式与排序策略

混合模式EBlendMode 枚举排序方向说明
TranslucentBLEND_TranslucentBackToFront（从远到近）标准半透明，按深度从远到近渲染以正确混合
AdditiveBLEND_AdditiveBackToFront / Any加法混合顺序无关，但仍按深度排序以利用 Early-Z
ModulateBLEND_ModulateFrontToBack（从近到远）乘法混合必须从近到远，否则结果错误
AlphaCompositeBLEND_AlphaCompositeBackToFront预乘 Alpha 合成
AlphaHoldoutBLEND_AlphaHoldoutBackToFrontAlpha 抠除（用于遮罩合成）

6.2 扭曲材质（Distortion）

扭曲材质通过 RenderDistortion 函数（位于 DistortionRendering.cpp）在单独的 Pass 中渲染。写入 SceneTextures.Distortion 纹理，在后处理阶段用于屏幕空间 UV 偏移。关键参数包括 r.Distortion 和 r.Distortion.SceneColorFringe。

6.3 OIT（Order-Independent Transparency）乱序透明

UE5.8 支持 OIT 乱序透明渲染（OIT/OIT.cpp），用于解决传统排序半透明中的顺序问题。OIT 在单个 Pass 中收集所有半透明片元，使用加权混合或链表排序生成最终颜色：

  • Weighted Blended OIT：使用 r.OIT.Weighted 启用，通过权重函数近似颜色混合，单 Pass 完成
  • Linked-List OIT：使用 r.OIT.LinkedList 启用，需要 UAV 写入，逐像素存储有序片元链表
  • 半透明体积：RenderTranslucency 从 TranslucencyLightingVolumeTextures 采样 3D 体素光照，避免对每个半透明像素单独执行光照计算

7. DBuffer 贴花

DBuffer（Decal Buffer）贴花是 UE5 延迟贴花系统的重要组成部分。贴花材质（MD_DeferredDecal）通过 DBuffer 纹理投影到 GBuffer，修改不透明表面的法线、粗糙度、颜色等属性。

7.2 贴花材质流程

DBuffer 贴花的完整渲染流程是一个多阶段管线：

CompositionLighting.ProcessBeforeBasePass → RenderDBuffer
1. DBufferTextures 创建：FDBufferTextures 结构体在 DBufferTextures.h 中定义，包含 DBufferA（法线+粗糙度）、DBufferB（金属度+高光度+不透明度）、DBufferC（基础色）三张纹理
2. 贴花材质遍历：收集所有可见的 MD_DeferredDecal 材质，对每个贴花网格构建 MeshDrawCommand（通过 FDeferredDecalPassMeshProcessor）
3. 投影渲染：每个贴花以其实体盒（Decal Box）为投影区域，渲染到 DBuffer 纹理。Pixel Shader 使用贴花材质计算法线/粗糙度/颜色修改值
4. BasePass 应用：在 BasePass 写入 GBuffer 时，采样 DBuffer 纹理并混合到最终 GBuffer 值中（不透明像素的材质属性有 DBuffer 偏移）
Substrate 路径差异：经典路径下 DBuffer 混合在 BasePass PS 中完成。Substrate 启用后，Substrate::AddSubstrateDBufferBasePass（L3216）和 Substrate::AddSubstrateDBufferPass（L3226）提供独立的 DBuffer 处理链路。

8. 材质着色模型

UE5.8 支持两种材质着色模型框架：传统的 EMaterialShadingModel 枚举系统和 UE5 新增的 Substrate 材质分类框架。

8.1 Substrate 材质分类框架

Substrate 是 UE5 引入的实验性材质框架，取代了传统的单一 ShadingModel 选择。在 Substrate 框架中，材质被表示为多个层次（Slab）的组合，每个 Slab 可以有不同的材质属性。分类 Pass（Substrate::AddSubstrateMaterialClassificationPass）在 BasePass 后对每个像素进行材质分类：

// DeferredShadingRenderer.cpp L3219-L3224 // Substrate classification is done either in a standalone pass (here) // or done within the StochasticLightingTileClassificationMark pass const bool bNeedsClassificationPass = !(RequiresStochasticLightingPass() && Substrate::UsesStochasticLightingClassification(ShaderPlatform)); if (bNeedsClassificationPass) { Substrate::AddSubstrateMaterialClassificationPass(GraphBuilder, SceneTextures, Views); }

分类 Pass 产出一张像素级分类纹理（Material Classification Map），用于后续的随机光照 Tile 分类 (StochasticLightingTileClassificationMark)，指导 MegaLights 等系统在每个 Tile 选择合适的光源采样策略。

Substrate vs 传统 ShadingModel：Substrate 不依赖固定的 ShadingModelID，而是通过 Substrate Slab 组合来表达任意复杂的材质层次。启用 Substrate 后，GBufferB.A 中不再存储 ShadingModelID，而是存储 Substrate 编码的 Slab 索引。Substrate 材质分类 Pass 为每个像素生成分类标签，用于后续光照阶段的高效分类调度。
