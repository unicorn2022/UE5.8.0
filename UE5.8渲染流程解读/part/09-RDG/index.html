1. 管线中的位置

RDG（Render Dependency Graph）是 UE5.8 渲染管线的核心调度框架，FRDGBuilder 负责管理所有渲染 Pass 的依赖关系、资源生命周期和自动屏障插入，是整个延迟渲染流程的"骨架"。

FRDGBuilder 在 Render() 函数中 L1829 附近创建：FRDGBuilder GraphBuilder(RHICmdList, ...)。它属于管线段 1（视图与管线初始化）中最早构建的核心对象之一，立即跟随 FGPUScene 的更新。构造后紧接着预热 RDG 黑板：Substrate::InitialiseSubstrateFrameSceneData(GraphBuilder, ...) 和 HairStrands::InitialiseHairStrandsFrameSceneData(GraphBuilder, ...)，将 Substrate/HairStrands 的帧级场景数据注入。

RDG 覆盖管线中几乎所有渲染 Pass——从管线段 4（PrePass + Nanite）到段 8（后处理），全部通过 GraphBuilder.AddPass() 注册。真正的 GPU 提交发生在管线段 9 收尾前，GraphBuilder.Execute()（L4423-L4456）一次性将所有注册的 Pass 编译为依赖图、分配 Transient 资源、插入屏障、调度执行。

以下是 FDeferredShadingSceneRenderer::Render() 中 RDG 相关的关键行号：

行号事件说明
L1829 FRDGBuilder GraphBuilder(...) 构造 RDG 依赖图构建器，注册所有后续 Pass。参数包括 RHICmdList、ShaderPlatform、GSupportsEfficientAsyncCompute 等。
L2241 FinishGatherDynamicMeshElements MeshDrawCommand 生成（RDG 外部执行，但决定了后续 BasePass/DepthPass 中可见的 DrawCall 列表）。
L2476-2534 PrePass + Nanite 光栅化 首批 RDG Pass 注册：深度预通道（RenderPrePass）和 Nanite 主光栅化（Nanite::RenderMainPass），生成 SceneDepth 和 Nanite VisBuffer。
L2729 Nanite::BuildShadingCommands 提前构建 Nanite 着色命令（CPU 端着色命令构建，不注册 GPU Pass，但为后续 Nanite BasePass 着色做准备）。
L3098 RenderBasePass 最大 RDG Pass 组：传统几何硬件光栅化 + Nanite Compute Shader 着色，写入 GBuffer A/B/C/D/E。
L3507 RenderLights 延迟光照计算：Clustered Deferred Shading 光源遍历，读取 GBuffer 评估 BRDF 贡献。
L4057 RenderTranslucency 半透明前向渲染：按混合模式排序，支持折射扭曲和 OIT 乱序透明。
L4274 AddPostProcessingPasses 后处理管线：Bloom、Tonemap、DOF、MotionBlur、TSR 等，全部通过 RDG 注册。
L4423-4456 GraphBuilder.Execute() 提交所有 Pass：触发依赖图编译、拓扑排序、Transient 资源分配、屏障插入、Pass 调度执行，帧渲染收尾。
RDG 的核心价值：将传统命令式资源管理（手动 CreateTexture / ReleaseTexture / RHICmdList.TransitionResource）转变为声明式依赖图。Pass 只需声明"我需要读 Texture A、写 Texture B"，RDG 自动处理：资源分配与回收、跨 Pass 屏障插入（UAV Barrier / RT Transition）、无重叠资源的内存别名复用。这不仅消除了一大类资源管理 bug，还通过 Transient 资源池显著降低了显存峰值占用。

2. 核心概念

RDG 的核心概念包括 Pass、资源（Texture/Buffer）、依赖边、Transient 资源分配和自动屏障系统，这些共同构成了 GPU 管线的高效调度模型。

2.1 三大原语

原语 1
FRDGBuilder
渲染依赖图构建器，源码位于 RenderGraphBuilder.h。职责：注册 Pass、创建图资源、构建依赖图、在 Execute() 时调度执行。单帧生命周期：构造 → AddPass/CreateTexture → Execute → 析构。关键方法：AddPass() 注册渲染 Pass、CreateTexture() 创建图纹理句柄、CreateBuffer() 创建图缓冲区句柄、QueueTextureExtraction() 将图资源移出供后续使用。
原语 2
FRDGTextureRef / FRDGBufferRef
编译时引用计数句柄，并非实际 GPU 资源。仅在 Execute() 时由 TransientResourceAllocator 实际分配。不可跨帧持有（FRDGBuilder 析构后失效）。使用 QueueTextureExtraction 将资源移出 RDG 图供后续管线阶段使用。支持 External Texture 注册（RegisterExternalTexture）将已有池化纹理借入 RDG。
原语 3
RDG Pass
四种类型：Raster（ERDGPassFlags::Raster，图形管线）、Compute（ERDGPassFlags::Compute，计算管线）、AsyncCompute（异步计算队列）、Copy（资源拷贝）。每个 Pass 声明包含：参数结构体（SHADER_PARAMETER_STRUCT）+ 执行 Lambda + Pass Flags。Pass 间通过资源引用隐式建立依赖关系——无需手动指定执行顺序。

2.2 访问声明与依赖推导

每个 Pass 声明对资源的访问方式（Read/Write），RDG 据此自动判定 Pass 间依赖、插入资源屏障、分析内存别名机会。访问模式由参数结构体中的宏自动推导：SHADER_PARAMETER_RDG_TEXTURE 声明读访问，SHADER_PARAMETER_RDG_TEXTURE_UAV 声明写访问，SHADER_PARAMETER_RDG_BUFFER_SRV / SHADER_PARAMETER_RDG_BUFFER_UAV 同理。

"注册 vs 实际分配"是 RDG 最核心的设计理念：CreateTexture 只是声明意图，实际 GPU 内存分配发生在 Execute() 阶段，由 RDG 根据生命周期分析统一优化。这意味着：(1) 创建了但从未被 Pass 引用的资源根本不会分配；(2) 生命周期不重叠的多个资源可能共享同一块显存（别名复用）；(3) Pass 被裁剪时其输出资源也一并消除。

3. Pass 注册与执行

每个渲染 Pass 通过 AddPass 注册到 FRDGBuilder 中，声明其输入/输出资源，RDG 自动分析跨 Pass 依赖关系并决定执行顺序。

每个 RDG Pass 通过 AddPass 注册。RDG 在编译时通过 C++ 模板元编程反射参数结构体，提取所有资源引用和访问模式。这是 RDG 区别于传统手动渲染框架的核心机制——编译期类型推导使得资源依赖关系在 Pass 注册时自动建立，无需手动维护 Barrier 顺序。

3.1 AddPass 内部流程

FRDGBuilder::AddPass(PassName, ParameterStruct, PassFlags, Lambda)
1. Lambda 参数捕获 → 编译时通过 TParameterStructTypeInfo 进行类型推导，反射遍历参数结构体中的所有 SHADER_PARAMETER_RDG_* 声明
2. 遍历参数结构体反射信息，注册每个参数：Texture SRV/UAV、Buffer SRV/UAV、RenderTarget、UniformBuffer
3. 根据访问方式（Read/Write）建立资源依赖边（Producer → Consumer），写方为 Producer，读方为 Consumer
4. 当所有 Pass 注册完毕，Execute() 调用内部 Compile() 进行拓扑排序
5. 按排序结果依次 Dispatch Pass Lambda 到 RHI 命令列表

3.2 拓扑排序与无用 Pass 裁剪

RDG 在 Execute() 阶段对已注册的所有 Pass 执行编译分析，核心包含三个优化步骤：

拓扑排序
RDG 将 Pass 构建为 DAG 节点，通过 Kahn 算法（入度归零）进行拓扑排序。依赖边由资源引用自动生成：Pass B 读取 Pass A 写入的资源 → A 必须在 B 之前执行。多个无依赖的 Pass 可以并行提交（AsyncCompute 队列）。
无用 Pass 裁剪（Culling）
只有对最终输出有贡献的 Pass 才会执行。最终输出 = 被 QueueTextureExtraction 提取的资源 + External 注册且外部可见的资源。无下游消费者的 Pass 被自动剔除——即便有 GPU 副作用也不会执行。SceneColor/SceneDepth 外提 Pass 不会被裁剪。
Pass 合并
相邻的同类型 Pass（无依赖关系、无屏障需求）可被 RDG 合并为单个 Dispatch 以减少 CPU 开销。合并条件包括：同 Pass 类型（Raster/Compute/Copy）、无资源依赖冲突、无跨 Pass 屏障。最终的 SceneColor/SceneDepth 外提 Pass 和带外部副作用的 Pass 不会被合并。

3.3 注册示例

Raster Pass 注册

// 全屏 Raster Pass 注册 — 使用 FPixelShaderUtils 便捷接口 struct FMyFullscreenPS : FGlobalShader { DECLARE_GLOBAL_SHADER(FMyFullscreenPS); SHADER_USE_PARAMETER_STRUCT(FMyFullscreenPS, FGlobalShader); BEGIN_SHADER_PARAMETER_STRUCT(FParameters, ) SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture) SHADER_PARAMETER_SAMPLER(SamplerState, InputSampler) RENDER_TARGET_BINDING_SLOTS() END_SHADER_PARAMETER_STRUCT() }; // 注册 Pass auto* PassParameters = GraphBuilder.AllocParameters<FMyFullscreenPS::FParameters>(); PassParameters->InputTexture = SceneColorTexture; PassParameters->InputSampler = TStaticSamplerState<SF_Bilinear>::GetRHI(); PassParameters->RenderTargets[0] = FRenderTargetBinding(OutputTexture, ERenderTargetLoadAction::ELoad); FPixelShaderUtils::AddFullscreenPass( GraphBuilder, View.ShaderMap, RDG_EVENT_NAME("MyFullscreenPass"), Shader, PassParameters, View.ViewRect);

Compute Pass 注册

// Compute Pass 注册 — 使用 FComputeShaderUtils 便捷接口 struct FMyComputeCS : FGlobalShader { DECLARE_GLOBAL_SHADER(FMyComputeCS); SHADER_USE_PARAMETER_STRUCT(FMyComputeCS, FGlobalShader); BEGIN_SHADER_PARAMETER_STRUCT(FParameters, ) SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture) SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D, OutputUAV) SHADER_PARAMETER(FVector2f, TextureSize) END_SHADER_PARAMETER_STRUCT() }; // 注册 Pass auto* PassParameters = GraphBuilder.AllocParameters<FMyComputeCS::FParameters>(); PassParameters->InputTexture = InputRDGTexture; PassParameters->OutputUAV = GraphBuilder.CreateUAV(OutputRDGTexture); PassParameters->TextureSize = FVector2f(Width, Height); FComputeShaderUtils::AddPass( GraphBuilder, RDG_EVENT_NAME("MyComputePass"), ERDGPassFlags::Compute, Shader, PassParameters, FIntVector(GroupCountX, GroupCountY, 1));
Lambda 中不要做重量级 CPU 操作：Lambda 在 Execute() 阶段于 RenderThread 执行，直接阻塞 GPU 提交。材质编译、纹理流送、资源加载等应在 Pass 注册前完成，仅保留 RHI 命令录制在 Lambda 中。如果某个 Pass 的 Lambda 耗时超过 0.1ms，考虑将前置 CPU 工作提前到注册阶段完成，或通过 TaskGraph 异步化。

4. 资源生命周期

RDG 资源由 FRDGBuilder 统一管理生命周期，从创建到最后一次被引用结束后自动释放，避免手动资源管理的内存泄漏和使用后释放（UAF）问题。

5. Transient 资源分配

TransientResourceAllocator 负责临时资源的池化分配与别名复用，通过分析资源活跃区间（Live Range）将不重叠的资源分配到同一物理显存块，显著降低显存占用。

6. 依赖分析与自动屏障

RDG 根据 Pass 间资源的读写关系自动插入 RHI 屏障（UAV Barrier、Render Target Transition 等），确保正确的 GPU 数据依赖和缓存一致性。

8. RDG 验证层

RDG 内置验证层可以检测未使用的资源声明、资源访问冲突、Pass 执行顺序错误等问题，是开发阶段定位渲染 bug 的重要工具。

9. RDG Dump 与调试

通过 GraphViz 导出、调试标记和 RDG Insights 等工具，可视化分析实际的渲染图依赖结构，定位性能瓶颈和依赖错误。

10. 性能与陷阱

RDG 使用中的常见性能陷阱，包括过度细分的 Pass 导致编译开销过大、不必要的资源拷贝、Transient 资源过度分配等问题及最佳实践。
