1. 管线中的位置

RDG（Render Dependency Graph）是 UE5.8 渲染管线的核心调度框架，FRDGBuilder 负责管理所有渲染 Pass 的依赖关系、资源生命周期和自动屏障插入，是整个延迟渲染流程的"骨架"。

FRDGBuilder 在 Render() 函数中 L1829 附近创建：FRDGBuilder GraphBuilder(RHICmdList, ...)。它属于管线段 1（视图与管线初始化）中最早构建的核心对象之一，立即跟随 FGPUScene 的更新。构造后紧接着预热 RDG 黑板：Substrate::InitialiseSubstrateFrameSceneData(GraphBuilder, ...) 和 HairStrands::InitialiseHairStrandsFrameSceneData(GraphBuilder, ...)，将 Substrate/HairStrands 的帧级场景数据注入。

RDG 覆盖管线中几乎所有渲染 Pass——从管线段 4（PrePass + Nanite）到段 8（后处理），全部通过 GraphBuilder.AddPass() 注册。真正的 GPU 提交发生在管线段 9 收尾前，GraphBuilder.Execute()（L4423-L4456）一次性将所有注册的 Pass 编译为依赖图、分配 Transient 资源、插入屏障、调度执行。

以下是 FDeferredShadingSceneRenderer::Render() 中 RDG 相关的关键行号：

行号事件说明
L1829 FRDGBuilder GraphBuilder(...) 构造 RDG 依赖图构建器，注册所有后续 Pass。参数包括 RHICmdList、ShaderPlatform、GSupportsEfficientAsyncCompute 等。
L2241 FinishGatherDynamicMeshElements MeshDrawCommand 生成（RDG 外部执行，但决定了后续 BasePass/DepthPass 中可见的 DrawCall 列表）。
L2476-2534 PrePass + Nanite 光栅化 首批 RDG Pass 注册：深度预通道（RenderPrePass）和 Nanite 主光栅化（Nanite::RenderMainPass），生成 SceneDepth 和 Nanite VisBuffer。
L2729 Nanite::BuildShadingCommands 提前构建 Nanite 着色命令（CPU 端着色命令构建，不注册 GPU Pass，但为后续 Nanite BasePass 着色做准备）。
L3098 RenderBasePass 最大 RDG Pass 组：传统几何硬件光栅化 + Nanite Compute Shader 着色，写入 GBuffer A/B/C/D/E。
L3507 RenderLights 延迟光照计算：Clustered Deferred Shading 光源遍历，读取 GBuffer 评估 BRDF 贡献。
L4057 RenderTranslucency 半透明前向渲染：按混合模式排序，支持折射扭曲和 OIT 乱序透明。
L4274 AddPostProcessingPasses 后处理管线：Bloom、Tonemap、DOF、MotionBlur、TSR 等，全部通过 RDG 注册。
L4423-4456 GraphBuilder.Execute() 提交所有 Pass：触发依赖图编译、拓扑排序、Transient 资源分配、屏障插入、Pass 调度执行，帧渲染收尾。
RDG 的核心价值：将传统命令式资源管理（手动 CreateTexture / ReleaseTexture / RHICmdList.TransitionResource）转变为声明式依赖图。Pass 只需声明"我需要读 Texture A、写 Texture B"，RDG 自动处理：资源分配与回收、跨 Pass 屏障插入（UAV Barrier / RT Transition）、无重叠资源的内存别名复用。这不仅消除了一大类资源管理 bug，还通过 Transient 资源池显著降低了显存峰值占用。

2. 核心概念

RDG 的核心概念包括 Pass、资源（Texture/Buffer）、依赖边、Transient 资源分配和自动屏障系统，这些共同构成了 GPU 管线的高效调度模型。

2.1 三大原语

原语 1
FRDGBuilder
渲染依赖图构建器，源码位于 RenderGraphBuilder.h。职责：注册 Pass、创建图资源、构建依赖图、在 Execute() 时调度执行。单帧生命周期：构造 → AddPass/CreateTexture → Execute → 析构。关键方法：AddPass() 注册渲染 Pass、CreateTexture() 创建图纹理句柄、CreateBuffer() 创建图缓冲区句柄、QueueTextureExtraction() 将图资源移出供后续使用。
原语 2
FRDGTextureRef / FRDGBufferRef
编译时引用计数句柄，并非实际 GPU 资源。仅在 Execute() 时由 TransientResourceAllocator 实际分配。不可跨帧持有（FRDGBuilder 析构后失效）。使用 QueueTextureExtraction 将资源移出 RDG 图供后续管线阶段使用。支持 External Texture 注册（RegisterExternalTexture）将已有池化纹理借入 RDG。
原语 3
RDG Pass
四种类型：Raster（ERDGPassFlags::Raster，图形管线）、Compute（ERDGPassFlags::Compute，计算管线）、AsyncCompute（异步计算队列）、Copy（资源拷贝）。每个 Pass 声明包含：参数结构体（SHADER_PARAMETER_STRUCT）+ 执行 Lambda + Pass Flags。Pass 间通过资源引用隐式建立依赖关系——无需手动指定执行顺序。

2.2 访问声明与依赖推导

每个 Pass 声明对资源的访问方式（Read/Write），RDG 据此自动判定 Pass 间依赖、插入资源屏障、分析内存别名机会。访问模式由参数结构体中的宏自动推导：SHADER_PARAMETER_RDG_TEXTURE 声明读访问，SHADER_PARAMETER_RDG_TEXTURE_UAV 声明写访问，SHADER_PARAMETER_RDG_BUFFER_SRV / SHADER_PARAMETER_RDG_BUFFER_UAV 同理。

"注册 vs 实际分配"是 RDG 最核心的设计理念：CreateTexture 只是声明意图，实际 GPU 内存分配发生在 Execute() 阶段，由 RDG 根据生命周期分析统一优化。这意味着：(1) 创建了但从未被 Pass 引用的资源根本不会分配；(2) 生命周期不重叠的多个资源可能共享同一块显存（别名复用）；(3) Pass 被裁剪时其输出资源也一并消除。

3. Pass 注册与执行

每个渲染 Pass 通过 AddPass 注册到 FRDGBuilder 中，声明其输入/输出资源，RDG 自动分析跨 Pass 依赖关系并决定执行顺序。

每个 RDG Pass 通过 AddPass 注册。RDG 在编译时通过 C++ 模板元编程反射参数结构体，提取所有资源引用和访问模式。这是 RDG 区别于传统手动渲染框架的核心机制——编译期类型推导使得资源依赖关系在 Pass 注册时自动建立，无需手动维护 Barrier 顺序。

3.1 AddPass 内部流程

FRDGBuilder::AddPass(PassName, ParameterStruct, PassFlags, Lambda)
1. Lambda 参数捕获 → 编译时通过 TParameterStructTypeInfo 进行类型推导，反射遍历参数结构体中的所有 SHADER_PARAMETER_RDG_* 声明
2. 遍历参数结构体反射信息，注册每个参数：Texture SRV/UAV、Buffer SRV/UAV、RenderTarget、UniformBuffer
3. 根据访问方式（Read/Write）建立资源依赖边（Producer → Consumer），写方为 Producer，读方为 Consumer
4. 当所有 Pass 注册完毕，Execute() 调用内部 Compile() 进行拓扑排序
5. 按排序结果依次 Dispatch Pass Lambda 到 RHI 命令列表

3.2 拓扑排序与无用 Pass 裁剪

RDG 在 Execute() 阶段对已注册的所有 Pass 执行编译分析，核心包含三个优化步骤：

拓扑排序
RDG 将 Pass 构建为 DAG 节点，通过 Kahn 算法（入度归零）进行拓扑排序。依赖边由资源引用自动生成：Pass B 读取 Pass A 写入的资源 → A 必须在 B 之前执行。多个无依赖的 Pass 可以并行提交（AsyncCompute 队列）。
无用 Pass 裁剪（Culling）
只有对最终输出有贡献的 Pass 才会执行。最终输出 = 被 QueueTextureExtraction 提取的资源 + External 注册且外部可见的资源。无下游消费者的 Pass 被自动剔除——即便有 GPU 副作用也不会执行。SceneColor/SceneDepth 外提 Pass 不会被裁剪。
Pass 合并
相邻的同类型 Pass（无依赖关系、无屏障需求）可被 RDG 合并为单个 Dispatch 以减少 CPU 开销。合并条件包括：同 Pass 类型（Raster/Compute/Copy）、无资源依赖冲突、无跨 Pass 屏障。最终的 SceneColor/SceneDepth 外提 Pass 和带外部副作用的 Pass 不会被合并。

3.3 注册示例

Raster Pass 注册

// 全屏 Raster Pass 注册 — 使用 FPixelShaderUtils 便捷接口 struct FMyFullscreenPS : FGlobalShader { DECLARE_GLOBAL_SHADER(FMyFullscreenPS); SHADER_USE_PARAMETER_STRUCT(FMyFullscreenPS, FGlobalShader); BEGIN_SHADER_PARAMETER_STRUCT(FParameters, ) SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture) SHADER_PARAMETER_SAMPLER(SamplerState, InputSampler) RENDER_TARGET_BINDING_SLOTS() END_SHADER_PARAMETER_STRUCT() }; // 注册 Pass auto* PassParameters = GraphBuilder.AllocParameters<FMyFullscreenPS::FParameters>(); PassParameters->InputTexture = SceneColorTexture; PassParameters->InputSampler = TStaticSamplerState<SF_Bilinear>::GetRHI(); PassParameters->RenderTargets[0] = FRenderTargetBinding(OutputTexture, ERenderTargetLoadAction::ELoad); FPixelShaderUtils::AddFullscreenPass( GraphBuilder, View.ShaderMap, RDG_EVENT_NAME("MyFullscreenPass"), Shader, PassParameters, View.ViewRect);

Compute Pass 注册

// Compute Pass 注册 — 使用 FComputeShaderUtils 便捷接口 struct FMyComputeCS : FGlobalShader { DECLARE_GLOBAL_SHADER(FMyComputeCS); SHADER_USE_PARAMETER_STRUCT(FMyComputeCS, FGlobalShader); BEGIN_SHADER_PARAMETER_STRUCT(FParameters, ) SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture) SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D, OutputUAV) SHADER_PARAMETER(FVector2f, TextureSize) END_SHADER_PARAMETER_STRUCT() }; // 注册 Pass auto* PassParameters = GraphBuilder.AllocParameters<FMyComputeCS::FParameters>(); PassParameters->InputTexture = InputRDGTexture; PassParameters->OutputUAV = GraphBuilder.CreateUAV(OutputRDGTexture); PassParameters->TextureSize = FVector2f(Width, Height); FComputeShaderUtils::AddPass( GraphBuilder, RDG_EVENT_NAME("MyComputePass"), ERDGPassFlags::Compute, Shader, PassParameters, FIntVector(GroupCountX, GroupCountY, 1));
Lambda 中不要做重量级 CPU 操作：Lambda 在 Execute() 阶段于 RenderThread 执行，直接阻塞 GPU 提交。材质编译、纹理流送、资源加载等应在 Pass 注册前完成，仅保留 RHI 命令录制在 Lambda 中。如果某个 Pass 的 Lambda 耗时超过 0.1ms，考虑将前置 CPU 工作提前到注册阶段完成，或通过 TaskGraph 异步化。

4. 资源生命周期

RDG 资源由 FRDGBuilder 统一管理生命周期，从创建到最后一次被引用结束后自动释放，避免手动资源管理的内存泄漏和使用后释放（UAF）问题。

4.2 External 资源注册

RDG 并不是孤立的资源管理系统——引擎中已存在大量"外部"纹理和缓冲区（如 SceneTextures.Color.Target、SceneDepthZ 等池化渲染目标）。RegisterExternalTexture() 和 RegisterExternalBuffer() 将已有 RHI 资源"借入"RDG 图，使得 RDG Pass 可以像使用内部资源一样引用它们，但不负责其生命周期管理。

关键规则：

  • External 资源不受 RDG 分配/释放管理——RDG 仅持有引用，不会在 Pass 执行后释放它们
  • 必须确保 External 资源在 RDG 图执行期间持续有效——析构前必须先 GraphBuilder.Execute()
  • 如果 RDG 内部创建的资源需要在 Execute() 后继续使用（非 Transient），必须通过 QueueTextureExtraction() 将其"提取"回外部世界
  • 提取后 RDG 不再管理该资源——调用方负责最终释放
// 将 SceneTextures 池化纹理注册到 RDG FRDGTextureRef SceneColorRDG = GraphBuilder.RegisterExternalTexture( SceneTextures.Color.Target, TEXT("SceneColor")); FRDGTextureRef SceneDepthRDG = GraphBuilder.RegisterExternalTexture( SceneTextures.Depth.Target, TEXT("SceneDepth")); // ... 在 RDG Pass 中使用 SceneColorRDG / SceneDepthRDG ... // Execute 后提取回外部 TRefCountPtr TRefCountPtr<IPooledRenderTarget> ExtractedColor; GraphBuilder.QueueTextureExtraction(SceneColorRDG, &ExtractedColor); GraphBuilder.Execute(RHICmdList); // Execute 后 ExtractedColor 指向更新后的纹理，可继续使用 RHICmdList.CopyTexture(..., ExtractedColor->GetRHI(), ...);

4.3 Upload 与 Readback

RDG 提供两个专用辅助 Pass 用于 GPU 数据传输，分别处理 CPU→GPU 和 GPU→CPU 方向：

AddUploadPass — CPU → GPU
将 CPU 端数据异步上传到 RDG Buffer。RDG 自动创建 staging buffer，复制数据，然后 GPU 端使用该 buffer。典型场景：逐帧更新的 Shader 参数常量、蒙皮矩阵数组、实例化变换数据。 FRDGBufferRef UploadedBuf = GraphBuilder.CreateBuffer(...); GraphBuilder.AddUploadPass(UploadedBuf, &CpuData, SizeInBytes);
AddReadbackPass — GPU → CPU
从 GPU Buffer 异步回读到 CPU 内存。RDG 内部创建 staging buffer，GPU 端复制完成后通知 CPU。典型场景：GPU 剔除结果回读、统计数据收集、调试可视化数据输出。 GraphBuilder.AddReadbackPass(GPUBuffer, &CpuReadbackData);
Readback 是异步的：AddReadbackPass 提交的 GPU→CPU 回读不会在 Execute() 返回后立即完成。CPU 端需要在后续帧（通常通过 FRHIGPUBufferReadback 或 Fence 轮询）等待 GPU 完成回读后再读取数据。直接在 Execute() 之后访问回读目标是未定义行为——数据可能尚未传输完成。正确处理模式：每帧检查上一帧的回读 Fence，就绪后再读取。

5. Transient 资源分配

TransientResourceAllocator 负责临时资源的池化分配与别名复用，通过分析资源活跃区间（Live Range）将不重叠的资源分配到同一物理显存块，显著降低显存占用。

5.1 声明与分配分离

CreateTexture() 和 CreateBuffer() 调用只是声明意图——告诉 RDG "我需要一个具有这些描述信息的资源"。此时并没有任何 GPU 内存被分配，返回值 FRDGTextureRef / FRDGBufferRef 仅是一个轻量级的引用计数句柄（约 32 字节），用于后续 Pass 的参数绑定。

真正的 GPU 内存分配发生在 GraphBuilder.Execute() 阶段，由 FRHITransientResourceAllocator（或 RHI 层的具体实现）统一决策和分配。这种分离机制带来三个核心优势：

  • 延迟决策：分配器在 Execute 时已掌握所有资源的完整生命周期信息，可以全局优化分配策略
  • 别名复用：生命周期不重叠的资源可以分配到同一块物理内存（见 5.2 节），大幅减少显存峰值占用
  • 按需分配：从未被任何 Pass 引用的资源根本不会触发分配，消除了无用资源的浪费

分配策略上，RDG 根据资源尺寸和使用模式选择两类路径：堆分配（适合 GBuffer/SceneColor 等大纹理，环形缓冲区式复用）和独立分配（适合小纹理/中间结果，独立创建与销毁）。

5.2 别名分析（Aliasing）

别名分析的核心原理是生命周期区间分析：RDG 在 Execute() 编译阶段扫描所有资源在每个 Pass 中的首次使用和最后使用时间点，建立每个资源的 Live Range（活跃区间）。任意两个 Live Range 不重叠的资源可以被分配到同一块物理内存——因为它们在时间线上不会同时存在。上图中 Resource B（紫色）在 Pass1 结束后释放，Resource D（绿色）在 Pass3 才开始使用，两者的生命周期完全隔离，因此 RDG 可以将 D 映射到 B 之前占用的物理地址空间，节省了 32 MB 显存。

别名复用的实际收益取决于 Pass 依赖链的深度和中间资源的体积。一个典型的 UE5 延迟渲染帧中，Transient 别名优化可以节省 25%~40% 的中间纹理显存峰值占用。

5.3 堆分配策略

RDG 底层使用两类 RHI 分配器处理不同尺寸和用途的 Transient 资源：

FRHITransientHeapAllocator — 大纹理堆
为大纹理（GBuffer、SceneColor、SceneDepth 等）提供环形缓冲区式堆分配。预分配一块大显存区域（通常数百 MB），按 Pass 顺序线性分配，帧末整体回收。类似帧分配器的思想——每帧从头开始分配，无需逐资源释放，帧结束整帧回收。适合生命周期覆盖整帧或大部分帧的持久资源，以及单帧内体积巨大的中间渲染目标。
FRHITransientResourceHeapAllocator — 小资源页
为小资源（UAV 中间缓冲区、Staging Buffer、小纹理等）提供页式分配。将内存划分为固定大小的页（通常 4 KB~64 KB），通过自由链表跟踪可用页，支持按需分配和即时回收。适合数量多、个体小、生命周期短的小型中间资源，避免外部碎片化。

可通过以下 CVar 控制 Transient 分配行为：

CVar说明
r.TransientAllocator.TextureCacheSize Transient 纹理缓存最大大小（MB），默认 512。控制环形堆的预分配容量
r.TransientAllocator.BufferCacheSize Transient Buffer 缓存最大大小（MB），默认 64
r.TransientAllocator.GarbageCollectInterval 垃圾回收间隔帧数，默认 60。控制页式分配器中未使用页的回收频率

6. 依赖分析与自动屏障

RDG 根据 Pass 间资源的读写关系自动插入 RHI 屏障（UAV Barrier、Render Target Transition 等），确保正确的 GPU 数据依赖和缓存一致性。

6.1 依赖图构建流

依赖图的构建完全自动化——Pass 只需声明"我需要读 Texture A、写 Texture B"，RDG 在注册阶段自动记录这些引用，并在 Compile() 阶段完成完整的依赖分析：

依赖分析的输入是每个 Pass 的资源访问声明（SRV = 读、UAV = 写、RTV = 写），RDG 内部建立三类标准依赖关系：RAW（Read After Write — Pass B 读的内容由 Pass A 写入）、WAR（Write After Read — Pass B 覆写前 Pass A 必须先读完）、WAW（Write After Write — 两次写入需严格排序）。这些依赖边共同构成一个有向无环图（DAG），拓扑排序后即可确定 Pass 的合法执行顺序。

6.2 自动屏障插入

在确定 Pass 执行顺序后，RDG 在每个需要资源转换的相邻 Pass 之间自动插入 RHI 转换屏障。以下是最常见的三种屏障场景：

UAV → SRV 转换
Pass A 通过 UAV（Unordered Access View）写入 Buffer，后续 Pass B 通过 SRV（Shader Resource View）读取同一 Buffer。RDG 自动在两个 Pass 之间插入 UAV → SRV 转换屏障，确保 Pass B 读取到的是 Pass A 完整写入后的结果。这是 Compute Shader 管线中最常见的屏障类型——忘记手动插入 UAV Barrier 会导致 Pass B 读到不完整或陈旧的数据。
RTV → SRV 转换
Pass A 以 RTV（Render Target View）写入纹理（例如 GBuffer 写入），后续 Pass B 以 SRV 采样同一纹理（例如延迟光照 Pass 采样 GBuffer A~E）。RDG 自动插入 RenderTarget → ShaderResource 转换屏障，确保像素着色器写入在采样前完成并刷新到一致缓存。延迟渲染管线中几乎每一帧都在发生这类转换。
同 Pass 内隐式串行
同一 Pass 内对同一资源的多次访问不插入额外屏障。GPU 的同一 CommandList（或同一 Dispatch/DrawCall）内，硬件保证先提交的指令先完成——无需显式 Barrier。RDG 仅在跨 Pass 边界（即不同 Pass 之间）插入屏障。将多个依赖操作合并在一个 Pass 中可以减少屏障开销，但代价是可能的并行度损失。

6.3 底层 D3D12 ResourceBarrier 生成

RDG 的自动屏障最终翻译为平台 API 的原生资源屏障调用。这一翻译过程经历两层抽象：

  • RDG Barrier → RHI 抽象 Barrier：RDG 内部的资源状态转换（如 ERHIAccess::UAVCompute → ERHIAccess::SRVCompute）先翻译为 RHI 层的 FRHITransitionInfo 结构体，描述资源的 Before/After 状态
  • RHI Barrier → D3D12/Vulkan 原生 Barrier：RHI 层根据平台（D3D12 或 Vulkan）将 FRHITransitionInfo 映射为 D3D12_RESOURCE_BARRIER 或 VkImageMemoryBarrier / VkBufferMemoryBarrier
  • Barrier 批处理（Batching）：相邻的多个屏障被合并为一个数组，通过单次 API 调用提交——D3D12 的 ResourceBarrier(NumBarriers, pBarriers) 可批量处理，减少 CPU 端 API 调用开销
  • Split Barrier 机制：对于跨队列（Graphics → AsyncCompute → Graphics）的资源转换，D3D12 支持 D3D12_RESOURCE_BARRIER_FLAG_BEGIN_ONLY / END_ONLY 标志。Begin Barrier 由源队列发出，End Barrier 由目标队列发出，中间的间隔可让两个队列并行执行无冲突的操作，提升并行粒度
// RDG 自动屏障插入示意 — 三 Pass 依赖链的 Barrier 插入位置 // 资源声明：PassA 写入 Texture T，PassB 读取 T，PassC 再次写入 T FRDGTextureRef T = GraphBuilder.CreateTexture(Desc, TEXT("Intermediate")); // Pass A: UAV 写入 T GraphBuilder.AddPass( RDG_EVENT_NAME("PassA_Write"), PassParamsA, // SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D, T) ERDGPassFlags::Compute, [](FRHIComputeCommandList& RHICmdList) { /* UAV 写入 ... */ }); // ★ RDG 自动插入 UAV→SRV 屏障 // D3D12: ResourceBarrier(T, UAV → SRV) // Vulkan: vkCmdPipelineBarrier( // srcStageMask = VK_PIPELINE_STAGE_COMPUTE_SHADER_BIT, // dstStageMask = VK_PIPELINE_STAGE_COMPUTE_SHADER_BIT, // srcAccessMask = VK_ACCESS_SHADER_WRITE_BIT, // dstAccessMask = VK_ACCESS_SHADER_READ_BIT); // Pass B: SRV 读取 T GraphBuilder.AddPass( RDG_EVENT_NAME("PassB_Read"), PassParamsB, // SHADER_PARAMETER_RDG_TEXTURE(Texture2D, T) ERDGPassFlags::Compute, [](FRHIComputeCommandList& RHICmdList) { /* SRV 采样 ... */ }); // ★ RDG 自动插入 SRV→UAV 屏障 // D3D12: ResourceBarrier(T, SRV → UAV) // Pass C: UAV 再次写入 T GraphBuilder.AddPass( RDG_EVENT_NAME("PassC_Rewrite"), PassParamsC, // SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D, T) ERDGPassFlags::Compute, [](FRHIComputeCommandList& RHICmdList) { /* UAV 写入 ... */ });
手动屏障是 GPU 崩溃的头号原因：在直接使用 RHI 编写渲染代码时，忘记插入 UAV→SRV 转换屏障、误用 Barrier 顺序、或未正确处理跨队列资源状态转换，是 GPU 崩溃和渲染错误的最高频原因。RDG 通过声明式资源访问 + 自动屏障推导完全消除了这一类问题——开发者不再需要手动编写任何 RHICmdList.TransitionResource() 或 D3D12 ResourceBarrier() 调用。这是 RDG 相对于传统命令式渲染框架质的飞跃。

7. 自定义 RDG Pass 完整模板

从 Shader 参数结构体定义到 AddPass 注册，展示一个完整的自定义 RDG Pass 编写模板，包含 Shader 实现和 C++ 调度代码。

7.1 光栅化 Pass 模板（Pixel Shader）

下面是编写一个完整的 Raster RDG Pass 的标准步骤——从 .usf Shader 代码到 C++ 调度：

  1. 定义 .usf Shader 文件中的入口函数（MainVS + MainPS），声明输入结构体和输出语义
  2. 创建 FGlobalShader 派生类，绑定 Shader 文件路径和入口点
  3. 使用 SHADER_PARAMETER_STRUCT 宏在 BEGIN / END 之间声明所有参数（SRV/UAV/Uniform）
  4. GraphBuilder.AddPass() 注册 Raster Pass，通过 RENDER_TARGET_BINDING_SLOTS 设置 RenderTarget
  5. Lambda 中获取 Shader、设置 PSO，调用 FPixelShaderUtils::DrawFullscreenTriangle 提交全屏三角形

完整示例：

// ===== MyFullscreenPass.usf ===== void MainVS(in float4 InPosition : ATTRIBUTE0, in float2 InTexCoord : ATTRIBUTE1, out float4 OutPosition : SV_POSITION, out float2 OutTexCoord : TEXCOORD0) { OutPosition = InPosition; OutTexCoord = InTexCoord; } Texture2D InputTexture; SamplerState InputSampler; float4 TintColor; void MainPS(in float4 SvPosition : SV_POSITION, in float2 TexCoord : TEXCOORD0, out float4 OutColor : SV_Target0) { OutColor = InputTexture.Sample(InputSampler, TexCoord) * TintColor; } // ===== MyFullscreenPass.h ===== class FMyFullscreenPS : FGlobalShader { public: DECLARE_GLOBAL_SHADER(FMyFullscreenPS); SHADER_USE_PARAMETER_STRUCT(FMyFullscreenPS, FGlobalShader); BEGIN_SHADER_PARAMETER_STRUCT(FParameters, ) SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture) SHADER_PARAMETER_SAMPLER(SamplerState, InputSampler) SHADER_PARAMETER(FVector4f, TintColor) RENDER_TARGET_BINDING_SLOTS() END_SHADER_PARAMETER_STRUCT() }; // 在 .cpp 中实现 Shader IMPLEMENT_GLOBAL_SHADER(FMyFullscreenPS, "/Engine/Private/MyFullscreenPass.usf", "MainPS", SF_Pixel); // ===== MyRendering.cpp — 注册 & 调度 ===== auto* PassParameters = GraphBuilder.AllocParameters<FMyFullscreenPS::FParameters>(); PassParameters->InputTexture = SceneColorTexture; PassParameters->InputSampler = TStaticSamplerState<SF_Bilinear>::GetRHI(); PassParameters->TintColor = FVector4f(1.0f, 1.0f, 1.0f, 1.0f); PassParameters->RenderTargets[0] = FRenderTargetBinding(OutputTexture, ERenderTargetLoadAction::EClear); GraphBuilder.AddPass( RDG_EVENT_NAME("MyFullscreenPass"), PassParameters, ERDGPassFlags::Raster, [PassParameters, &View](FRHICommandList& RHICmdList) { TShaderMapRef<FMyFullscreenPS> PixelShader(View.ShaderMap); FGraphicsPipelineStateInitializer GraphicsPSOInit; RHICmdList.ApplyCachedRenderTargets(GraphicsPSOInit); GraphicsPSOInit.BlendState = TStaticBlendState<>::GetRHI(); GraphicsPSOInit.RasterizerState = TStaticRasterizerState<>::GetRHI(); GraphicsPSOInit.DepthStencilState = TStaticDepthStencilState<false, CF_Always>::GetRHI(); GraphicsPSOInit.BoundShaderState.VertexDeclarationRHI = GFilterVertexDeclaration.VertexDeclarationRHI; GraphicsPSOInit.BoundShaderState.VertexShaderRHI = View.ShaderMap->GetShader<FMyFullscreenVS>().GetVertexShader(); GraphicsPSOInit.BoundShaderState.PixelShaderRHI = PixelShader.GetPixelShader(); SetGraphicsPipelineState(RHICmdList, GraphicsPSOInit, 0); SetShaderParameters(RHICmdList, PixelShader, PixelShader.GetPixelShader(), *PassParameters); FPixelShaderUtils::DrawFullscreenTriangle(RHICmdList, 1); });

7.2 Compute Pass 模板（Compute Shader）

Compute Pass 比 Raster Pass 更简洁——不需要 PSO 设置和全屏三角形，只需声明线程组数量和 Dispatch：

  1. 定义 .usf 入口，声明线程组大小：[numthreads(8,8,1)] void MainCS(...)
  2. 创建 FGlobalShader 派生类 + SHADER_PARAMETER_STRUCT，包含 SHADER_PARAMETER_UAV(RWTexture2D, Output) 等写资源声明
  3. FComputeShaderUtils::AddPass(GraphBuilder, RDG_EVENT_NAME("MyPass"), ComputeShader, Parameters, FIntVector(GroupCountX, GroupCountY, 1)) 一键注册
  4. RDG 自动处理 UAV 屏障——Compute Pass 输出被后续 Pass 读取时自动插入 UAV → SRV 转换
// ===== MyComputePass.usf ===== Texture2D InputTexture; RWTexture2D<float4> OutputUAV; float2 TextureSize; [numthreads(8, 8, 1)] void MainCS(uint3 DispatchThreadId : SV_DispatchThreadID) { if (any(DispatchThreadId.xy >= (uint2)TextureSize)) return; float2 UV = (DispatchThreadId.xy + 0.5f) / TextureSize; float4 Color = InputTexture.Load(int3(DispatchThreadId.xy, 0)); // 示例：亮度提取 float Luminance = dot(Color.rgb, float3(0.2126f, 0.7152f, 0.0722f)); OutputUAV[DispatchThreadId.xy] = float4(Luminance, Luminance, Luminance, 1.0f); } // ===== MyComputePass.h ===== class FMyComputeCS : FGlobalShader { public: DECLARE_GLOBAL_SHADER(FMyComputeCS); SHADER_USE_PARAMETER_STRUCT(FMyComputeCS, FGlobalShader); BEGIN_SHADER_PARAMETER_STRUCT(FParameters, ) SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture) SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D<float4>, OutputUAV) SHADER_PARAMETER(FVector2f, TextureSize) END_SHADER_PARAMETER_STRUCT() }; IMPLEMENT_GLOBAL_SHADER(FMyComputeCS, "/Engine/Private/MyComputePass.usf", "MainCS", SF_Compute); // ===== MyRendering.cpp — 注册 ===== const int32 GroupSize = 8; const int32 GroupCountX = FMath::DivideAndRoundUp(Width, GroupSize); const int32 GroupCountY = FMath::DivideAndRoundUp(Height, GroupSize); auto* PassParameters = GraphBuilder.AllocParameters<FMyComputeCS::FParameters>(); PassParameters->InputTexture = InputRDGTexture; PassParameters->OutputUAV = GraphBuilder.CreateUAV(OutputRDGTexture); PassParameters->TextureSize = FVector2f(Width, Height); FComputeShaderUtils::AddPass( GraphBuilder, RDG_EVENT_NAME("MyComputePass"), ERDGPassFlags::Compute, ComputeShader, PassParameters, FIntVector(GroupCountX, GroupCountY, 1));

7.3 RDG 黑板（Blackboard）跨 Pass 通信

FRDGBlackboard 是 RDG 提供的轻量级键值存储，用于在 Pass 间传递元数据——即非 GPU 资源的信息，如控制标志、统计计数、共享中间计算结果。与 FRDGTextureRef / FRDGBufferRef 等 GPU 资源不同，黑板数据仅存储在 CPU 端，不占用显存，不触发屏障。

// 在 Pass 注册前将数据写入黑板 auto& MyData = GraphBuilder.Blackboard.Create<FMyPassMetadata>(); MyData.bShouldRenderSSR = bHasReflectiveSurfaces; MyData.NumActiveLights = VisibleLights.Num(); // 后续 Pass 中读取黑板数据 const auto& MyData = GraphBuilder.Blackboard.Get<FMyPassMetadata>(); if (!MyData.bShouldRenderSSR) return; // 跳过 SSR Pass 注册，节省编译和调度开销
适用场景：黑板最适合存放三类数据：(1) 控制标志——本帧是否需要 SSR、是否需要 VSM 页面标记等；(2) 统计数据——可见光源数量、Nanite Cluster 数量、裁剪后的 DrawCall 计数；(3) 共享计算中间结果——预计算的 Tile 分布、分类后的光源列表索引。注意：黑板数据在 Execute() 后失效，不可跨帧持有。

7.4 CVar 控制开关

在生产环境中，自定义 Pass 应通过 CVar 提供运行时开关，方便调试和性能分析时快速禁用/启用特定 Pass 以隔离性能热点：

// 定义 CVar — 放在 .cpp 文件顶层 static TAutoConsoleVariable<int32> CVarMyCustomPass( TEXT("r.MyCustomPass.Enable"), 1, TEXT("Enable/Disable the MyCustomPass render pass.\n") TEXT(" 0: Disabled\n") TEXT(" 1: Enabled (default)\n") TEXT(" 2: Enabled with debug visualization"), ECVF_RenderThreadSafe); // 在 Pass 注册前检查 CVar 值 const int32 MyPassMode = CVarMyCustomPass.GetValueOnRenderThread(); if (MyPassMode == 0) return; // Pass 被禁用，不注册任何 RDG Pass，零开销 const bool bDebugVisualize = (MyPassMode == 2); // 注册 Pass 逻辑... auto* PassParameters = GraphBuilder.AllocParameters<FMyCustomPS::FParameters>(); PassParameters->bDebugMode = bDebugVisualize ? 1 : 0; // ...

标准实践：CVar 使用 ECVF_RenderThreadSafe 标志，通过 GetValueOnRenderThread() 在 RenderThread 安全读取。在 Pass 注册之前检查 CVar 值，可避免在禁用状态下创建无用的 RDG 资源和 Pass 节点，实现真正的零开销关闭。

8. RDG 验证层

RDG 内置验证层可以检测未使用的资源声明、资源访问冲突、Pass 执行顺序错误等问题，是开发阶段定位渲染 bug 的重要工具。

8.1 启用方式

验证层通过在命令行或 CVar 中启用，在 Execute() 时对所有已注册 Pass 执行额外检查。两种等价启用方式：

// 方式 1：命令行参数启动 UnrealEditor.exe <ProjectPath> -rdgdebug // 方式 2：运行时 CVar 设置 r.RDG.Debug 1

验证层在 GraphBuilder.Execute() 阶段运行——此时所有 Pass 已注册完毕，依赖图已构建完成。验证逻辑拦截在 Pass 调度执行之前（或与执行交叉进行，取决于验证级别），对所有已注册 Pass 执行完整性检查。验证失败时不会崩溃，而是通过 UE_LOG 输出详细错误信息，标明出错的 Pass 名称、资源名称和错误原因。

8.2 验证级别

r.RDG.Debug 支持三个验证级别，开销逐级递增：

级别值检查内容性能开销
Verbose 1 打印 Pass 列表、资源使用摘要、依赖图统计（节点数、边数、Transient 分配量） 极低（仅日志）
Extended 2 资源生命周期追踪、未使用资源警告、Pass 间依赖验证、外部资源注册一致性检查 低
Max 3 所有 Pass Lambda 在启用 RHI 验证的线程执行，检查所有资源访问合法性、UAV 冲突检测、跨队列资源状态一致性 高（仅调试用）

8.3 常见检测到的错误

验证层在开发过程中能捕获以下四类最常见的 RDG 使用错误：

错误 1
Resource not registered
Pass 访问了未通过参数结构体声明的资源。这是最常见的 RDG 错误——在 Lambda 中直接使用外部纹理（如 SceneTextures.Color.Target）而未调用 RegisterExternalTexture。RDG 无法感知未注册的资源，导致依赖链不完整、屏障缺失。修复：用 GraphBuilder.RegisterExternalTexture() 或通过 SHADER_PARAMETER_RDG_TEXTURE 声明。
错误 2
Cycle detected
Pass 依赖图中出现环。典型场景：Pass A 写资源 X 后 Pass B 读 X 并写 Y，Pass C 又写 X 并读 Y，形成 A → B → C → A 的循环依赖。DAG 不能有环——RDG 拒绝执行并报告循环涉及的 Pass 和资源链。修复：拆分 Pass、使用中间资源打破循环，或合并有冲突的 Pass。
错误 3
Read-After-Write hazard
两个 Pass 在同一资源上存在未解决的读写冲突。RDG 正常会自动插入屏障解决 RAW/WAW/WAR 依赖，出现此错误说明验证层检测到屏障插入失败——通常由资源在不同队列（Graphics vs AsyncCompute）上的状态转换异常引发。检查 Pass Flags 和跨队列资源声明。
错误 4
Unreferenced resource
资源已通过 CreateTexture / CreateBuffer 创建，但未被任何 Pass 引用。这通常是死代码残留——某个 Pass 被移除后其输出资源声明未清理。RDG 不会为这些资源分配显存（懒分配优化），但会在验证日志中警告，提示可能存在遗漏的参数绑定或已废弃的逻辑。

以下是验证层报错的典型输出示例（Resource not registered 错误）：

// ===== RDG 验证层输出示例 ===== LogRDG: Error: Pass [SSR_IndirectComposite] is accessing resource [SceneDepthZ] which is not registered in the parameter struct. This resource must be declared via SHADER_PARAMETER_RDG_TEXTURE or registered externally via GraphBuilder.RegisterExternalTexture(). Pass Stack: FDeferredShadingSceneRenderer::RenderSSR() → AddScreenSpaceReflectionsPass() → GraphBuilder.AddPass(SSR_IndirectComposite, ...) Resource Trace: SceneDepthZ is owned by FSceneTextures (external pool) → Call GraphBuilder.RegisterExternalTexture(SceneTextures.Depth.Target, TEXT("SceneDepth")) → Use the returned FRDGTextureRef in Pass parameter struct
CI 建议：在持续集成（CI）的自动化测试中启用 Extended 级别验证（r.RDG.Debug=2）。Extended 级别的开销很低（额外日志 + 资源追踪，约 0.1~0.3ms），但能自动捕获 Resource not registered、Unreferenced resource、依赖异常等大部分 RDG 使用错误。配合自动化截图对比测试，可以在合入主干前拦截渲染管线集成问题。

9. RDG Dump 与调试

通过 GraphViz 导出、调试标记和 RDG Insights 等工具，可视化分析实际的渲染图依赖结构，定位性能瓶颈和依赖错误。

9.1 GraphViz 可视化导出

RDG 支持将当前帧的完整依赖图导出为 GraphViz DOT 格式文件，配合 dot 命令行工具可生成 PNG/SVG 可视化图，每个 Pass 和每条依赖边都清晰标注：

  • 每个节点 = 一个 RDG Pass，节点标签显示 Pass 名称（来自 RDG_EVENT_NAME）和 Pass 类型
  • 每条有向边 = 一个资源依赖，箭头从生产者指向消费者，边上标注资源名称和访问方式
  • 节点颜色表示 Pass 类型：Raster（蓝色）、Compute（绿色）、AsyncCompute（橙色）、Copy（灰色）
// 启用依赖图 Dump（控制台命令） r.RDG.DumpGraph 1 // 输出路径 // Windows: <Project>/Saved/Profiling/RDG/ // 文件命名: RDG_<FrameNumber>_<ViewIndex>.dot // 转换为 PNG（需要安装 GraphViz） dot -Tpng RDG_0000_0.dot -o RDG_Graph.png // 转换为 SVG（矢量格式，推荐用于大型图） dot -Tsvg RDG_0000_0.dot -o RDG_Graph.svg // 限制输出规模 — 仅 Dump 前 N 个 View r.RDG.DumpGraph 1 // Dump 当前帧所有 View r.RDG.DumpGraphUnknowns 1 // 还包含未归类资源（排查用）
实用建议：对于理解复杂帧的 Pass 依赖关系非常有用，建议在性能分析时定期导出 GraphViz 图。一张典型的 UE5 延迟渲染帧包含 200~400 个 Pass，手动跟踪依赖关系几乎不可能——GraphViz 可视化能直观展示 Pass 拓扑、识别不必要的串行化依赖、发现可并行化的 Pass 组。配合 r.RDG.CullBadPasses 裁剪无用 Pass 后再 Dump，可以得到更清晰的图。

9.3 Unreal Insights 集成

RDG Pass 自动生成 Unreal Insights Trace Event——通过 RDG_EVENT_NAME("MyPass") 宏定义的名称会直接显示在 Insights 的 CPU/GPU 时间线中。无需额外插桩，无需手动 Trace 宏，所有通过 AddPass 注册的 Pass 自动出现在 Trace 中：

Timing 视图
CPU 轨道显示每个 Pass 的注册时间（窄条，AddPass 调用本身的 CPU 耗时）和执行时间（宽条，Lambda 内 RHI 命令录制耗时）。GPU 轨道显示 Pass 的 GPU 执行时长（从 Dispatch 到 GPU 完成信号）。通过对比 CPU 和 GPU 轨道可判断瓶颈类型：CPU 轨道长于 GPU 轨道 → CPU Bound（Pass 注册或录制开销过大）；GPU 轨道远长于 CPU 轨道 → GPU Bound（Shader 计算瓶颈）。
RDG 专用轨道
Insights 5.5+ 版本提供了 RDG 专用可视化面板，展示：Pass 依赖 DAG（交互式选中 Pass 高亮其上下游依赖）、每个 Pass 的资源进出列表（列出该 Pass 读取/写入的所有 RDG 纹理和 Buffer）、Transient 分配热图（显示帧内显存分配的峰值时刻和分配/释放事件序列）。通过专用面板可以精确定位资源生命周期异常和屏障风暴。
常见瓶颈定位
Pass 注册 CPU 开销过高 → 合并微小 Pass（将多个相邻小 Pass 的逻辑合并到单个 Shader），或减少 AddPass 调用频率（通过 Indirection Map 批量处理）。屏障风暴 → 检查不必要的 UAV ↔ SRV 转换，考虑将频繁转换的资源的不同使用阶段合并到同一 Pass 内。Transient 分配峰值 → 检查资源生命周期异常（某资源因缺少 QueueTextureExtraction 而生命周期过长）。

调试相关的完整 CVar 列表，配合使用可全面诊断 RDG 运行时行为：

CVar说明
r.RDG.DumpGraph 导出当前帧依赖图为 GraphViz DOT 格式，输出到 Saved/Profiling/RDG/ 目录
r.RDG.DumpGraphUnknowns Dump 时还包含类型未知的资源（External 未注册、参数类型不完整），用于排查依赖缺失
r.RDG.ImmediateMode 立即执行模式：Pass 注册后立即提交执行而非批量延迟。严重降低性能，仅用于调试 Pass 级崩溃时的崩溃点定位
