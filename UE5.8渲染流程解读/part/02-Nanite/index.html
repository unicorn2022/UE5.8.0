1. 管线中的位置

Nanite 在 FDeferredShadingSceneRenderer::Render() 管线中有三个关键介入点，分别对应可见性计算、光栅化和着色三个阶段。

介入点 1 — Phase 3 (L1991-L2017)
可见性查询启动
在视图与可见性阶段，渲染器调用 NaniteVisibility.BeginVisibilityFrame() 开启 Nanite 可见性帧， 随后通过 NaniteVisibility.BeginVisibilityQuery() 提交异步 GPU 裁剪任务。 该查询输出 FNaniteVisibilityResults，记录每个 RasterBin 和 ShadingBin 的可见性。
介入点 2 — Phase 4 (L2511-L2517)
光栅化 (RenderNanite)
在预通道之后，RenderNanite() 被调用以执行 GPU 端的两级 BVH 裁剪 + 光栅化。 产出 VisBuffer64（64 位可见性缓冲）和 DepthBuffer（Nanite 深度）， 供后续 BasePass 着色阶段读取。
介入点 3 — Phase 5 (L2726-L2734 + L3096-L3128)
CS 着色 (BuildShadingCommands + BasePass)
Nanite::BuildShadingCommands() 在预通道后提前执行（CPU 端任务重叠优化）， 构建 Material Bin 着色命令。在 BasePass 阶段，RenderBasePass() 将 Nanite 着色命令传入，通过 Compute Shader 并行走 Material Bin 着色通道，写入 G-Buffer。

2. 核心概念

Nanite 虚拟几何系统的四个核心概念：层级数据结构、GPU 驱动裁剪、VisBuffer 可见性缓冲、延迟材质着色。

GPU 驱动裁剪
两级裁剪管线：CULLING_PASS_OCCLUSION_MAIN 执行视锥 + HZB + 距离 + LOD 裁剪， CULLING_PASS_OCCLUSION_POST 执行二阶段遮挡裁剪。裁剪结果写入 VisibleClustersSWHW 缓冲区。
VisBuffer 64 位编码
64 位可见性缓冲编码每个像素的深度值（32 位 float）和三角形/Cluster ID（29 位）。 光栅化阶段产出 VisBuffer64，着色阶段从中解码三角形信息并计算材质属性。
延迟材质着色
类似延迟渲染的材质着色：先光栅化所有可见三角形的 ID 到 VisBuffer，再通过 ShadingMask 将像素按材质分组（Material Bin），最后用 Compute Shader 分批计算 G-Buffer。
软件/硬件混合光栅化
大三角形走硬件光栅化（HW Raster），小三角形（像素级）走 Compute Shader 光栅化（SW Raster）。 通过 r.Nanite.ComputeRasterization 和 r.Nanite.MinPixelsPerEdgeHW 精细控制切换阈值。

3. 可见性阶段

Nanite 可见性查询在 Phase 3 启动，用于确定哪些 Material Bin（RasterBin + ShadingBin）在当前帧是可见的，从而跳过不可见材质的 GPU 计算。

可见性计算的核心入口位于 DeferredShadingRenderer.cpp::1991-2017，在 BeginInitViews 之前执行。

// DeferredShadingRenderer.cpp L1991-L2017 if (bNaniteEnabled) { TArray<FConvexVolume, TInlineAllocator<2>> NaniteCullingViews; for (int32 ViewIndex = 0; ViewIndex < Views.Num(); ViewIndex++) { FViewInfo& View = Views[ViewIndex]; NaniteCullingViews.Add(View.ViewFrustum); } FNaniteVisibility& NaniteVisibility = Scene->NaniteVisibility[ENaniteMeshPass::BasePass]; const FNaniteRasterPipelines& NaniteRasterPipelines = Scene->NaniteRasterPipelines[ENaniteMeshPass::BasePass]; const FNaniteShadingPipelines& NaniteShadingPipelines = Scene->NaniteShadingPipelines[ENaniteMeshPass::BasePass]; // 开启可见性帧 NaniteVisibility.BeginVisibilityFrame(); // 提交异步可见性查询 NaniteBasePassVisibility.Visibility = &NaniteVisibility; NaniteBasePassVisibility.Query = NaniteVisibility.BeginVisibilityQuery( Allocator, *Scene, NaniteCullingViews, &NaniteRasterPipelines, &NaniteShadingPipelines, InitViewTaskDatas.VisibilityTaskData->GetComputeRelevanceTask() ); }

FNaniteVisibilityQuery 内部结构

可见性查询通过 CPU 端图元遍历，对每个 Nanite 图元检查其引用的 RasterBin 和 ShadingBin 是否在视锥体内。结果存储在两个 TBitArray 中：

// NaniteVisibility.cpp: FNaniteVisibilityQuery TArray<FConvexVolume> Views; // 裁剪视锥体列表 TArray<TAtomic<bool>> RasterBinVisibility; // 每个 RasterBin 的可见性 TArray<TAtomic<bool>> ShadingBinVisibility; // 每个 ShadingBin 的可见性 FNaniteVisibilityResults Results; // 查询完成后的聚合结果 UE::Tasks::FTask CompletedEvent; // 异步任务完成信号

查询结果通过 FNaniteVisibilityResults 暴露关键接口：

RasterBin 查询
IsRasterBinVisible(BinIndex) —— 判断指定 RasterBin 是否在视锥体内。RasterBin 对应光栅化时的材质分组。
ShadingBin 查询
IsShadingBinVisible(BinIndex) —— 判断指定 ShadingBin 是否可见。ShadingBin 对应着色阶段的 Material Bin。
取可见性任务句柄
Nanite::GetVisibilityTask(Query) —— 获取 UE::Tasks::FTask 句柄，等待异步查询完成后才执行光栅化。
CustomDepth 检查
ShouldRenderCustomDepthPrimitive(PrimitiveId) —— 检查指定图元是否需要渲染自定义深度。

4. 光栅化阶段

光栅化是 Nanite 核心步骤，在 GPU 端完成 BVH 裁剪 + 两级光栅化，输出 VisBuffer64 和深度缓冲。

RenderNanite 调用

// DeferredShadingRenderer.cpp L2511-L2517 { Scene->WaitForCacheNaniteMaterialBinsTask(); if (bNaniteEnabled && InViews.Num() > 0) { RenderNanite(GraphBuilder, InViews, LocalSceneTextures, bIsEarlyDepthComplete, InNaniteBasePassVisibility, NaniteRasterResults, PrimaryNaniteViews, FirstStageDepthBuffer); } }

RenderNanite() 内部流程：先调用 Nanite::GStreamingManager.EndAsyncUpdate() 确保流式数据就绪，再按视图遍历执行：

// 每个视图的执行流程 InitRasterContext(GraphBuilder, SharedContext, ViewFamily, InitParams) → IRenderer::Create(GraphBuilder, Scene, View, SharedContext, RasterContext, Configuration) → IRenderer->Draw() // 执行裁剪 + 光栅化

VisBuffer 64 位编码

VisBuffer64 是 Nantie 光栅化的核心输出（FRasterParameters::OutVisBuffer64, RWTexture2D<UlongType>），每个像素 64 位：

低 32 位: 深度 (Depth)
以 float 格式编码的屏幕空间深度值，用于 Early-Z 和后续着色阶段确定像素所属的三角形位置。
高 29 位: 三角形/Cluster ID
标识该像素属于哪个 Cluster 中的哪个三角形，着色阶段据此查找材质属性和顶点数据。
高 3 位: 标志位
包含 Patch ID / Assembly 标志位，用于曲面细分和装配体渲染的像素分类。
光栅化竞争
多个三角形可能覆盖同一像素，通过原子操作（InterlockedMax/Min）写入最近深度的三角形 ID。

裁剪 Pass 类型

Pass 常量值说明
CULLING_PASS_NO_OCCLUSION0无遮挡裁剪：仅执行视锥、HZB、距离、LOD 裁剪
CULLING_PASS_OCCLUSION_MAIN1主遮挡裁剪：前一帧 HZB 遮挡测试

5. 着色阶段

Nanite 着色管线采用延迟材质着色方案：先将 VisBuffer 像素按材质分批（Material Bin），再用 Compute Shader 并行计算 G-Buffer 属性。

BuildShadingCommands —— 提前执行优化

着色命令的构建（CPU 端）在阶段 4 后立即执行（L2726-L2734），而非等到阶段 5。这是 UE5.8 的关键性能优化点，源码注释明确指出：

// DeferredShadingRenderer.cpp L2725-L2734 // Run Nanite compute commands early in the frame to allow some task overlap // on the CPU until the base pass runs. if (bNaniteEnabled && RendererOutput != ERendererOutput::DepthPrepassOnly) { Nanite::BuildShadingCommands(GraphBuilder, *Scene, ENaniteMeshPass::BasePass, Scene->NaniteShadingCommands[ENaniteMeshPass::BasePass]); if (bAnyLumenEnabled && RendererOutput == ERendererOutput::FinalSceneColor) { Nanite::BuildShadingCommands(GraphBuilder, *Scene, ENaniteMeshPass::LumenCardCapture, Scene->NaniteShadingCommands[ENaniteMeshPass::LumenCardCapture]); } }

Material Bin 批处理机制

Nanite 着色使用 ShadingBin 机制将 VisBuffer 中的像素按材质分组：

ShadingBin 扫描
FShadingBinBuildCS 扫描 VisBuffer64，将每个像素分配到对应的 ShadingBin。相同材质的像素被收集到连续内存块中。
ShadingBin Reserve
FShadingBinReserveCS 为每个 ShadingBin 预留内存空间，计算各 Bin 的像素数量和偏移量。
ShadingGroup CS
FShadingGroupCS 将多个 ShadingBin 合并为 ShadingGroup，优化 Compute Shader 的线程组调度粒度。
Compute Shading Dispatch
FNaniteShadingPassParameters 通过 Indirect Draw 方式提交 Compute Shader 着色 Pass，每个 Bin 独立 dispatch。

ShadingMask 与 GBuffer 合并流程

着色管线通过 ShadingMask 中介实现从 VisBuffer 到 G-Buffer 的转换：

// 着色管线数据流 VisBuffer64 (每个像素: Depth|TriangleID|Flags) → ShadeBinning(GraphBuilder, Scene, View, ShadingCommands, RasterResults) // ① 扫描 VisBuffer → 分配 ShadingBin → ShadingMask (像素级材质分类图) // ② 每像素映射到 ShadingBin Index → BuildShadingCommands 构建的命令表 // ③ ShadingBin → PipelineState + Material 参数 → RecordShadingParameters // ④ 为每个 Bin 设置 MaterialCB + Shader 资源 → Compute Shader Dispatch (按 Bin 分组) // ⑤ 并行计算 GBuffer(A/B/C/D/E) 和 LightingChannels
关键机制：ShadingMask 是一张像素级纹理，每个像素存储对应 ShadingBin 的索引。 GBuffer 写入由 Compute Shader 完成，直接通过 SV_DispatchThreadID 坐标读取 ShadingMask， 解码三角形 ID 后计算材质属性并写入 GBuffer RT。这与传统 Mesh Draw 的 Pixel Shader 路径完全不同。

7. 与传统管线交互

Nanite 通过明确定义的接口与 VSM、Lumen、光追等子系统集成。

子系统交互接口数据方向说明
VSM (虚拟阴影贴图) IRenderer::Create 创建阴影上下文 Nanite → VSM Nanite 为 VSM 渲染阴影深度。通过 EOutputBufferMode::DepthOnly 模式仅输出深度，跳过 VisBuffer 编码。
VSM 性能反馈 FConfiguration::bExtractVSMPerformanceFeedback VSM → Nanite VSM 将性能反馈传回 Nanite，调整裁剪和 LOD 策略。
Lumen (全局光照) ENaniteMeshPass::LumenCardCapture Nanite → Lumen Nanite 以专用 MeshPass 渲染 Lumen Mesh Cards，用于构建 Surface Cache。
Lumen AsyncCompute r.Nanite.AsyncRasterization.LumenMeshCards 并行调度 Lumen Mesh Cards 光栅化可通过 AsyncCompute 并行执行。
光追 (Ray Tracing) Nanite::GRayTracingManager Nanite → RT 当 Nanite 页面发生变化时，通知光追管理器重建受影响的 BLAS。
源码：NaniteRayTracing.cpp / NaniteRayTracingASCache.cpp
光追覆盖模式 bHasRayTracedOverlay 控制开关 当光追覆盖模式启用时，跳过 Nanite BasePass（L3092 if (!bHasRayTracedOverlay)）。
BasePass Scene->NaniteShadingCommands[ENaniteMeshPass::BasePass] Nanite → BasePass BasePass 接收预构建的 Nanite 着色命令，通过 RenderBasePass 参数传入。传统材料和 Nanite 材料写入同一组 GBuffer。
SceneTextures NaniteRasterResults.VisBuffer64 / DepthBuffer 共享资源 Nanite 产出的深度和 VisBuffer 被后续阶段（Lumen、后处理）作为输入使用。
集成要点：Nanite 的 GBuffer 与传统的 BasePass GBuffer 完全兼容。 Nanite 通过 Compute Shader 写入的 G-Buffer 格式与传统硬件光栅化 Pixel Shader 写入的格式完全一致， 因此延迟光照阶段无需区分几何体来源——GBuffer 统一采样器同时适用于传统和 Nanite 路径。
