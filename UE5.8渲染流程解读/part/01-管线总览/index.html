Render 函数定位

FDeferredShadingSceneRenderer::Render() 是 UE5.8 延迟渲染管线的总入口函数，位于 DeferredShadingRenderer.cpp 第 1826 行，横跨约 2600 行代码到第 4456 行结束。该函数以 RDG（Render Dependency Graph, 渲染依赖图）为全帧资源调度中枢，将一帧渲染划分为九个逻辑阶段。

Render 函数接受 FRDGBuilder& GraphBuilder 和 FSceneRenderUpdateInputs* 两个核心参数，前者负责构建渲染依赖图并自动管理临时资源的生命周期，后者携带当前帧的更新输入数据（场景更新、渲染器组等）。

关键数据流特征：Render 函数按严格顺序组织阶段，前序阶段产出的 GPU 资源（深度、G-Buffer、场景颜色等）通过 RDG 的 FRDGTextureRef 引用传递给后续阶段。同时，大量独立的 CPU 子任务（光源收集、Nanite 着色命令构建等）通过 TaskGraph 和 UE::Tasks 异步并行执行，以最大化 CPU 利用率。

Render 函数核心参数与概念

1. FRDGBuilder — RDG 渲染图构建器

FRDGBuilder 是 UE5 Render Dependency Graph（渲染依赖图）的核心调度器，定义在 RenderGraphBuilder.h:50。它负责管理所有渲染 Pass 的注册、临时资源的生命周期、Pass 间依赖关系和屏障转换。

生命周期：每次调用 Render() 时在栈上临时创建（SceneRenderBuilder.cpp:873），离开作用域时析构函数自动调用 Execute() 提交所有已注册的 Pass。创建时接收 FRHICommandListImmediate& 引用，多个 GraphBuilder 串行实例共享同一个命令列表。

FRDGBuilder 核心能力一览：
  • 资源注册与创建：RegisterExternalTexture/Buffer() 将外部 RHI 资源注册到图；CreateTexture/Buffer() 创建图内临时资源并自动管理生命周期
  • Pass 管理：AddPass() 向图中添加渲染 Pass（lambda + SHADER_PARAMETER_STRUCT），RDG 自动推导资源依赖、插入屏障、剔除无用 Pass
  • Blackboard 黑板：FRDGBlackboard（第 433 行）提供类型安全的键值存储，在 Pass 间共享临时数据（如光追的 LightFunctionMap）
  • 异步任务：AddSetupTask() / AddCommandListSetupTask() 在 RDG 编译前并行执行 CPU 准备任务
  • 资源提取：QueueTextureExtraction() / QueueBufferExtraction() 将 GPU 资源提取为池化指针，使其在图执行后仍存活

OpenGL 开发者可类比：FRDGBuilder 相当于一个"自动管理所有 glBindTexture + glUseProgram + glFramebufferTexture2D 状态切换和依赖排序的声明式系统"。你声明"这个 Pass 读写这些资源"，RDG 自动处理余下的一切。

2. FSceneRenderUpdateInputs — 帧渲染输入数据包

定义在 SceneRendering.h:2841，是一个轻量级的输入上下文结构体，描述"这一帧渲染什么场景、用哪个平台、渲染哪些视图"：

字段 类型 说明 OpenGL 近似类比
ShaderPlatformEShaderPlatform着色器平台（如 SM6/D3D12）GL Context 版本（GL 4.6 Core Profile）
FeatureLevelERHIFeatureLevel::Type特性级别（SM5/SM6/ES3_1）检查扩展可用（GL_ARB_compute_shader）
SceneFScene*场景对象（所有模型/光源/材质）无对等物（GL 不管场景管理）
FXSystemFFXSystemInterface*粒子/VFX 系统接口无对等物
GlobalShaderMapFGlobalShaderMap*全局着色器缓存（见下文）所有 glCreateProgram() 编译好的程序集合
RenderersTConstArrayView<FSceneRenderer*>本帧要执行的渲染器数组无对等物（多个渲染循环的调度器）
ViewFamiliesTConstArrayView<FViewFamilyInfo*>所有视图族VR 左右眼视图族（GL 需手动管理）
CommonShowFlagsESceneRenderCommonShowFlags跨视图族公共 ShowFlags一堆 bool bDrawShadows; bUseAO; 标志

核心区别：OpenGL 是命令式 API——手动 glBindTexture、glUseProgram、glDrawElements。而 UE 是数据驱动的——打包好"渲染什么"的输入数据，由 FRDGBuilder + DeferredShadingRenderer 自动调度数百个 Pass。

3. FSceneRenderer 层级 — "一整帧"不是"一次 Draw"

FSceneRenderUpdateInputs::Renderers 数组中每个元素是 FSceneRenderer*，这是一个完整帧渲染器，粒度为"一整帧"，而非单次 Draw。类层级：

类 定位 每次 Render() 产生的 Pass 数
FSceneRenderer虚基类—
FDeferredShadingSceneRenderer高端桌面/主机延迟渲染器数百个 RDG Pass
FMobileSceneRenderer移动端前向/延迟渲染器数十到上百个 RDG Pass
FCustomRenderPassBase自定义渲染通道（场景捕获等）可变

层级关系：1 个 FSceneRenderer → 数百个 RDG Pass（通过 GraphBuilder.AddPass 添加）→ 数千个 FMeshDrawCommand（每次对应一次 RHI Draw）→ 最终映射到 D3D12/Vulkan/OpenGL 的 draw 命令。通常 Renderers 数组中只有 1 个元素（即 FDeferredShadingSceneRenderer），多个 Renderer 仅出现在启用 Custom Render Pass 时。

如何在不修改 Render() 的情况下添加自定义 Pass？UE 提供了场景扩展（Scene Extensions）机制：
  • Scene Extensions Renderer（ISceneExtensionRenderer）— 在 PreInitViews/UpdateViewData/PreRender/PostRender 等回调注入 Pass，Render() 中第 2039/2326/2330/2439 行已有调用点
  • Scene View Extension（FSceneViewExtensionBase）— 挂钩 PreRenderViewFamily/PostRenderViewFamily 等
  • Post Process Material — 蓝图/材质方式在后处理链中插入效果，完全无需 C++
  • Custom Render Pass（FCustomRenderPassBase）— Render() 第 2615 行已有完整分发循环
  • 直接修改 Render() — 侵入式，仅引擎核心功能（Nanite/Lumen/MegaLights）使用

Epic 的插件（nDisplay/HairStrands/Water 等）几乎全部使用前四种非侵入方式。

4. FGlobalShaderMap — 全局着色器缓存

定义在 GlobalShader.h:181，是一个平台级预编译着色器缓存，存储引擎内置的"全局着色器"——不依赖任何材质或网格的基础 Shader。引擎启动时已编译完毕，不可运行时动态添加。

对比维度 FGlobalShaderMap 其他 Shader 体系
管理内容引擎内置工具 Shader（全屏三角形 VS、纹理拷贝 CS、清屏、降采样、Mip 生成等）—
材质 Shader不存FMaterialShaderMap 按材质按需编译
网格相关 Shader不存FMeshMaterialShaderMap
光追 Shader不存FRayTracingShaderMap
实例数每个 ShaderPlatform 一个—
注册机制IMPLEMENT_GLOBAL_SHADER 宏（编译期注册）—

存储结构：FGlobalShaderMap → 每个 Section 对应一个 .usf 文件 → FShaderType → 编译好的 Shader 字节码 + 多 Permutation 索引。

5. 带材质物体的着色器代码 — 从节点图到 GPU 执行

当渲染一个带材质的不透明物体时，着色器代码经历了四个阶段的生成与绑定：

阶段 输入 核心组件 产出
1. HLSL 生成 材质编辑器节点图 FHLSLMaterialTranslator::GetMaterialShaderCode()
遍历每个 UMaterialExpression 节点生成对应 HLSL 代码片段		 CalcPixelMaterialInputs() 函数体
2. 模板合并 MaterialTemplate.ush
+ BasePassPixelShader.usf		 材质生成的 HLSL 注入 MaterialTemplate.ush，再被 #include 到 BasePassPixelShader.usf 完整的 PS 入口代码（MainPS）
3. 编译缓存 合并后 HLSL 平台 Shader 编译器（dxc/glslangValidator/Metal） FMaterialShaderMap
4. 运行时绑定 FMaterialShaderMap 缓存 FMeshDrawCommand 最终的 RHI Draw → GPU 执行

合并后的最终 PS 代码结构：

// === 引擎框架部分 (BasePassPixelShader.usf) ===
#include "Common.ush"
#include "DeferredShadingCommon.ush"
// ...各种引擎头文件和UB定义...

// === 材质生成的代码 (MaterialTemplate.ush) ===
void CalcPixelMaterialInputs(...)
{
    // ← FHLSLMaterialTranslator 从材质图自动生成的 HLSL
    float2 UV = Parameters.TexCoords[0].xy;
    float4 TexColor = Texture2DSample(Material_Texture2D_0, ...);
    PixelMaterialInputs.BaseColor = TexColor.rgb * Material_VectorExpressions[0].rgb;
    PixelMaterialInputs.Roughness = 0.5;
}

// === 引擎框架部分 (BasePassPixelShader.usf) ===
void FPixelShaderInOut_MainPS(...)
{
    FMaterialPixelParameters MaterialParameters = GetMaterialPixelParameters(...);
    FPixelMaterialInputs PixelMaterialInputs;
    CalcPixelMaterialInputs(MaterialParameters, PixelMaterialInputs); // ★ 调用材质代码
    // 写入 GBuffer...
}

// === 引擎包装 (PixelShaderOutputCommon.ush) ===
void MainPS(...) // ← D3D/Vulkan 的实际入口点
{
    FPixelShaderInOut_MainPS(...);
}

FMeshDrawCommand 运行时绑定：渲染时 CPU 不关心 HLSL 源码。每个 FMeshDrawCommand 持有编译好的字节码指针（VS + PS）和预编译的 PSO（Pipeline State Object），执行时直接提交 SubmitDrawCall(MeshDrawCommand)。RHI 层转换此调用为 D3D12 DrawIndexedInstanced / Vulkan vkCmdDrawIndexed / OpenGL glDrawElements。

源码行级解读

L1834-1846: Views 遍历 + ViewState 关联

Views 是 FSceneRenderer 基类成员（SceneRendering.h:2187），类型为 TArray<FViewInfo>。每个 FViewInfo 对应一个"渲染视点"：主相机、SceneCapture 2D/3D、VR 左右眼、自定义渲染 Pass。遍历所有 View，将每个 View 的 ViewState（跨帧持久化状态，存储 TAA 历史、时间戳等）关联到当前 Scene。这是每帧渲染最开始的簿记工作。

L1848-1856: 光追调试可视化自动激活

GetRayTracingVisualizationData() 返回全局单例 FRayTracingVisualizationData，管理约 40 种光追调试模式（Barycentrics、BaseColor、WorldNormal、InstanceOverlap 等）。HasOverrides() 检查控制台变量 r.RayTracing.Visualize 是否已设值。如果检测到覆盖则自动调用 SetRayTracingDebug(true)，免去开发者手动切换到 viewmode RayTracingDebug 的步骤。这是一种调试流程的 UX 优化。

L1857: GetRendererOutput() — 渲染输出模式

ERendererOutput 枚举有两个值：DepthPrepassOnly（仅渲染深度预通道，SceneCapture 仅需深度时走快速路径）和 FinalSceneColor（完整渲染管线）。判断逻辑：非 SceneCapture → 完整管线；SceneCapture 仅需深度 + 优化开关 → 深度预通道快速路径；否则完整管线。这个标志在后续大量 if (RendererOutput == ERendererOutput::FinalSceneColor) 分支中使用。

L1859-1860: bNaniteEnabled + bHasRayTracedOverlay

bNaniteEnabled 是帧级全局判断，贯穿整个 Render()，决定是否执行 Nanite VisBuffer 裁剪/光栅化、深度预通道是否跳过 Nanite 几何、VSM 是否使用 Nanite 快速路径等。bHasRayTracedOverlay 检测是否需要渲染光追调试叠加层。

L1862-1865: RenderCapture 帧捕获钩子（!UE_BUILD_SHIPPING）

通过控制台变量 r.CaptureNextDeferredShadingRendererFrame 控制：-1 禁用（默认），0 立即捕获下一帧，>0 延迟 N 帧后捕获。在 RDG 中插入标记使 RenderDoc/PIX 等工具能定位捕获区域。构造后自动递减为 -1 防止重复触发。
关于 Shipping：UE 有三种构建配置 — Debug（全调试符号）、Development（部分优化 + 日志/Console 命令）、Shipping（最终发行版，全优化，剥离所有调试功能）。#if !UE_BUILD_SHIPPING 包裹的代码在 Shipping 中完全不编译，零开销。

源码行级解读

RAII 资源生命周期管理对象

• FRDGExternalAccessQueue — 延迟执行队列，收集 Render 线程外部对 RDG 资源的访问请求，最后在 Submit() 时统一提交，确保 GPU 同步点正确
• FVirtualTextureUpdater — 虚拟纹理帧级更新句柄，TUniquePtr 确保提前 return 不泄漏
• FLumenSceneFrameTemporaries — Lumen 帧临时数据，根据 Views 数量预分配，衔接后续 Lumen 任务链
• FGPUSceneScopeBeginEndHelper — 纯 RAII 守卫：构造时 GPUScene.BeginRender()，析构时 GPUScene.EndRender()，无论正常返回还是提前退出都对

虚拟纹理（Virtual Texture）帧级更新

RVT（Runtime Virtual Texture）用于将复杂地形材质"烘焙"成纹理按需流送。UseVirtualTexturing() 检查平台支持。保护条件 RendererOutput != DepthPrepassOnly 跳过深度捕获路径（深度预通道不需要 VT）。BeginUpdate() 启动更新并收集需要刷新的 VT 页面；VirtualTextureFeedbackBegin() 将上一帧 GPU 的页面需求反馈取回，驱动本帧优先级和分配。后续有 EndUpdate() 和 FinalizeRequests() 完成整个更新周期。

CommitFinalPipelineState() + GSystemTextures

CommitFinalPipelineState() 遍历所有 Renderer（分屏多玩家可能有多个），计算并冻结本帧所有相互依赖的特性开关（光追/Lumen/VSM/Nanite 最终状态、各 Pass 是否执行）。冻结后整帧查询结果一致，避免中途状态变化导致逻辑矛盾。
GSystemTextures.InitializeTextures() 确保内置系统纹理（White/Black/Gray/DefaultNormal 等）已创建，幂等操作，已存在则不做事。

UpdateLightFunctionAtlasTask — 光照函数图集异步更新

光照函数（Light Function）是贴在光源上的材质，用于生成复杂光影图案（投影纹理、树影斑驳、舞台灯光等）。这些材质被烘焙到共享图集中。
LaunchSceneRenderTask 启动渲染线程异步任务更新脏的 Light Function 材质，无需等待可见性结果。返回的 UpdateLightFunctionAtlasTask 句柄后续传给 GatherAndSortLights 作为前置依赖，确保光源收集排序等待图集更新完成。

阴影 + VSM + Lumen + Nanite 可见性（FShadowSceneRenderer 作用域内）

1. 全局距离场视点更新：UpdateGlobalDistanceFieldViewOrigin() 更新观察原点 + 相机速度偏移（CameraVelocityOffset），必须在 Lumen 之前执行。
2. VSM 初始化：VirtualShadowMapArray.Initialize() 决定本帧是否启用 VSM 并按需分配 Shadow Page。
3. Lumen 场景任务启动：BeginUpdateLumenSceneTasks() + BeginGatherLumenLights()，后者等待 UpdateLightFunctionAtlasTask。
4. Nanite 可见性预查询：只对 ENaniteMeshPass::BasePass 做 CPU 端预裁剪。详见下方"Nanite 可见性 CPU 预裁剪"专题。

专题：Nanite 可见性 CPU 预裁剪（BeginVisibilityQuery）
ENaniteMeshPass 枚举（PrimitiveSceneInfo.h:246）

Nanite 支持的三种 Pass：BasePass（GBuffer 材质着色）、LumenCardCapture（Lumen 表面缓存捕获）、MaterialCache（材质缓存预评估）。每种 Pass 有独立的三组场景级数组：

NaniteVisibility[Pass] 该 Pass 的 CPU 端可见性状态
NaniteRasterPipelines[Pass] 光栅化管线配置（硬件/软件/混合调度、裁剪参数、输出模式）
NaniteShadingPipelines[Pass] 材质着色管线数据（ShadingBin 分类、CS Dispatch 参数、Material Slots）
BeginVisibilityQuery() 执行流程

1. 前置检查：GNaniteMaterialVisibility == 0 或未调用 Begin 或视图为空 → 直接返回 nullptr
2. 创建 FNaniteVisibilityQuery，包含所有视锥体引用、RasterBin/ShadingBin 可见性位数组
3. 启动 UE::Tasks 任务（可异步），前置依赖为 GetCacheNaniteMaterialBinsTask() + ComputeRelevanceTask
4. 在任务内调用 PerformNaniteVisibility()： 

for (每个已注册的Nanite图元) {
    // 1. 视锥体裁剪 — 图元包围盒 ∩ 所有视锥体
    bPrimitiveVisible = IsNanitePrimitiveVisible(Query, SceneInfo);
        // → 取 PrimitiveBounds[Index].BoxSphereBounds
        // → View.IntersectBox(Origin, BoxExtent)

    if (bPrimitiveVisible) {
        // 2. 标记该图元所属 RasterBin 为可见
        for (RasterBin : References.RasterBins)
            Query->RasterBinVisibility[BinIndex] = true;

        // 3. 标记该图元所属 ShadingBin 为可见
        for (ShadingBin : References.ShadingBins)
            Query->ShadingBinVisibility[BinIndex] = true;
    }
}
裁剪粒度：Bin 级别，不是 Cluster 级别

这里裁剪的目标不是三角形或 Cluster，而是 RasterBin / ShadingBin：
• RasterBin：同一光栅化配置的图元集合，GPU 光栅化时同一 Bin 用同一 Pass
• ShadingBin：同一材质着色配置的图元集合，CS 着色时同 Bin 一起 dispatch
如果一个图元不可见，它对应的 RasterBin 和 ShadingBin 标记为 false，GPU 端后续裁剪时整个 Bin 的 Cluster 都可跳过，大幅减少 GPU 裁剪输入规模。

"图元"（Primitive）在 UE 中的含义

图元 = UPrimitiveComponent 的渲染表示 FPrimitiveSceneInfo。一个 Static Mesh Actor、一个 Skeletal Mesh 角色、一个 Landscape 地形、一个 ISM 组件各自对应一个图元。关键区别：ISM 内部可能有 1000 个实例，但这里只拿图元整体包围盒做测试，不裁剪单个实例。实例级精细裁剪在 InstanceCullingManager 的 GPU 端完成。策略是"剪得保守，宁可放进来也不漏掉"。

异步并行执行

UE5.8 在 Render 函数中大量使用 TaskGraph 和 AsyncCompute 来最大化 CPU/GPU 利用率。以下标注了关键的异步并行点。

阶段 3 — CPU 异步
光源收集与排序
GatherAndSortLights 以 UE::Tasks::TTask 异步执行，在可见性计算期间并行收集场景中所有光源、按类型/重要性排序、构建 SortedLightSet。阶段 6 调用 GatherAndSortLightsTask.Wait() 等待结果。
阶段 3 — CPU 异步
光照函数图集更新
UpdateLightFunctionAtlasTask 以 LaunchSceneRenderTask 启动 TaskGraph 任务，异步更新光照函数图集纹理。与可见性计算并行，不阻塞主 Render 线程。
阶段 4-5 间隙 — CPU 异步
Nanite 着色命令构建（提前执行）
Nanite::BuildShadingCommands（L2729）在预通道完成后立即执行，而非等到 BasePass 阶段。源码注释明确："Run Nanite compute commands early in the frame to allow some task overlap on the CPU until the base pass runs." 这是 CPU 端任务重叠的典型优化。
阶段 6 前 — GPU AsyncCompute
Lumen 间接光照异步调度
DispatchAsyncLumenIndirectLightingWork（L3287）将 Lumen GI 计算（Surface Cache 更新 + Screen Probe Gather）以 AsyncCompute 管线提前发出，与后续的图形管线 Pass 重叠执行，最大化 GPU 利用率。
阶段 6 前 — GPU AsyncCompute
MegaLights 体积 Pass 异步
DispatchAsyncMegaLightsVolumePasses（L3088）以 AsyncCompute 执行 MegaLights 的体积光照注入 Pass。与 HZB 构建和 Nanite 流式加载等操作并行，减少 GPU 空闲时间。
阶段 6 前 — GPU AsyncCompute
体积云异步渲染
RenderVolumetricCloud（L3299-3306）在 bAsyncComputeVolumetricCloud 为 true 时以 AsyncCompute 路径执行。注释注明："Kick off volumetric clouds async dispatch after Lumen... so should run second and overlap opaque work after Lumen async is done."
