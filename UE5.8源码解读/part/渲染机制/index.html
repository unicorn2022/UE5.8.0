1. 概述 & 定位

1.1 核心问题

渲染机制模块回答一个根本性问题：场景中的几何体如何变成 GPU 上的 DrawCall？ 这个问题在 UE5.8 中被拆分为四个子系统协同解决：

场景渲染器 (FSceneRenderer)

统筹整帧渲染，作为"总指挥"协调各子系统按序执行。派生 FDeferredShadingSceneRenderer（桌面/主机延迟渲染）和 FMobileSceneRenderer（移动端），按平台分流。

可见性计算 (SceneVisibility)

决定哪些物体需要渲染——视锥裁剪、遮挡剔除、距离裁剪、LOD 选择。采用多线程异步任务图并行执行，支持 HZB 和硬件遮挡查询两种模式。

Mesh Pass 处理 (FMeshPassProcessor)

将可见物体制成 FMeshDrawCommand，按 Pass 类型分组。每个 Pass 定义为不同的渲染目标配置（DepthPass、BasePass、ShadowDepth...共 39 种）。

GPU 场景数据 (FGPUScene)

在 GPU 可访问的缓冲区中维护图元+实例数据，使得 Shader 可自行从三角形 ID 反查图元属性，避免了传统 UBO 逐 Draw 切换的开销。

1.2 静态/动态渲染双路径全景

图 1-1: 几何体到 DrawCall 的静态/动态双路径全景

核心设计哲学：静态材质 = 预缓存 + 快速筛选，动态材质 = 按需构造 + 灵活性优先。 两条路径在 InstanceCulling 阶段汇合，统一发往 GPU。GPU Scene 提供统一的数据查询接口，让 Shader 侧无需区分静态/动态图元。

2.2 FPrimitiveSceneProxy — 代理模式

FPrimitiveSceneProxy 是渲染线程侧对应游戏线程 UPrimitiveComponent 的"镜像"对象。 其核心设计动机：

线程安全

游戏线程 (GT) 可随时修改组件的属性。Proxy 在创建时快照所有渲染所需数据到渲染线程侧，二者此后独立。GT 修改通过 ENQUEUE_RENDER_COMMAND 异步通知 Proxy。

接口统一

无论底层是静态网格、骨骼网格、粒子系统还是程序化几何，渲染管线只需操作 FPrimitiveSceneProxy*，通过虚函数获取 MeshBatch、统一参数等。

生命周期解耦

Proxy 生命周期由渲染线程引用计数控制。GT 可以销毁 Component，而 Proxy 会等渲染线程完成最后一帧引用后才释放。

GPU Scene 数据源

Proxy 是填充 FPrimitiveUniformShaderParameters 的权威来源，包含世界变换、包围盒、LOD 参数等 Shader 所需的核心数据。

2.3 FMeshPassProcessor — Mesh Pass 处理器

FMeshPassProcessor 是传统 Mesh 管线的核心抽象，负责将 FMeshBatch 转换为 FMeshDrawCommand。 关键设计要点：

FMeshBatch

表示一个网格的渲染批次，包含顶点工厂、材质、索引缓冲区、LOD 信息等。是所有 Pass 的通用输入。

FGraphicsMinimalPipelineStateInitializer

管道状态最小集合（着色器绑定、混合、光栅化、深度模板），不含渲染目标格式。使 Pass 可以共享 PSO 缓存而无需关心 RT 配置变化。

BuildMeshDrawCommands()

模板方法，组合 Pass 特定的 Shader 变体与 MeshBatch 数据，产出可执行的 FMeshDrawCommand，支持 PSO 预缓存。

SortKey 机制

每个 FMeshDrawCommand 携带排序键（含深度、材质、顶点工厂等），GPU 侧按 SortKey 排序以最小化状态切换。

2.5 自定义 MeshPass 注册与执行流程

以下展示在 UE5.8 中添加自定义 MeshPass（例如环境光遮蔽通道）的标准流程：

// ========== 步骤 1: 定义 Pass 类型 ==========
// 在 MeshPassProcessor.h 的 EMeshPass::Type 枚举中添加新项
namespace EMeshPass {
    enum Type : uint8 {
        // ... 现有 Pass ...
        MyCustomAOPass,  // 新增
        Num,
    };
}

// ========== 步骤 2: 实现 FMeshPassProcessor 派生类 ==========
class FMyCustomAOMeshProcessor : public FMeshPassProcessor
{
public:
    FMyCustomAOMeshProcessor(const FScene* Scene, const FSceneView* InView,
        FMeshPassDrawListContext* InDrawListContext)
        : FMeshPassProcessor(EMeshPass::MyCustomAOPass,
            Scene, ERHIFeatureLevel::SM6, InView, InDrawListContext)
    {}

    virtual void AddMeshBatch(const FMeshBatch& MeshBatch,
        uint64 BatchElementMask,
        const FPrimitiveSceneProxy* PrimitiveSceneProxy,
        int32 StaticMeshId = -1) override
    {
        // 1. 判断材质是否支持此 Pass
        const FMaterialRenderProxy* MaterialProxy = MeshBatch.MaterialRenderProxy;
        const FMaterial& Material = MaterialProxy->GetMaterialWithFallback(
            FeatureLevel, MaterialProxy);

        if (Material.GetShadingPath() == EShadingPath::Deferred)
        {
            // 2. 构建 Shader 绑定
            FMyCustomAOShaders Shaders;
            if (!Material.TryGetShaders(MyCustomAOShaderType,
                MeshBatch.VertexFactory->GetType(), Shaders)) return;

            // 3. 调用模板方法构建 FMeshDrawCommand
            BuildMeshDrawCommands(
                MeshBatch, BatchElementMask,
                PrimitiveSceneProxy,
                *MaterialProxy, Material,
                PassDrawRenderState, Shaders,
                FM_Solid, CM_CW,
                FMeshDrawCommandSortKey::Default,
                EMeshPassFeatures::Default,
                ShaderElementData);
        }
    }
};

// ========== 步骤 3: 注册 MeshPass 处理器 ==========
extern void RegisterMyCustomAOPass()
{
    FPassProcessorManager::RegisterPass(
        EMeshPass::MyCustomAOPass,
        EMeshPassFlags::MainView,  // 标志: 主视图 Pass
        [](const FScene* Scene, const FSceneView* View,
           FMeshPassDrawListContext* Ctx) -> FMeshPassProcessor* {
            return new FMyCustomAOMeshProcessor(Scene, View, Ctx);
        });
}

3.2 可见性裁剪三板斧

CPU 视锥裁剪 (Frustum Cull)

通过视锥 6 平面与包围盒相交测试，快速剔除视锥外物体。CVar: r.Visibility.FrustumCull.Enabled。 支持八叉树加速 (UseOctree=1) 和球体预测试 (UseSphereTestFirst=1) 优化。 可通过 NumPrimitivesPerTask 控制并行粒度。

GPU 遮挡剔除 (Occlusion Cull)

两套系统可选：HZB 遮挡 (r.HZBOcclusion 1) 使用上一帧深度缓冲做层次 Z 测试， 或硬件遮挡查询 (0) 用 GPU Query。HZB 默认模式更省 GPU/CPU。 支持 r.AllowSubPrimitiveQueries 对 HISM 子图元做精细化查询。

CPU 距离裁剪 (Distance Cull)

基于 MinDrawDistance / MaxDrawDistance 做距离筛选。与 ViewDistanceScale 联动。 支持平滑淡入淡出 (Time-based LOD Fade)，由 r.LODFadeTime 控制过渡时间。

可见性任务调度由 r.Visibility.TaskSchedule 控制：0 为 RenderThread 主导 + 并行辅助， 1 为异步 Task Graph (默认，若平台支持)。

3.3 VisibleMeshCollect → MeshPassSetup → Dispatch

可见性结果为每个 View 生成可见图元集合。接下来进入 Mesh Pass 分发链：

  1. 分配 ViewCommands — 为每个 View 创建 FViewCommands 结构，为每种 Pass 分配可见网格列表的存储空间。
  2. 遍历 StaticMesh 可见列表 — 对每个 EMeshPass::Type，将可见的静态网格 DrawCommand 插入对应 Pass 的队列。静态路径直接复用缓存，不需要重新构建 DrawCommand。
  3. 调用 SetupMeshPass() (SceneRendering.cpp:5010) — 遍历 39 种 Pass，为标记了 MainView 标志的 Pass 创建对应的 FMeshPassProcessor 派生实例，传入动态网格的 FMeshBatch 调用 AddMeshBatch()。
  4. 构建 InstanceCulling 上下文 — 将 DrawCommand 数组传给 FInstanceCullingContext，在 GPU 侧做实例级可见性裁剪（ISMP/HISM）。
  5. Dispatch — 各 SubPass (DepthPass, BasePass, Velocity...) 通过 SubmitMeshDrawCommands() 或 FParallelMeshDrawCommandPass 向 GPU 提交。

4. 设计决策 & 权衡

4.1 静态 vs 动态渲染路径分离

为什么需要两条路径？

UE5 场景中绝大多数是静态几何体（StaticMesh），它们的材质+顶点工厂组合在各帧间不会变化。动态物体（粒子、骨骼动画）每帧都不同。合并处理会让所有物体承担动态路径的开销，而分离后静态物体只需一次缓存 + 每帧筛选。

静态路径优势

MeshDrawCommand 在注册时预构建并缓存，RHI 资源指针固定。每帧仅需从可见性列表中读取索引，跳过整个 AddMeshBatch 流程。排序键也缓存，减少 CPU 排序开销。对 ISM/HISM 特别有效——GPU 侧 InstanceCulling 可批量裁剪。

静态路径局限

MeshDrawCommand 对象占用内存（每 LOD 每 Section 一个）。当材质参数或顶点工厂配置变化时，需要重新缓存。对于频繁变化的场景（大量动态材质实例），收益递减。

4.2 Proxy 模式的优劣

优势

  • 线程安全：GT 和 RT 完全解耦，无锁并发
  • 多态统一：StaticMesh/SkeletalMesh/Particle/ISM 对渲染器都是 FPrimitiveSceneProxy*
  • 数据快照：RenderState 创建时就确定了整个渲染帧的参数，避免竞态
  • 生命周期明确：引用计数管理，RenderState Dirty 标记触发重建

代价

  • 内存双倍：每个组件在 GT (UPrimitiveComponent) 和 RT (FPrimitiveSceneProxy) 各存一份渲染相关数据
  • 同步延迟：GT 修改须通过 ENQUEUE_RENDER_COMMAND，至少一帧延迟
  • 虚函数开销：GetDynamicMeshElements() 等核心调用是虚函数，对每一可见图元调用一次
  • 复杂度：新增图元类型需要同时维护 Component + Proxy 两个类

4.3 MeshBatch 复用策略

StaticMeshId 机制

每个 StaticMesh 的 Section 有唯一的 StaticMeshId（在场景侧分配）。可见性系统通过 FViewCommands 维护每个 Pass 的可见 StaticMeshId 集合。Move 操作比 Copy 快数百倍——FMeshDrawCommand 可以直接指针引用缓存版本。

动态分配器 (FMeshElementCollector)

动态路径使用 FMeshElementCollector，基于帧分配器 (Frame Allocator) 快速分配 FMeshBatch。分配的内存随渲染器生命周期释放，无需逐对象 delete。但 FMeshDrawCommand 仍需每帧构建。

SortKey 设计

FMeshDrawCommandSortKey 包含材质 ID、顶点工厂 ID、深度、Shader 绑定等。静态路径下 SortKey 与 DrawCommand 一同缓存。GPU 侧按 SortKey 排序可减少 PSO 切换——相邻 Draw 共享同一 Pipeline State。

4.4 GPU Scene 的数据粒度选择

为什么是 Primitive + Instance 两级？

图元 (Primitive) 数据如包围盒、LOD 参数在各实例间共享（通常几十字节），实例 (Instance) 数据如世界变换、自定义数据（通常 128-256 字节）每实例独立。分离后 HISM 的 10000 实例只存一份图元数据。通过 PrimitiveId 从 Instance 索引反查 Primitive 数据。

Buffer 布局

GPUSceneInstanceSceneData (StructuredBuffer float4) — 每实例 8 个 float4。
GPUSceneInstancePayloadData (StructuredBuffer float4) — 可变额外数据。
GPUScenePrimitiveSceneData (StructuredBuffer float4) — 每图元固定大小。
GPUSceneLightmapData — Lightmap UV/索引。
GPUSceneLightData (ByteAddressBuffer) — 光源数据。使用分块 (Tile) 策略，r.GPUScene.InstanceDataTileSizeLog2 默认 12 (4096 实例/块)。

动态图元管理

FGPUScenePrimitiveCollector 通过 Add() 收集动态图元，Commit() 分配 GPU 空间并上传。空间使用 FSpanAllocator 管理，支持回缩 (GrowOnly=0，5.3+ 默认)。

6. UE5.8 演进变化

GPU Scene 增强
  • Reserved Resources 默认启用 (r.GPUScene.UseReservedResources=true) — 使用虚拟地址预留 + 物理页提交模式，减少 buffer 重建成本
  • 分块 (Tile) 策略成熟 (r.GPUScene.InstanceDataTileSizeLog2=12) — 支持海量实例场景下的部分更新
  • FGPUSceneWriterUniformParameters 引入单独 UAV 写入路径 — Compute Shader 可直接写 GPU Scene，用于 InstanceCulling Feedback 和 Post-Transform 缓存
  • ViewDistanceScale 联动 (r.GPUScene.ViewDistanceScaleWorkaround=true) — 缩放变化时自动全量重上传以保证数据一致性
Instance Culling 集成深化
  • GPU 驱动裁剪 — FInstanceCullingContext 在 GPU 侧执行实例级可见性，CPU 不再逐个检查实例
  • Occlusion Feedback 集成 — 渲染反馈回 InstanceCulling，实现闭环剔除 (r.OcclusionFeedback.Enable，当前仅移动端)
  • Sub-Primitive Query — HISM 支持子图元级别的遮挡查询 (r.AllowSubPrimitiveQueries=1)
  • ISM 间接绘制 — FInstanceCullingDrawParams 生成 GPU Indirect Draw 参数，减少 CPU 提交次数
Nanite 可见性合并
  • Scene Culling 统一入口 — FSceneCullingRenderer 作为 Scene Extension，统一管理 Nanite 和传统几何的 GPU 裁剪
  • Nanite Visibility 预处理 — FNaniteVisibility::BeginVisibilityQuery() 在 CPU 端做预裁剪，结果合并到可见性流程
  • NaniteMeshPass (EMeshPass #31) — 作为 Nanite 几何的迷你深度通道，与传统 MeshPass 并行处理
  • Lumen Card 捕获双路径 — LumenCardCapture(#26) 传统 + LumenCardNanite(#27) Nanite，统一外观
并发与性能优化
  • SceneRenderingAllocator 全面使用 — 帧分配器替代传统 new/delete，减少分配开销
  • UE::FMutex 替代 FCriticalSection — 用于 DispatchedShadowDepthPasses 等热路径的细粒度锁
  • 可见性任务图优化 — r.Visibility.TaskSchedule=1 异步任务图成为默认（受平台支持时）
  • DynamicMeshElements 并行化 — r.Visibility.DynamicMeshElements.NumMainViewTasks 控制动态网格收集的并行任务数（默认 4）
  • PSO 预缓存 — IPSOCollector 接口与 CollectPSOInitializers() 结合，FMeshPassProcessor 继承此接口，实现材质加载期间的 PSO 预热
