实时光线追踪系统

源码根路径：Engine/Source/Runtime/Renderer/Private/RayTracing/  |  核心文件：RayTracing.cpp · RayTracingMaterialHitShaders.cpp · RayTracingShadows.cpp · RayTracingScene.cpp · RayTracingShaderBindingTable.cpp · D3D12RayTracing.cpp

UE5.8.0 的实时光线追踪系统通过 DXR / VulkanRT 的完整封装，将硬件光追能力深度集成到延迟渲染管线中，为阴影、反射、全局光照（GI）和路径追踪提供统一的底层支撑。它不是一个孤立的渲染特性，而是一套横跨 RHI、引擎层和应用层的三重抽象体系——从 D3D12 的底层 BuildRaytracingAccelerationStructure 调用，到引擎层的 SBT（Shader Binding Table）管理和材质 Hit Shader 编译链，再到阴影/反射/Lumen/GI 等应用层功能，形成一条完整的端到端渲染路径。

1. 概述与定位

UE5 光线追踪的核心理念

UE5 将光线追踪视为一种通用的射线查询基础设施，而非仅限于全局光照。所有需要空间射线查询的渲染功能——阴影遮挡测试、反射射线追踪、Lumen 硬件光追加速、路径追踪——都通过同一套加速结构（BLAS/TLAS）和同一套 SBT 机制来完成。这种统一设计避免了为每个功能独立维护光追资源的冗余。

2. 顶层架构设计

光追三层抽象架构

图2：UE5 光追三层抽象架构 -- RHI 层封装平台差异，引擎层管理场景和着色器，应用层消费光追查询
设计要点：引擎层的 FRayTracingScene 和 FRayTracingShaderBindingTable 是上层功能和底层 API 之间的关键桥梁。它们管理实例收集、可见性标记、SBT 分配，使得阴影/反射/GI 等模块不需要直接操作 D3D12 或 Vulkan 的光追对象，只需调用统一的 C++ 接口。

加速结构体系：BLAS + TLAS

UE5 采用标准的两级加速结构（Two-Level Acceleration Structure）模型，这是 DXR 和 VulkanRT 共同支持的标准架构：

层级全称构建粒度更新频率管理方式
BLAS Bottom-Level Acceleration Structure（几何底层层级） 单个 FRayTracingGeometry，对应一个 Static Mesh 或 Skeletal Mesh 静态几何仅构建一次；动态几何（骨骼网格）每帧更新 FRayTracingGeometryManager -- GeometryHandle 跟踪
TLAS Top-Level Acceleration Structure（实例顶层层级） 场景中所有可见实例的变换矩阵 + BLAS 引用 每帧重建（FastBuild）或更新（FastTrace） FRayTracingScene -- 按 Layer 管理、支持 View 级可见性

TLAS 通过 FRayTracingScene::Update() 构建实例 Buffer，然后在 FRayTracingScene::Build() 阶段调用 RHIBuildAccelerationStructures() 执行：

Shader Binding Table（SBT）封装设计

SBT 是 DXR/VulkanRT 中连接几何实例与着色器程序的核心数据结构。UE5 的 SBT 封装提供了三个关键抽象层次：

抽象职责生命周期
FRayTracingSBTAllocation 单个几何体的 SBT 记录分配，存储 BaseRecordIndex + SegmentCount 与 FRayTracingGeometry 生命周期一致
FRayTracingShaderBindingTable 管理所有 SBT 分配、Persistent SBT ID 分配/释放、脏绑定收集 与 View/Scene 帧生命周期一致
FRHIShaderBindingTable (RHI) 平台原生 SBT 资源（D3D12 中的 Shader Table Buffer） 按需创建/销毁

Persistent SBT 机制

UE5.8 引入了 Persistent SBT 机制（r.RayTracing.PersistentSBT，默认开启），将 SBT 数据持久化在 GPU 内存中，仅更新"脏"记录：

  1. 静态几何体 SBT：一次性分配，几乎不更新（仅在 Compaction 或材质变更时标记脏）
  2. 动态几何体 SBT：每帧按需更新（骨骼网格等变形的几何体）
  3. Inline Binding Data：与 Persistent SBT 关联的内联绑定数据 Buffer，通过 GetPersistentInlineBindingDataBuffer() 获取

这种设计大幅降低了 CPU 端每帧的 SBT 更新开销，特别是在大规模场景中。通过 FRayTracingShaderBindingTable::GetDirtyBindings() 收集所有标记为脏的绑定记录，然后仅更新这些记录。

场景实例收集的并行化设计

UE5 的场景几何体收集采用了多层并行化策略，以应对大型场景中数万个几何体实例的收集开销：

  1. CreateGatherInstancesTaskData() -- 创建任务数据 为每个 View 创建独立的 FGatherInstancesTaskData，包含场景引用、RayTracingProxyType 过滤等上下文信息。
  2. BeginGatherInstances() -- 启动并行收集 启动多个异步任务，每个任务负责收集场景中特定类型的几何体（静态网格、骨骼网格、HISM、Nanite 代理等）。通过 GActiveRayTracingProxyTypes 过滤启用的代理类型。
  3. FDynamicRayTracingInstancesContext -- 动态实例并行收集 通过 FDynamicPrimitiveIndexQueue 实现 work-stealing 模式，多个工作线程从队列中竞争获取 Primitive 进行收集。任务数量由 r.RayTracing.DynamicInstances.Parallel.NumTasks 控制。
  4. FinishGatherInstances() -- 汇总并提交到 Scene 在渲染线程完成所有收集任务的结果汇总，将实例添加到 FRayTracingScene 并分配 SBT 记录。

    Key ^= CachedMeshCommandFlags.bAllSegmentsCastShadow ? 0x1ull << 41 : 0x0;
    Key ^= CachedMeshCommandFlags.bAnySegmentsCastShadow ? 0x1ull << 42 : 0x0;
    Key ^= CachedMeshCommandFlags.bAnySegmentsDecal ? 0x1ull << 43 : 0x0;
    Key ^= CachedMeshCommandFlags.bAllSegmentsDecal ? 0x1ull << 44 : 0x0;
    Key ^= CachedMeshCommandFlags.bTwoSided ? 0x1ull << 45 : 0x0;
    Key ^= CachedMeshCommandFlags.bIsSky ? 0x1ull << 46 : 0x0;
    Key ^= CachedMeshCommandFlags.bAllSegmentsTranslucent ? 0x1ull << 47 : 0x0;

相同的几何体即使有多个变换实例，只要材质、标志、光照通道一致，就可以通过 InstancingKey 共享同一个 SBT 分配，大幅减少 SBT 条目数量。

4. 设计决策与权衡

内联光追 vs 动态 Shader 管线

内联光追（Inline Ray Tracing）

优点：

  • 在 Compute Shader 中直接调用 TraceRayInline()，无需 RayGen/Miss/HitGroup 切换
  • Shading 开销更低，适合简单的遮挡测试（如阴影）
  • GPU 上 Wave 利用率更高（无 Shader Table 查找开销）
  • UE 中通过 FOcclusionCS 实现，用于方向光硬阴影等场景

限制：

  • 不支持递归追踪，无法在 Hit 后发起次级射线
  • 无法利用材质 Hit Shader 的动态定制能力
  • 需要手动管理 Payload（所有数据通过 registers 传递）
  • 要求 1:1 Wave 映射（ThreadGroupSize = 8x4）

动态 Shader 管线（RayGen/Hit/Miss）

优点：

  • 支持复杂的材质着色逻辑（CHS 中可执行完整 BasePass 计算）
  • SBT 按几何体自动分发正确的 Hit Shader
  • 支持递归射线（反射中的次级射线、路径追踪的多次弹射）
  • UE 中 FOcclusionRGS 用于软阴影、带材质的阴影

限制：

  • RTPSO 创建开销大（需编译所有可能的 Shader Record 组合）
  • SBT 查找和 Shader 切换有额外延迟
  • 复杂材质下的寄存器压力可能限制 Occupancy
UE5.8 的选择：两种方式并存。阴影系统同时提供 FOcclusionRGS（动态管线，支持软阴影+半透明）和 FOcclusionCS（内联，用于方向光硬阴影）。通过 r.RayTracing.Shadows.AcceptFirstHit 等 CVar 可以在此二者间切换策略。Lumen HWRT 偏向内联光追，因为它需要大量简单射线查询；Path Tracer 必须使用动态管线，因为它依赖 CHS 来计算复杂材质。

TLAS 全量重建 vs 增量更新

策略触发条件BuildFlags性能特征
全量重建 每帧（r.RayTracing.Scene.BuildMode=0 FastBuild） AllowUpdate + FastBuild 构建快、遍历慢。适合场景几何频繁变化的动态场景
增量更新 每帧（r.RayTracing.Scene.BuildMode=1 FastTrace 默认） AllowUpdate + FastTrace 构建慢、遍历快。适合场景几何相对稳定的渲染
Batched Build r.RayTracing.Scene.BatchedBuild=true（默认） 批量提交 减少 GPU Barrier，提高多 View / 多 Layer 场景效率
Compact Instances r.RayTracing.Scene.CompactInstances=true 仅活跃实例 减少实例 Buffer 大小。不支持 Indirect TLAS 的平台需 GPU→CPU Readback

增量更新的关键约束：首次构建时必须声明 AllowUpdate 标志。之后的更新操作仅修改实例变换矩阵和 SBT 索引，而不重建 BVH 拓扑。这要求场景中几何体数量不变（或仅实例变换变化）。

材质复杂度与 SBT 大小的平衡

UE5 的材质 Hit Shader 编译会产生数百种排列，如果所有排列都独立占用 SBT 条目，SBT 大小将线性膨胀。UE 通过以下机制平衡：

  • InstancingKey 去重：相同材质 + 相同光照贴图策略 + 相同标志的几何体共享 SBT 分配
  • Trivial Shader 回退：当 r.RayTracing.EnableMaterials=0 或编译未完成时，所有几何体使用简单的 OpaqueShadowCHS
  • Shader Record 缓存（r.D3D12.RayTracing.CacheShaderRecords=1）：自动缓存和复用 SBT Hit Group Record，大幅减少相同网格实例的重复上传
  • Persistent SBT：仅更新脏记录的 UniformBuffer 内容，不变的部分保持在 GPU 上

BLAS（Bottom-Level Acceleration Structure）是 DXR 中最基础的几何体加速结构。UE5 在 FD3D12CommandContext::RHIBuildAccelerationStructures() 中完成完整的 BLAS 构建流程：

{

SBT 的 Shader Record 布局

每个几何体段在 SBT 中的布局由 RAY_TRACING_NUM_SHADER_SLOTS 决定（两个 Record 槽位）：

Slot Index用途绑定的着色器类型使用场景
0 (MATERIAL)材质着色 RecordCHS + 可选 AHS/IS反射、GI、路径追踪等需要材质颜色的 Ray
1 (SHADOW)阴影着色 RecordOpaqueShadowCHS（简化，无颜色计算）阴影射线（仅需确定遮挡，不需要材质输出）

对于 Miss Shader，同样有 2 个槽位：

Slot Index用途行为
0 (DEFAULT)默认 Miss设置 HitT = 表示"未命中"的值（FMinimalPayload::SetMiss()）
1 (LIGHTING)光照 Miss可用于在光追反射和半透明中评估光源辐亮度

在 FRayTracingSBTAllocation::GetRecordIndex() 中计算实际的 Record Index：

C++ RayTracingShaderBindingTable.cpp:77-90 -- Record Index 计算 
uint32 FRayTracingSBTAllocation::GetRecordIndex(
    ERayTracingShaderBindingLayer Layer,
    uint32 SegmentIndex) const
{
    // 1. 计算低于目标 Layer 的所有已分配 Layer 中的 Slot 数量
    uint32 LayerMask = (1 << (uint32)Layer) - 1;
    LayerMask = (uint32)AllocatedLayers & LayerMask;
    uint32 RecordTypeBaseOffset = FMath::CountBits(LayerMask) * RecordsPerLayer;
    // 2. RecordIndex = BaseRecordIndex + Layer偏移 + SegmentIndex * NUM_SHADER_SLOTS
    return BaseRecordIndex + RecordTypeBaseOffset +
           SegmentIndex * RAY_TRACING_NUM_SHADER_SLOTS;
}

6. UE5.8 演进变化

Lumen HWRT 与独立光追的融合趋势

UE5.8 的一个重要趋势是Lumen Hardware Ray Tracing（HWRT）与独立光线追踪的深度整合。以往，Lumen 的软件光追（Signed Distance Field）和独立的硬件光追反射/阴影是两套相对独立的系统。5.8 中：

  • Lumen 的 Surface Cache 和 Radiance Cache 越来越多地利用硬件光追加速，而非纯粹的 SDF 追踪
  • RayTracingScene 通过 LumenHardwareRayTracing::SetRayTracingSceneOptions() 直接配置光追场景选项，实现"一次构建、多处使用"
  • MegaLights 的光照采样通过 MegaLights::IsUsingLightingChannels() 影响场景的 AHS 配置
  • SceneOptions 构造函数统一处理 Lumen/反射/阴影/半透明的几何体包含需求（bTranslucentGeometry、bIncludeSky 等）

Path Tracer 采样改进

UE5.8 的路径追踪器进一步成熟：

  • Payload 类型扩展：新增 ERayTracingPayloadType::PathTracingMaterial（可与 Decals Payload 组合），Path Tracer 使用独立的 Payload 布局以容纳更多数据
  • Decals 支持：PathTracing 可包含 Decals 层（PathTracing::UsesDecals()），使用独立的 ERayTracingPayloadType::Decals
  • 参考大气和 DOF：PathTracing::UsesReferenceAtmosphere() 和 PathTracing::UsesReferenceDOF() 增加了对物理精确的大气散射和景深效果的支持
  • 非阻塞 RTPSO：Path Tracer 的 Hit Shader 编译通过非阻塞管线机制，在着色器就绪前使用不透明黑色回退，不会阻塞交互式操作

Work Graph 对光追管线的影响

虽然 Work Graph 在 5.8 中仍处于实验阶段，但其对光追的影响趋势已经显现：

  • 动态 Shader 负载：Work Graph 的核心思想是 GPU 端动态决定下一个执行的 Shader。这与光追的动态 Hit Shader 分发天然契合——光线命中后，GPU 可以根据材质类型直接 dispatch 对应的着色工作，不需要返回到 RayGen 进行二次分类
  • SBT 简化：在 Work Graph 模型中，SBT 的 Shader Record 分配可以被 Graph Node 的动态记录替换，RTPSO 创建可能被 Work Graph 的紧凑 PSO 创建取代
  • Nanite 集成：Nanite 的 GPU-Driven 管线与 Work Graph 理念一致——VisBuffer 解码后直接在 GPU 端确定需要的材质计算节点
