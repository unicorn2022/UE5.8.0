RHI 渲染硬件接口层

源码根路径：Engine/Source/Runtime/RHI/  |  核心文件：RHI.h · DynamicRHI.h · RHICommandList.h · RHIResources.h · RHIDefinitions.h

RHI（Render Hardware Interface）是 UE 渲染系统中承上启下的核心抽象层——向上为渲染器提供统一的 GPU 资源管理和绘制 API，向下将命令翻译到 DX12 / Vulkan / Metal 等平台原生接口。理解 RHI 才能理解 UE 如何用一套渲染代码跑遍所有平台。

1. 概述与定位

为什么需要 RHI？

现代图形 API（D3D12、Vulkan、Metal）的设计理念差异巨大：D3D12 基于 Descriptor Heap + Command List 模型，Vulkan 使用 Descriptor Set + Command Buffer，Metal 则是 Argument Buffer + Command Encoder。如果没有抽象层，渲染代码将为每个平台写一套——维护成本以平台数指数级增长。

UE 的策略是：将所有平台差异封装在 FDynamicRHI 虚接口背后。上层渲染器（延迟渲染管线、Nanite、Lumen 等）只面向统一的 RHI 命令列表编程，完全不感知底层是 DX12 还是 Vulkan。

RHI 分层架构

关键角色： GDynamicRHI（全局变量，定义于 DynamicRHI.h:1000）是 RHI 层的唯一入口——一个指向当前平台 FDynamicRHI 子类实例的全局指针。所有 RHI 全局函数（如 RHICreateTexture）都通过它调用平台实现。

2. 顶层架构设计

FDynamicRHI 抽象接口

FDynamicRHI（DynamicRHI.h:198）是 RHI 层的设计核心。它是一个纯虚基类，声明了约120+ 个虚函数，按职责可划分为以下几类：

资源管理 资源生命周期

纹理创建/更新（RHICreateTextureInitializer、RHIUpdateTexture2D/3D）、缓冲区创建（RHICreateBufferInitializer）、Lock/Unlock 操作、Shader 编译、UniformBuffer 创建。每个方法都标注了 FlushType 约束。

管线状态 PSO 创建

RHICreateGraphicsPipelineState / RHICreateComputePipelineState 将 Rasterizer/Blend/DepthStencil 状态与 Shader 组合编译为平台 PSO。Compute PSO 较简单（仅 Shader），Graphics PSO 编译最昂贵。

绘制命令 提交 Draw/Dispatch

通过 RHICreateBoundShaderState 绑定 Shader，再由 IRHICommandContext 接口发出 DrawCall、Dispatch、Clear 等命令。注意：FDynamicRHI 本身不直接包含 DrawPrimitive——这属于 IRHICommandContext。

查询 时间戳与遮挡

RHICreateRenderQuery、RHIGetRenderQueryResult —— 支持 GPU 时间戳（RQT_AbsoluteTime）和遮挡查询（RQT_Occlusion）。查询结果可能回读延迟 1-3 帧。

视口 窗口与 SwapChain

RHICreateViewport、RHIResizeViewport、RHIEndDrawingViewport —— 管理窗口系统和 BackBuffer 交换。依赖平台窗口句柄。

光追 Ray Tracing

RHICreateRayTracingScene、RHICreateRayTracingPipelineState、RHICreateShaderBindingTable —— 自 UE5 引入的硬件光追抽象。未实现平台返回 nullptr（通过 checkNoEntry() 兜底）。

FRHICommandList 命令缓冲模式

RHI 命令不会立即发送给 GPU，而是记录在 FRHICommandList 中，通过命令缓冲（Command Buffer）模式批量化处理。UE5.8 支持三种执行模式：

模式行为适用场景
Bypass（立即执行）命令不记录到列表，直接调用平台 RHI 接口执行编辑器非游戏视口、RHIThread 关闭时
延迟执行命令记录到命令列表，在 ImmediateFlush 时批量回放标准游戏模式，单线程执行
RHIThread 异步执行渲染线程记录命令，RHIThread 并行翻译为平台命令并提交 GPU主机/PC 高性能模式（默认）

命令记录机制

每个 RHI 命令在 FRHICommandList 中都是一个 FRHICommandBase 的派生对象，通过 ALLOC_COMMAND 宏在 MemStack 上分配。例如 DrawPrimitive 的典型实现（RHICommandList.h:3745）：

C++RHICommandList.h
// Bypass 模式：直接调用平台上下文
inline void DrawPrimitive(uint32 BaseVertexIndex, uint32 NumPrimitives, uint32 NumInstances)
{
    if (Bypass())
    {
        GetContext().RHIDrawPrimitive(BaseVertexIndex, NumPrimitives, NumInstances);
        return;
    }
    ALLOC_COMMAND(FRHICommandDrawPrimitive)(BaseVertexIndex, NumPrimitives, NumInstances);
}

核心设计思想：Top of Pipe（ToP）vs Bottom of Pipe（BoP）。ToP 表示命令仍在记录阶段（记录到链表），BoP 表示命令正在执行（Bypass 或 RHIThread 回放）。通过 IsBottomOfPipe() 判断当前处于哪个阶段。

使用示例 1：通过 RHI 创建纹理并上传数据

以下展示从渲染层代码调用 RHI 创建纹理的完整流程——从设置创建描述符到最终获得可用的纹理引用。

C++典型上层调用 —— 通过 RHI 创建纹理
#include "RHI.h"
#include "RHICommandList.h"
#include "RHIResources.h"

// 1. 构造纹理创建描述符
FRHITextureCreateDesc TextureDesc = FRHITextureCreateDesc::Create2D(
    TEXT("MyTexture"),           // Debug 名称
    1024, 1024,                    // 尺寸
    EPixelFormat::PF_R8G8B8A8    // 像素格式
);
TextureDesc.SetClearValue(FClearValueBinding::None);
TextureDesc.AddFlags(ETextureCreateFlags::ShaderResource);

// 2. 获取 RHI 命令列表并请求纹理初始器
FRHICommandListImmediate& RHICmdList = FRHICommandListImmediate::Get();
FRHITextureInitializer Initializer = GDynamicRHI->RHICreateTextureInitializer(RHICmdList, TextureDesc);

// 3. 如果需要在创建时填充数据
if (const FRHITextureInitializer::FTextureData* WritableData = Initializer.GetWritableData())
{
    uint8* DestData = static_cast<uint8*>(WritableData->Data);
    // 逐行填充纹理数据（DestData + Row * WritableData->Stride）...
    FMemory::Memcpy(DestData, MySourceData, DataSize);
}

// 4. 完成创建，获取 RHI 纹理引用
FTextureRHIRef Texture = Initializer.Finalize();
// 此时 Texture 已准备就绪，可绑定为 SRV 或 RenderTarget
底层发生了什么？ RHICreateTextureInitializer 调用链依次经过：GDynamicRHI（可能是 FValidationRHI 包装）-> FD3D12DynamicRHI::RHICreateTextureInitializer（D3D12Texture.cpp:1069）-> CreateTextureInternal -> CreateD3D12Textures，最终调用 ID3D12Device::CreateCommittedResource 或 CreatePlacedResource 分配 GPU 显存。

使用示例 2：通过 RHI 执行一次 DrawCall

完整展示从创建 PSO 到绑定资源再到发出绘制命令的全过程：

C++通过 RHI 执行 DrawCall 的完整流程
#include "RHICommandList.h"
#include "PipelineStateCache.h"
#include "RHIStaticStates.h"

// 1. 获取命令列表
FRHICommandListImmediate& RHICmdList = FRHICommandListImmediate::Get();

// 2. 设置 PSO 初始器（Graphics PSO 最复杂）
FGraphicsPipelineStateInitializer PSOInit;
PSOInit.SetBoundShaderState(
    RHICreateBoundShaderState(
        VertexDecl, VertexShader, PixelShader, nullptr));
PSOInit.BlendState      = TStaticBlendState<>::GetRHI();
PSOInit.RasterizerState  = TStaticRasterizerState<>::GetRHI();
PSOInit.DepthStencilState = TStaticDepthStencilState<>::GetRHI();
PSOInit.PrimitiveType    = EPrimitiveType::PT_TriangleList;

// 3. 通过 PipelineStateCache 获取或创建 PSO（自动缓存复用）
FGraphicsPipelineStateRHIRef PSO =
    PipelineStateCache::GetAndOrCreateGraphicsPipelineState(RHICmdList, PSOInit);

// 4. 开始 RenderPass，设置 PSO
FRHIRenderPassInfo RPInfo(RenderTarget, ERenderTargetActions::Load_Store);
RHICmdList.BeginRenderPass(RPInfo, TEXT("MyDrawPass"));
RHICmdList.ApplyCachedRenderTargets(PSOInit);
RHICmdList.SetGraphicsPipelineState(PSO);

// 5. 绑定资源（顶点缓冲、UniformBuffer、纹理）
RHICmdList.SetStreamSource(0, VertexBuffer, 0);
RHICmdList.SetShaderParameters(PixelShader, ShaderParams, ...);

// 6. 发出 DrawCall（记录到命令列表或 Bypass 直接执行）
RHICmdList.DrawPrimitive(0, NumPrimitives, 1);

RHICmdList.EndRenderPass();

这个 DrawCall 最终在 D3D12 平台到达 FD3D12CommandContext::RHIDrawPrimitive（D3D12Commands.cpp:1213），调用 SetupDraw 进行状态刷新，然后 GraphicsCommandList()->DrawInstanced(VertexCount, NumInstances, BaseVertexIndex, 0) 直接写入 D3D12 命令列表。

3. 执行流程与数据流

RHI 命令的完整生命周期

  1. 构建阶段（Render Thread）——上层渲染代码调用 RHICmdList.DrawPrimitive() 等。如果 Bypass 关闭，命令被序列化为 FRHICommandBase 节点，通过 ALLOC_COMMAND 宏在 MemStack 上分配，追加到命令链表尾部。
  2. 入队阶段（Render Thread）——FRHICommandListImmediate::ImmediateFlush() 被调用，命令列表通过 FRHICommandListExecutor::Submit() 提交到 RHI 执行管线。
  3. RHI Thread 执行阶段（RHI Thread）——RHI 线程（如果启用）逐个执行 FRHICommandBase::ExecuteAndDestruct()，将每个命令翻译为平台 RHI 上下文调用。如果启用 Parallel Translate，多个命令列表可并行为不同的 Task 线程翻译。
  4. GPU 提交阶段（RHI Thread）——FDynamicRHI::RHIFinalizeContexts() 关闭平台命令列表，RHISubmitCommandLists() 通过 ID3D12CommandQueue::ExecuteCommandLists（或 Vulkan 的 vkQueueSubmit）提交到 GPU 执行。

RHI 命令执行时序图

D3D12 提交路径（D3D12Submission.cpp:260）： FD3D12DynamicRHI::RHISubmitCommandLists 收集所有已完成的 FD3D12FinalizedCommands，将它们批量 ExecuteCommandLists 到 D3D12 命令队列。随后通过 Fence 机制异步等待 GPU 完成——RHIThread 不会阻塞等待 GPU。

4. 设计决策与权衡

细粒度 RHI 抽象——统一 API 的优势

UE 的 20+ 个渲染 Pass（深度预通道、BasePass、光照、半透明、后处理等）只需维护一份代码。新平台接入只需实现 FDynamicRHI 的纯虚函数，渲染管线零改动。

已接入 DX12、Vulkan、Metal、Agx（Apple Silicon 原生）

虚拟函数调用开销

每次调用 RHI 接口都经过虚函数表——在紧密循环中（如批量 DrawCall）会累积可测量的开销。UE 通过命令缓冲记录将虚调用转换为 MemStack 上的数据结构写入，然后批量回放，极大减少虚函数调用频率。

RHIThread 分离——CPU/GPU 解耦

渲染线程只负责记录命令（极快），RHIThread 负责翻译命令为平台 API 调用。两者并行执行：当 RHIThread 还在翻译上一帧的命令时，RenderThread 已经开始记录下一帧（流水线并行）。主机平台的核心优化。

控制台变量 r.RHIThread.Enable 1

额外延迟 + 同步复杂度

命令进入队列到 GPU 执行之间增加了一层缓冲延迟（通常 0.5-1 帧）。Lock/Unlock 操作需要 Flush RHIThread（FScopedRHIThreadStaller），打断了并行性。资源生命周期管理变得复杂——必须确保 GPU 使用完毕才能释放。

Validation 层的成本

行为开发期（WITH_RHI_VALIDATION=1）发布期（Shipping）
FValidationRHI 包装激活，所有 RHI 调用经 FValidationRHI 转发完全移除，直接调用平台 RHI
资源访问校验追踪每个资源的 ERHIAccess 状态，检测错误的读写模式不检查（假设代码正确）
UniformBuffer 校验验证 Shader 期望的 Layout Hash 与绑定的 Buffer 一致不检查
DrawCall 参数校验检查 PSO 与 RenderPass 兼容性（Depth 模式等）不检查
资源泄漏检测追踪引用计数，报告未释放资源不追踪
典型开销~5-15% RHI 层 CPU 时间增加零开销

为什么不用统一 Shader 语言？

UE 的策略是：上层使用 HLSL 编写所有 Shader，但在编译时通过 SPIRV-Cross 将 HLSL 生成的 DXIL/SPIR-V 转译为各平台原生格式。RHI 层不做 Shader 语言抽象——Shader 编译管线（ShaderCompileWorker）在 RHI 之上独立运行。

HLSL 统一编写

所有 Shader 用 HLSL 编写（UE Shader 模型基于 HLSL 语法扩展）。D3D 编译器直接编译为 DXIL；对于 Vulkan/Metal，使用 hlslcc 或 DXC 生成 SPIR-V。

SPIRV-Cross 转译

Vulkan 上 SPIR-V 直接使用；Metal 上 SPIRV-Cross 将 SPIR-V 转译为 MSL（Metal Shading Language）。这由编译管线处理，RHI 层接收到的已经是平台原生字节码。

FRHICreateShaderDesc 统一入口

RHI 层只需要 RHICreatePixelShader(FRHICreateShaderDesc(Code)) ——一个字节数组。D3D12 调用 CreatePixelShader，Vulkan 调用 vkCreateShaderModule，Metal 调用 newFunctionWithName。

6. UE5.8 演进变化

D3D12 Enhanced Barriers

UE5.8 的 D3D12RHI 新增了 D3D12EnhancedBarriers.cpp（1866 行），实现了对 D3D12 Enhanced Barrier API 的完整支持。通过 FD3D12EnhancedBarriersBatcher 批量管理 Global/Texture/Buffer 级别的 Barrier，替代旧的 Legacy Barrier 模型。

检测机制：查询 D3D12_FEATURE_DATA_D3D12_OPTIONS12::EnhancedBarriersSupported（D3D12Adapter.cpp:1242），运行时自动选择实现。

Vulkan Ray Tracing 扩展集成

FDynamicRHI 的光追接口（RHICreateRayTracingPipelineState、RHICreateShaderBindingTable）在 UE5.8 中进一步扩展，支持 Vulkan 的 VK_KHR_ray_tracing_pipeline 和 VK_KHR_acceleration_structure 扩展。

设计保障：未实现光追的平台（如 Metal 非 Apple Silicon）默认返回 nullptr，调用者通过检查返回值判断能力。

Metal 3 适配 (Agx)

新增 ERHIInterfaceType::Agx（RHIDefinitions.h:151），专门为 Apple Silicon GPU 提供原生 Apple GPU 家族支持。与标准 Metal 路径并行存在，利用 Metal 3 的新特性（如 Mesh Shaders、Ray Tracing）。

Bindless 资源支持

新增 FRHIResourceCollection 和 FRHIDescriptorRange 接口（DynamicRHI.h:509-525），支持 Bindless 资源绑定模式。D3D12 通过 Descriptor Heap 实现，Vulkan 通过 Descriptor Indexing 实现。

传统绑定每个 Shader 只能绑定有限数量的纹理/Buffer；Bindless 允许 Shader 访问整个资源数组——对 Nanite 材质系统和 MegaLights 至关重要。

5.8 API 清理

多个旧 API 被标记为 UE_DEPRECATED(5.8)：RHIPreferredPixelFormatHint（视口需显式指定像素格式）、RHICheckViewportHDRStatus、RHIZBuffer 枚举移除（深度反转已成既定事实）、RHIGetDefaultContext 不再允许上层直接获取平台上下文。

Work Graphs 支持

新增 RHICreateWorkGraphShader、RHICreateWorkGraphPipelineState、SF_WorkGraphRoot 等（RHIDefinitions.h:200-201），为 D3D12 Work Graphs API 提供 RHI 层抽象，支持 GPU 驱动的任务图调度。
