光源与阴影系统

从光源参数到屏幕上的光照效果——UE5 的光源类型体系与 VSM 阴影方案

1. 概述与定位

UE5 的光照系统是渲染管线的核心支柱之一，负责将所有光源的能量最终转换为屏幕像素的辐射度值。 整体链路从 CPU 端的光源收集与可见性判断开始，经由 GPU 端的阴影深度渲染， 再到 Clustered Deferred 光照注入，最终在屏幕空间完成对每个像素的光照累加。

当前实现以 Clustered Deferred Shading 为默认光照路径（r.UseClusteredDeferredShading）， 阴影方面则主推 Virtual Shadow Maps（VSM）——一种为 Nanite 虚拟几何量身打造的虚拟化阴影系统， 同时保留传统 CSM（Cascaded Shadow Maps）作为后备方案。

核心源码路径均位于 Engine/Source/Runtime/Renderer/Private/：

  • LightRendering.cpp — Clustered Deferred 光照渲染主入口（RenderLights() @ L1627）
  • ShadowDepthRendering.cpp — 传统阴影深度渲染（CSM / 逐光源）
  • VirtualShadowMaps/VirtualShadowMapArray.cpp — VSM 页面分配、标记、渲染
  • VirtualShadowMaps/VirtualShadowMapCacheManager.cpp — VSM 页面缓存与失效管理
  • VirtualShadowMaps/VirtualShadowMapClipmap.cpp — 方向光 Clipmap 层级管理
  • VirtualShadowMaps/VirtualShadowMapProjection.cpp — 阴影投影到屏幕空间

2.2 Clustered Deferred Shading

Clustered Deferred Shading 是 UE5 桌面平台默认的延迟光照路径。它将视锥体在 X、Y 方向按屏幕 Tile 切分， 在 Z（深度）方向按对数分布切分出多个切片，构成三维网格（称为 Froxel）。 每个 Froxel 单元维护一份影响该空间区域的光源列表，从而将逐像素的光源遍历时间从 O(N_lights) 降至 O(N_per_cell)。

图 2：Clustered Deferred 三维网格划分示意

在 RenderLights() 中，优先级决策路径如下：

}
// Fallback: traditional standard deferred (per-light fullscreen quad)

{
    RenderSimpleLightsStandardDeferred(GraphBuilder, SceneTextures, SortedLightSet, SubstrateSceneData);
}

2.3 Virtual Shadow Maps 架构

Virtual Shadow Maps 是 UE5 为 Nanite 时代设计的虚拟化阴影方案。核心思想与传统虚拟纹理类似： 将巨大的逻辑阴影贴图（16K×16K 虚拟分辨率）按 128×128 像素的物理页面 切分， 只在需要时分配和渲染实际页面。

Engine/Source/Runtime/Renderer/Private/VirtualShadowMaps/VirtualShadowMapArray.h:67
class FVirtualShadowMap
{
public:
    // PageSize × Level0DimPagesXY 定义虚拟地址空间，例 128×128 = 16K
    static constexpr uint32 PageSize          = VSM_PAGE_SIZE;            // 128
    static constexpr uint32 PageSizeMask      = VSM_PAGE_SIZE_MASK;        // 127
    static constexpr uint32 Log2PageSize      = VSM_LOG2_PAGE_SIZE;        // 7
    static constexpr uint32 Level0DimPagesXY  = VSM_LEVEL0_DIM_PAGES_XY;  // 128
    static constexpr uint32 MaxMipLevels      = VSM_MAX_MIP_LEVELS;        // 8
    static constexpr uint32 VirtualMaxResolutionXY = VSM_VIRTUAL_MAX_RESOLUTION_XY;
                                                                         // 16K
    static constexpr uint32 RasterWindowPages = VSM_RASTER_WINDOW_PAGES;

    static constexpr uint32 PhysicalPageAddressBits = 16U;
    static constexpr uint32 NumHZBLevels = Log2PageSize;   // 7 级 HZB

    // 物理页面池最大尺寸：65536×65536 纹素 = 512×512 页面
    static constexpr uint32 MaxPhysicalTextureDimPages =
        1U << PhysicalPageAddressBits;  // 65536
};

缓存系统通过 FVirtualShadowMapArrayCacheManager 管理，核心变量：

  • r.Shadow.Virtual.Cache.FramesStaticThreshold = 100 — 连续 100 帧未失效则转入静态缓存池
  • r.Shadow.Virtual.Cache.MaxPageAgeSinceLastRequest = 1000 — 页面未被请求的最大存活帧数
  • r.Shadow.Virtual.Cache.MaxLightAgeSinceLastRequest = 10 — 出视野光源及其页面保留帧数
架构要点： 方向光使用 Clipmap 结构（6~22 级同心圆环），每级覆盖范围翻倍； 局部光源（Point/Spot/Rect）使用 单页 VSM（Single Page VSM）； 物理页面池按 静态/动态分离 缓存，分别维护独立 HZB 以优化遮挡剔除效率。

4. 设计决策与权衡

VSM 128x128 页面粒度选择

128 像素 的页面大小在 GPU 硬件层面和虚拟内存效率之间取得了平衡： 太小的页面（如 32×32）会导致页面表过大和 HZB 层级过多； 太大的页面（如 256×256）则浪费内存（细小物体只占页面的一小部分）。 128 也是 2 的幂，兼容 GPU Tile 架构，HZB 级数为 log2(128)=7 级。

静态缓存 vs 动态重渲染

缓存通过 r.Shadow.Virtual.Cache.FramesStaticThreshold (默认 100 帧) 判断是否转静态。 失效判断逻辑（FVirtualShadowMapPerLightCacheEntry::Update()）：光源 Key 变化 (Transform/参数) → 立即失效；图元 Transform 变化、WPO 动画、Nanite LOD Delta → 标记失效。 静态缓存提供独立的 HZB 池，避免动态页面频繁失效污染静态数据。

Shadow Bias 策略对比

UE5 的 VSM 投影支持多种 Bias 策略：
Depth Bias — 在世界空间偏移深度值，简单但近距离容易漏光
Slope Scale Bias — 根据表面法线与光线夹角缩放偏移，平面上小 / 斜面上大
Normal Offset Bias (SMRT) — 沿表面法线偏移采样点，是 VSM 推荐的默认策略， 配合 r.Shadow.Virtual.SMRT 系列 CVar 控制光线步进和自适应采样。

CSM vs VSM 的适用场景

CSM (Cascaded Shadow Maps)：移动端 / 低品质预设 / 非Nanite 场景的备选方案。 优势在于内存可预测（固定级联数 × 固定分辨率）、无页面管理开销。
VSM：高端平台 + Nanite 场景的首选。优势在于自适应分辨率、动态页面分配、 与 Nanite 几何单 Pass 阴影渲染的深度集成。 VSM 对非Nanite 几何过多的场景可能触发页面溢出 ("VSM Non-Nanite Marking Job Queue overflow")。

5. 底层实现关联

5.1 RenderLights() 内部路径分发

FDeferredShadingSceneRenderer::RenderLights()（LightRendering.cpp:1627） 是整个延迟光照的入口函数。其内部按如下优先级调度：

  1. VSM 投影掩码渲染：ShadowSceneRenderer.RenderVirtualShadowMapProjectionMaskBits()
  2. Clustered Deferred：AddClusteredDeferredShadingPass() — 覆盖绝大多数光源
  3. 简单光源后备：RenderSimpleLightsStandardDeferred()
  4. 非 Clustered 批处理：逐光源 RenderLight() fullscreen quad
  5. MegaLights 跳过：bHandledByMegaLights 标记的光源在此阶段跳过
Froxel 与 Clustered 的配合： 在每个 Compute Shader 线程组中，首先根据像素的屏幕坐标和深度值（来自 GBuffer Depth） 查找所属的 Froxel 索引。Froxel 对应的光源列表在 CPU 端预注入（Light Grid Injection）。 着色器仅需遍历该 Froxel 内的光源，数量通常为个位数，大幅减少冗余计算。

5.2 VSM 页面分配与 Cache 命中逻辑

页面分配发生在 FVirtualShadowMapArray::BuildPageAllocations()。流程：

  1. 页面需求标记（BeginMarkPages）：根据屏幕空间像素对阴影分辨率的需求， 标记每个 VSM 需要哪些虚拟页面。支持 Froxel 近似标记和精确像素标记两种模式。
  2. Cache 查询：对每个所需页面，查询 FVirtualShadowMapCacheManager 是否已有有效缓存。
  3. 物理页面分配：未命中缓存的页面从物理页面池中分配新的物理地址。 如果池已满，按 LRU 策略淘汰最久未使用的页面。
  4. 页面表写入：将虚拟→物理映射写入 GPU 页面表纹理， 供投影 Pass 中的采样器查找。

5.3 Clipmap 级联更新

方向光的 Clipmap 由 FVirtualShadowMapClipmap 管理。每帧更新逻辑：

  • 层级数：由 r.Shadow.Virtual.Clipmap.FirstLevel (默认 6) 和 r.Shadow.Virtual.Clipmap.LastLevel (默认 22) 共同决定
  • 每级覆盖半径：逐级翻倍，level N 的覆盖半径 = BaseRadius × 2^N
  • Z 范围：由 r.Shadow.Virtual.Clipmap.ZRangeScale (默认 1000) 控制， 太小导致频繁深度范围失效，太大导致精度损失和阴影闪烁
  • 相机跟随：Clipmap 中心随相机移动，当偏移超过页面大小时触发逐级更新
  • WPO 动画距离：r.Shadow.Virtual.Clipmap.WPODisableDistance 可在远距离级联中禁用 WPO 动画以节省性能

FVirtualShadowMapCacheEntry::UpdateClipmapLevel()（VirtualShadowMapCacheManager.cpp:259） 在每帧检查缓存有效性——当 Z 范围变化超过 90% 保护带时，自动失效该级。

6. UE5.8 演进变化

6.1 MegaLights 与 RenderLights() 的关系

MegaLights 是 UE5.8 正式可用的大量动态可投影光源系统，它与传统的 RenderLights() 是互补而非替代关系。

在 RenderLights() 的光源排序中，bHandledByMegaLights 标记的光源 会被排到列表末尾并从标准管线中跳过（MegaLightsLightStart 索引）。 MegaLights 使用独立的渲染路径：

  • 采样点生成：GenerateMegaLightsSamples() — 屏幕空间或世界空间采样
  • 光源采样：MegaLightsSampling.cpp — 对每个采样点重要性采样影响光源
  • 光追加速：MegaLightsRayTracing.cpp — 硬件光追可选加速阴影查询
  • 降噪：MegaLightsDenoising.cpp — 时序/空间联合降噪
  • 合成：MegaLightsResolve.cpp — 最终合成到 SceneColor

RenderLights (Clustered Deferred)

适合：少量重要光源（主方向光、少数关键点光源）
优势：全分辨率 G-Buffer 光照，零噪声
局限：光源数量增加时 GPU 时间线性增长

5.8 MegaLights

适合：大量次要动态光源（数十至数百个）
优势：亚线性复杂度（采样驱动），支持大量光源
局限：需要降噪，存在噪声/模糊权衡

6.2 VSM 非Nanite 路径优化

UE5.8 中引入了多个优化非Nanite 几何阴影渲染的改动：

  • Coarse Pages 排除选项：r.Shadow.Virtual.NonNanite.IncludeInCoarsePages 0 可将非Nanite 几何从粗粒度页面中排除，减少页面需求以避免队列溢出。 在 Uniform Buffer 中体现为 bExcludeNonNaniteFromCoarsePages 标志。
  • VSM 延迟失效预算：r.Shadow.Virtual.DeferredInvalidationBudget (默认 -1 无限制)， 控制每帧消费的延迟失效标记数量，将尖峰工作分散到多帧。
  • Froxel 页面标记：r.Shadow.Virtual.MarkPagesUsingFroxels 实验性功能， 使用 Froxel（每 8×8 像素一个）近似标记页面需求，减小带宽压力。
  • Single-Pass 点光源：r.Shadow.OnePassPointLight.UseCachedMDCs 为单 Pass 点光源使用预缓存的 Mesh Draw Command，减少 CPU 渲染时间。
