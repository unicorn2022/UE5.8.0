UE5.8 渲染管线 / RDG 渲染依赖图

RDG：Render Dependency Graph

深入解读 UE5.8.0 的渲染依赖图系统 — 基于 DAG 的自动资源生命周期管理和 Pass 调度引擎

📄 Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h | RenderGraphResources.h | RenderGraphPass.h | RenderGraphUtils.h

1. 概述与定位

1.1 为什么需要渲染依赖图

传统渲染代码中，资源屏障（Barrier）必须由开发者手动插入。每当你写完一个 RHICmdList.DispatchComputeShader() 之后，如果想在后续 Pixel Shader 中读取结果，必须手动调用 RHICmdList.Transition() 来插入 UAV→SRV 屏障。这种方式有三个致命问题：

屏障遗漏

忘记插入或错误插入屏障会导致 GPU 数据竞争（data race）、渲染闪烁或黑屏。这些 bug 极难定位，通常只在特定 GPU 上复现。

过度同步

保守地插入过多屏障会打断 GPU 流水线，降低 GPU 占用率，导致性能下降 10%-30%。

资源泄漏

中间 RT/Buffer 的生命周期完全由开发者管理。忘记释放会导致显存逐渐耗尽，过早释放则导致 use-after-free。

RDG 的核心价值主张：将整个帧的渲染工作表达为一张有向无环图（DAG）， 其中 Pass 是节点，Resource 是边。让引擎自动完成依赖分析、屏障插入和资源生命周期管理。

依赖推导的核心逻辑：每个 Pass 通过 SHADER_PARAMETER_STRUCT 声明自己读写哪些 RDG 资源。 FRDGBuilder::AddPass() 自动枚举参数结构体中的 RDG_TEXTURE_ACCESS、 RDG_BUFFER_SRV、RDG_BUFFER_UAV 等宏标记的资源，根据读写语义推导出 Pass 之间的依赖边。

2.2 FRDGBuilder 的 API 设计

FRDGBuilder 是 RDG 系统的核心入口类，采用 栈上 RAII 生命周期：创建于渲染函数栈上，在析构前必须调用 Execute()。其 API 分为四大类：

2.3 Pass 类型体系

RDG Pass 的类型通过 ERDGPassFlags 位掩码来标识 GPU 工作负载类型，核心分类如下：

FRDGPass（基类）

所有 RDG Pass 的抽象基类。持有 ParameterStruct、Flags、Pipeline 以及 TextureStates/BufferStates 依赖信息。

TRDGLambdaPass<P,L>

模板化 Pass，将 Lambda 作为执行体。自动推断 TaskMode（Inline/Await/Async）。

TRDGEmptyLambdaPass<L>

无参数 Pass 的特化版本，适用于无需声明资源的辅助 Pass。自动附带 NeverCull+SkipRenderPass 标志。

FRDGDispatchPass / TRDGDispatchPass

专门用于用户自行创建多个 RHI CommandList 并并行录制的 Pass。U5.8 新增。

ERDGPassFlags 核心标志

标志含义
ComputeGraphics 管线的 Compute Pass
AsyncCompute异步 Compute 管线 Pass（与 Graphics 并行执行）
Raster光栅化 Pass（像素/顶点着色器）
Copy复制操作 Pass
NeverCull永远不会被裁剪，即使其输出未被消费
SkipRenderPass跳过 RHI RenderPass 的 Begin/End（用户自行管理）
ReadbackGPU 回读 Pass（CopySrc 访问）

2.4 使用示例：从一个后处理 Blur Downsample 看 RDG 迁移

以下示例展示如何使用 RDG 实现一个 Compute Shader 的模糊降采样后处理效果，对比传统 RHI 方式与 RDG 方式。

传统 RHI 方式（手动 Barrier）

C++ — 传统 RHI
// === 传统方式：手动管理 barrier 和资源生命周期 ===
void RenderBlurTraditional(FRHICommandListImmediate& RHICmdList, FRHITexture* Input, FRHITexture* Output)
{
    // 1. 手动创建中间纹理 —— 谁来释放？何时释放？
    TRefCountPtr<IPooledRenderTarget> HalfRes;
    GRenderTargetPool.FindFreeElement(RHICmdList, Desc, HalfRes, TEXT("HalfRes"));

    // 2. 手动插入屏障 Input → Compute SRV
    if (Input需要屏障) { RHICmdList.Transition(...); }

    // 3. Dispatch Downsample
    RHICmdList.SetComputeShader(DownsampleShader);
    RHICmdList.SetShaderResourceView(InputSRV);
    RHICmdList.SetUAVParameter(HalfResUAV);
    RHICmdList.DispatchComputeShader(...);

    // 4. 手动插入屏障 UAV → SRV（极易遗忘！）
    RHICmdList.Transition(HalfRes, ERHIAccess::UAVCompute, ERHIAccess::SRVCompute);

    // 5. Dispatch Blur
    RHICmdList.SetShaderResourceView(HalfResSRV);
    RHICmdList.SetUAVParameter(OutputUAV);
    RHICmdList.DispatchComputeShader(...);

    // 6. 手动屏障 Output → SRV（给后续 Pass 使用）
    RHICmdList.Transition(Output, ERHIAccess::UAVCompute, ERHIAccess::SRVCompute);
    // ⚠ 忘记释放 HalfRes？泄漏! 提前释放？崩溃!
}

RDG 方式（声明式）

C++ — RDG 声明式
// === RDG 方式：声明资源 + 注册 Pass，一切交给引擎 ===
void RenderBlurRDG(FRDGBuilder& GraphBuilder, FRDGTextureRef Input, FRDGTextureRef Output)
{
    // 1. 声明中间资源 —— 生命周期由 RDG 管理
    FRDGTextureDesc HalfResDesc = FRDGTextureDesc::Create2D(
        Input->Desc.Extent / 2, Input->Desc.Format, FClearValueBinding::None,
        TexCreate_ShaderResource | TexCreate_UAV);
    FRDGTextureRef HalfRes = GraphBuilder.CreateTexture(HalfResDesc, TEXT("HalfResBlur"));

    // 2a. Downsample Pass —— RDG 自动推断 Input→SRV, HalfRes→UAV
    auto* DownsampleParams = GraphBuilder.AllocParameters<FDownsampleCS::FParameters>();
    DownsampleParams->InputTexture  = Input;   // RDG 标记为 SRVCompute 读
    DownsampleParams->OutputTexture = GraphBuilder.CreateUAV(HalfRes); // 写

    FComputeShaderUtils::AddPass(GraphBuilder, RDG_EVENT_NAME("BlurDownsample"),
        DownsampleShader, DownsampleParams,
        FComputeShaderUtils::GetGroupCount(Input->Desc.Extent, 8));

    // 2b. Blur Pass —— RDG 自动推断 HalfRes→SRV, Output→UAV
    auto* BlurParams = GraphBuilder.AllocParameters<FBlurCS::FParameters>();
    BlurParams->InputTexture  = HalfRes;
    BlurParams->OutputTexture = GraphBuilder.CreateUAV(Output);

    FComputeShaderUtils::AddPass(GraphBuilder, RDG_EVENT_NAME("BlurHorizontal"),
        BlurShader, BlurParams,
        FComputeShaderUtils::GetGroupCount(HalfRes->Desc.Extent, 8));

    // ✅ 无需手动 Barrier —— RDG 根据 SRV/UAV 语义自动推导
    // ✅ 无需手动 Release —— HalfRes 在 Execute() 后自动释放
    // ✅ 若 HalfRes 未被消费，Downsample Pass 会被自动裁剪！
}
关键差异：RDG 版本删除了所有屏障代码和资源释放代码，代码量减少约 40%， 同时获得了自动 Pass 裁剪（如果 Output 是中间结果但未被消费则整个子图被跳过）和 Transient 资源池（HalfRes 可能完全不占用独立显存）优化。

3. 执行流程 / 数据流

3.1 构建阶段（Graph Construction）

渲染代码调用 FRDGBuilder::AddPass() 注册 Pass 时，Lambda 体不会立即执行。 RDG 仅在此时做以下工作：

1. 分配 Pass 对象

通过 Passes.Allocate<TRDGLambdaPass>() 从 RDG 内部分配器创建 Pass 实例。

2. 枚举参数结构体中的所有资源

遍历 ParameterStruct 中的 RDG_TEXTURE_ACCESS、RDG_BUFFER_SRV、 RDG_BUFFER_UAV 等绑定，为每个资源创建 FRDGPass::FTextureState 或 FRDGPass::FBufferState，记录访问类型和子资源范围。

3. 记录 LastProducer 状态

对于每个资源，更新 FRDGTexture::LastProducers[] 或 FRDGBuffer::LastProducer（按管线类型存于 FRDGProducerStatesByPipeline 中）。这是后续依赖推导的关键。

4. 存储 Lambda

Lambda 被 Move 到 TRDGLambdaPass 中，直到 Execute() 时才被调用。 这意味着所有 Lambda 捕获的变量必须保证生命周期覆盖到 Execute 时刻。

3.2 编译 / 裁剪阶段

在 Execute() 内部，Compile() 方法执行以下操作：

1. Pass 依赖建立（SetupPassDependencies）

遍历每个 Pass 的资源使用，调用 AddCullingDependency() 将 Producer-Consumer 关系写入 Pass 的 Producers 和 CrossPipelineConsumers 列表。

2. Pass 裁剪（Culling）

标记所有 Pass 为 bCulled = true，然后从 NeverCull Pass 和输出资源的 LastProducer 出发进行 DFS 反向遍历（CullPassStack），将可达 Pass 取消裁剪。 被裁剪 Pass 使用的资源计数会相应减少。

3. RenderPass 合批（Merge）

顺序遍历 Graphics 管线上的 Raster Pass。如果前后 Pass 使用相同的 RenderTarget 且没有 外部 Side-Effect（bRenderPassOnlyWrites），则合并到同一个 RHI RenderPass 中。 在合并时设置 Barrier Prologue/Epilogue Pass 来避免 RenderPass 内部发出屏障（非法操作）。

4. Async Compute 围栏建立

为 AsyncCompute Pass 建立 Fork/Join 关系，插入跨管线围栏。同时调整资源分配/释放时机， 因为 AsyncCompute 管线的资源必须覆盖整个异步执行区间。

3.3 Transient 资源分配

编译完成后，Execute() 通过遍历 Pass 序列来收集资源的分配和释放操作： CollectAllocations() 和 CollectDeallocations()。 这些操作生成 FCollectResourceOp 列表，按类型分为三个池：

Transient 资源

生命周期短暂、跨 Pass 不重叠的资源可走 Transient 分配器。多个 RDG 资源可能共享同一块 GPU 内存（Aliasing），大幅降低显存占用（典型节省 30%-50%）。

Pooled 资源

无法走 Transient 的资源（如 External、Extracted、UPLOAD）使用传统的 RenderTargetPool / BufferPool。分配器会尝试复用已有分配。

UniformBuffer / View

UniformBuffer 和 View（SRV/UAV）在资源分配完成后统一创建，因为它们依赖底层 RHI 资源已就绪。

Transient 判定的条件（FRDGBuilder::IsTransient()）：

  • 平台/分配器支持（bSupportsTransientTextures/Buffers）
  • 非 QueuedForUpload（上传 Buffer 必须持久化）
  • 非 FastVRAM（FastVRAM 资源总是 Transient）
  • 未被标记 bForceNonTransient
  • 被提取的资源受 GRDGTransientExtractedResources CVar 控制

3.4 屏障插入

屏障推导是 RDG 最核心的价值之一，由三个步骤完成：

CompilePassBarriers

遍历 Pass，对每个资源的每个子资源比较 StateBefore 和 StateAfter。 调用 FRDGSubresourceState::IsTransitionRequired() 判断是否需要屏障。

CollectPassBarriers

将屏障操作收集到 FRDGBarrierBatchBegin 批次中。一个批次可以包含多个 Transition， 批次的 End 位置关联到特定 Pass 的 Prologue 或 Epilogue。

CreatePassBarriers

调用 FRDGBarrierBatchBegin::CreateTransition() 生成 RHI 屏障对象， 在 Pass 执行时通过 ExecutePassPrologue() / ExecutePassEpilogue() 提交。

Split Barrier 设计：屏障可以放置在 Pass 的 Prologue（执行前）或 Epilogue（执行后）。 例如，Transient 资源的 Discard 操作（标记旧内容无效）必须在 Epilogue 中执行， 因为新 Pass 需要先完成写入才能释放前一个所有者的内存。

3.5 并行执行

执行阶段支持三种模式（ERDGPassTaskMode）：

模式说明
Inline直接在渲染线程上执行。适用于使用 FRHICommandListImmediate 的 Pass。
Await在 Task 中并行执行，Execute() 末尾自动等待。U5.8 中多个 Pass 可合并到一个 FParallelPassSet 并行运行。
Async在 Task 中执行但不自动等待。用户通过 FRDGAsyncTask 控制同步。适用于 DispatchPass。

4. 设计决策与权衡

4.1 声明式 API 的学习曲线 vs 运行时优化收益

维度成本收益
编程模型转换 需要理解 RDG 资源句柄（FRDGTextureRef）vs 直接使用 FRHITexture*；Lambda 延迟执行的心智模型 消除整类 Barrier 错误（数据竞争、闪烁）；代码量减少 30%-50%
参数结构体 每个 Pass 需要定义 SHADER_PARAMETER_STRUCT 自文档化、自动验证、ClearUnusedGraphResources 自动清理未绑定资源
编译开销 Compile() 遍历所有 Pass 和资源进行依赖分析 并行编译（ERDGBuilderFlags::ParallelCompile）、异步 Setup Queue 分摊开销
调试 Lambda 延迟执行增加了调试难度（断点时机） r.RDG.Debug 提供 Immediate Mode 逐 Pass 执行；RDG Dump/Visualize 工具

4.2 Transient 资源池的内存节省 vs 生命周期复杂度

Transient 分配器允许生命周期不重叠的多个 RDG 资源共享同一块物理 GPU 内存。这带来了显著的内存节省 （典型场景 30%-50%），但也增加了系统复杂度：

  • Aliasing 屏障：当两个 RDG 资源共享同一块显存时，需要在第二个资源首次使用前插入 ERHIAccess::Discard 操作来标记旧内容无效。这些 Aliasing 过渡在 FRDGBuilder::AddAliasingTransition() 中处理。
  • AsyncCompute 兼容：AsyncCompute 管线的资源生命周期必须覆盖整个异步区域（Fork 到 Join）， 这限制了 Transient 的重用机会。
  • Extraction 冲突：被 Extracted 的资源不能走 Transient（除非 GRDGTransientExtractedResources 开启），因为 Extracted 资源需要在 Execute 之后仍然存活。

4.3 RDG 如何与 RHI 命令列表交互

RDG 完全构建在 RHI 层之上：

执行时，RDG 通过 ExecutePass() 来调用用户 Lambda，Lambda 内部使用 FRHIComputeCommandList&（或 FRHICommandList&）执行实际的 RHI 调用。 ExecutePassPrologue() 在 Lambda 前提交屏障， ExecutePassEpilogue() 在 Lambda 后提交收尾屏障。

4.4 为什么 RDG Pass 不能直接调用 RHI

核心约束：RDG Pass 的 Lambda 内只能通过 FRDGTexture::GetRHI() / FRDGBuffer::GetRHI() 来获取底层 RHI 资源句柄（仅在 Pass 执行期间有效）。 不能绕过 RDG 资源句柄直接创建/销毁 RHI 资源。

原因有三：

  1. 屏障依赖完整性：如果 Pass A 通过原始 RHI 指针直接创建纹理并写入，RDG 不知道这个写入操作的存在，就无法为后续 Pass 插入正确的屏障。
  2. 资源生命周期保证：RDG 创建的 Texture/Buffer 仅在声明了该资源的 Pass 执行期间保证 RHI 底层资源存活。在未声明的 Pass 中访问会导致 use-after-free。
  3. 裁剪正确性：如果 Pass 被裁剪但其副作用通过 RHI 指针泄露了，会导致渲染管线状态不一致。

例外（外部访问模式）：通过 UseExternalAccessMode() 可以将资源切换到外部访问模式， 允许后续 Pass 直接访问底层 RHI 资源（只读）。这通常用于渐进式迁移现有代码到 RDG。

5. 底层实现关联

5.1 AddPass 内部依赖推导逻辑

当调用 GraphBuilder.AddPass(Name, ParameterStruct, Flags, Lambda) 时，内部经过以下路径：

C++ — AddPass 调用链
// 1. AddPass 公共 API（RenderGraphBuilder.h:221）
template <typename P, typename L>
FRDGPassRef AddPass(FRDGEventName&& Name, const P* ParameterStruct,
                      ERDGPassFlags Flags, L&& ExecuteLambda)
{
    // 覆盖 Pass Flags（AsyncCompute 强制开关等）
    Flags = OverridePassFlags(Name, Flags);
    // 转发到内部实现
    return AddPassInternal(Name, P::FTypeInfo::GetStructMetadata(),
                          ParameterStruct, Flags, MoveTemp(ExecuteLambda));
}

// 2. AddPassInternal → 分配 TRDGLambdaPass 实例
template <typename P, typename L>
FRDGPass* AddPassInternal(FRDGEventName&& Name,
    const FShaderParametersMetadata* Metadata,
    const P* ParameterStruct, ERDGPassFlags Flags, L&& ExecuteLambda)
{
    auto* Pass = Passes.Allocate<TRDGLambdaPass<P, L>>(...);

    // 分支：有参数 Struct → SetupParameterPass
    if (Metadata)
        return SetupParameterPass(Pass);  // ← 进入依赖推导
    else
        return SetupEmptyPass(Pass);      // 无参数 → 跳过推导
}

// 3. SetupParameterPass → 核心分析流程
FRDGPass* SetupParameterPass(FRDGPass* Pass)
{
    SetupPassInternals(Pass);   // 设置 Pipeline、Flags 预处理
    SetupPassResources(Pass);   // ← 枚举资源 + 推导依赖（关键！）
    MarkResourcesAsProduced(Pass); // 标记被写入的资源
    return Pass;
}

// 4. SetupPassResources — 依赖推导核心
void SetupPassResources(FRDGPass* Pass)
{
    // 4a. 枚举所有纹理访问
    EnumerateTextureAccess(Pass->ParameterStruct, Pass->Flags,
        [&](FRDGView* View, FRDGTextureRef Texture, ERHIAccess Access,
           ERDGTextureAccessFlags TexFlags, FRDGTextureSubresourceRange Range)
    {
        // 为纹理的每个子资源记录状态变化
        auto& State = Texture->LastProducers[Pipeline];
        // State.Access, State.Pass, State.NoUAVBarrierHandle 记录
        Pass->TextureStates.Emplace(Texture);
    });

    // 4b. 枚举所有 Buffer 访问
    EnumerateBufferAccess(Pass->ParameterStruct, Pass->Flags,
        [&](FRDGView* View, FRDGBufferRef Buffer, ERHIAccess Access)
    {
        auto& State = Buffer->LastProducer[Pipeline];
        Pass->BufferStates.Emplace(Buffer);
    });
}

// 5. 最终依赖在 Compile 阶段通过 AddCullingDependency 建立：
//    - 同管线：LastProducer → 当前 Pass
//    - 跨管线（AsyncCompute↔Graphics）：CrossPipelineProducer/Consumer
//    - UAV Barrier Skip：跟踪 NoUAVBarrierHandle 优化

5.2 Pass 执行状态机

FRDGPass 中多个 bitfield 标志协同控制执行行为：

标志字段作用
bSkipRenderPassBegin合批后跳过的中间 Pass，不调用 BeginRenderPass
bSkipRenderPassEnd跳过的中间 Pass，不调用 EndRenderPass
bAsyncComputeBegin该 Pass 是 AsyncCompute 区间的第一个 Pass
bAsyncComputeEnd该 Pass 是 AsyncCompute 区间的最后一个 Pass
bGraphicsForkGraphics 管线 Pass 需要 Fork 到 AsyncCompute
bGraphicsJoinGraphics 管线 Pass 需要等待 AsyncCompute Join
bCulledPass 被裁剪，不执行、不分配资源
bSentinel标记为哨兵 Pass（Prologue/Epilogue），不包含用户逻辑
bDispatchAfterExecute执行后 Flush 到 RHI 线程
bRenderPassOnlyWritesPass 仅在 RenderPass 范围内写入资源（允许合批）
bExternalAccessPass标记为外部访问模式切换 Pass（RDG 自动生成）

5.3 Transient 池管理

RDG 的 Transient 资源管理委托给 IRHITransientResourceAllocator 接口 （由 GRDGTransientResourceAllocator 全局变量持有）。核心交互流程：

  1. CollectAllocations() 生成 Allocate 操作，按 Pass 执行顺序排序
  2. CollectDeallocations() 生成 Deallocate 操作，标记资源不再使用
  3. AllocateTransientResources() 将操作列表提交给 Transient 分配器
  4. 分配器内部使用区间图算法：如果两个资源的生命周期区间不重叠，它们可以安全地共享同一块物理内存
  5. FRDGBuilder 在 Transient 资源首次使用时插入 FRHITransientAliasingInfo 来标记 Discard 操作

6. UE5.8 演进变化

RDG 覆盖率提升

UE5.8 相比 5.4/5.5，RDG 覆盖率继续扩大。更多渲染 Pass 从传统 FRHICommandList 直接操作迁移到 RDG 框架中。这包括：

  • MegaLights5.8 正式：全新的大规模动态光源系统完全基于 RDG 构建， 所有光源采样、光追、降噪 Pass 均使用 RDG 管理资源依赖。
  • Lumen Lite5.8 新增：轻量级 GI 变体，所有中间资源走 RDG Transient 路径。
  • Accumulation DOF5.8 正式：多帧累积景深效果，使用 RDG 的 QueueTextureExtraction 实现帧间资源保持。
  • Movie Render Graph5.8 正式：离线渲染管线的 RDG 集成。

Async Compute Pass 的 RDG 集成

UE5.8 进一步完善了 AsyncCompute 在 RDG 中的集成。关键改进包括：

  • 自动 Fork/Join 围栏：Compile() 中的 AsyncCompute 围栏逻辑（Compile() 的 AsyncComputeFences 段落） 自动推导 Graphics 和 AsyncCompute 间的 Fork/Join 点，插入跨管线屏障。
  • Transient 资源跨管线生命周期：当 AsyncCompute 管线启用时，Transient 分配器需要考虑跨管线资源重叠， ResourcesToBegin 和 ResourcesToEnd 将 AsyncCompute Pass 的资源生命周期关联到 Graphics Fork/Join Pass。
  • DispatchPass5.8 新增：FRDGDispatchPass 允许用户在 RDG 框架内手动创建多个 RHI CommandList 并并行录制，同时享受 RDG 的自动屏障和资源管理。

RDG 调试可视化工具

UE5.8 中 RDG 调试工具体系更加完善：

Immediate Mode 调试

r.RDG.Debug 1：逐 Pass 立即执行模式，Lambda 在 AddPass 时立即调用，便于断点调试和崩溃定位。

Resource Dump 诊断

RDG_DUMP_RESOURCES 编译宏：输出帧中所有 RDG 资源的创建/使用/释放日志。

Clobber 模式 验证

r.RDG.ClobberResources 1：在 Pass 执行后用随机值填充未读资源，检测未初始化数据读取。

Visualize Pass Outputs 可视化

r.RDG.VisualizeTexture：配合 VisualizeTexture 命令可视化 RDG Pass 的中间输出纹理。

UE5.8 新增关键宏：IF_RDG_ENABLE_TRACE 用于条件编译 RDG Trace 功能； 支持输出 Graphviz DOT 格式的图可视化数据，可与 UnrealInsights 集成查看 Pass 依赖拓扑。
