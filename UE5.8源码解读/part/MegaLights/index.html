MegaLights 系统深入解读

UE5.8 正式发布的革命性功能 — 单帧内支持数百个可投影动态光源。深入分析从架构到实现的完整技术全景。

1. 概述 & 定位

MegaLights 是 UE5.8 在 2025 年正式发布 (Production Ready) 的革命性功能，解决了传统延迟渲染管线在多光源场景中的核心瓶颈："每个光源都需要逐像素计算阴影"。

通过采用 随机光源采样 (Stochastic Light Sampling) 替代传统的 Tile-Based 光源裁剪，MegaLights 实现了以固定的 GPU 开销渲染数百个可投影动态光源。这意味着：

  • 场景中可以放置数百个动态点光源、聚光灯、矩形光源，每个都带有实时阴影
  • GPU 开销不再随光源数量线性增长
  • 每个像素只追踪固定数量的阴影光线 (Shadow Rays)，而非每个光源都追踪

前提条件：MegaLights 依赖硬件光线追踪 (HWRT)。引擎将自动检测条件，不满足时回退到传统的 Clustered Deferred 路径。同时支持虚拟阴影贴图 (VSM) 作为替代阴影方法。

2.3 分帧调度策略

MegaLights 使用固定 Ray Budget 策略覆盖任意数量的光源。核心思路：

  • 每像素 K 条光线: 默认 K=4 (2x2 采样模式)，通过降采样 (Half-resolution) 进一步减少
  • RIS (重要性重采样): 对每个像素，根据光源对该像素的潜在贡献 (亮度/距离/衰减) 计算 PDF，然后按 PDF 随机采样 K 个光源
  • Guide By History: 利用上一帧的可见光/隐藏光哈希表，降低对上一帧被遮挡光源的采样概率 (PDF 权重缩放至 10%)，减少噪声
  • Tile-Based 分类: 将屏幕划分为 8x8 Tile，根据材质复杂度 (Simple/Complex/Rect) 分派不同的着色路径
  • 光线合并: 当多条光线指向同一光源或不在半影区时，复用追踪结果

关键公式：

采样预算 = (分辨率 / DownsampleFactor) * NumSamplesPerPixel * (1 或 2 方向光额外比例)

通过 r.MegaLights.DownsampleMode 和 r.MegaLights.NumSamplesPerPixel 可在性能与质量间调节。

4. 设计决策 & 权衡

4.1 随机采样噪声 vs 降噪模糊的平衡

MegaLights 的核心权衡：采样数越少 → 噪声越高 → 降噪越激进 → 细节模糊越多。默认配置下 (每像素 4 采样, Half-res, 最多 12 帧时序累积)，引擎试图在此光谱中找到最佳平衡点。

  • 直接遍历式 (传统 Deferred): 无噪, 但 O(N) 开销
  • MegaLights 默认配置: 低噪, O(1) 开销, 12 帧积累后噪声收敛
  • MegaLights 激进配置: 每像素 1 采样, 2x2 降采样, 无时序 → 纯随机噪声, 但极快

4.2 硬光追 (HWRT) vs 软光追 (SDF) 的性能/质量光谱

4.3 为什么需要独立的降噪器 (而非复用 TSR)?

  • 数据域不同: MegaLights 降噪在光照域 (DiffuseLighting/SpecularLighting) 操作, 使用反照率解调 (Demodulation); TSR 在最终颜色域操作
  • 二阶矩追踪: MegaLights 维持 LightingMoments (二阶矩) 用于自适应空间滤波, 这是 TSR 没有的
  • 时序长度不同: MegaLights 时序累积可达 12 帧, TSR 典型为 1-2 帧
  • 集成需求: 通过 MegaFilters 管道将降噪后的结果交还给 TSR, 实现两个降噪器的级联

4.4 与传统 Light Pass 的互斥/互补关系

  • 互斥: 走 MegaLights 的光源 不再 参与传统的 Clustered Deferred Light Pass, 避免双重贡献
  • 互补: 不走 MegaLights 的光源 (天空光, 禁用 MegaLights 的光源, 方向光默认) 仍然走传统路径
  • 方向光特殊处理: MegaLights 可在 ForwardLightUniformParameters 中标记方向光为 "DirectionalLightHandledByMegaLights", 但默认关闭
  • 半透明分路径: 普通半透明走传统 Forward; Front Layer Translucency 走专用 MegaLights Pass; 其余层走 Translucency Lighting Volume (TLV)

5.2 采样策略与数据结构

MegaLights 的采样数据通过压缩纹理传输, 每条采样用 PF_R32_UINT 格式的纹理编码:

纹理格式大小内容
LightSamples PF_R32_UINT SampleViewSize (降采样 * NumSamples) 每个采样的光源索引 (16bit) + 表面位置编码 (16bit)
LightSampleRays PF_R32_UINT (2D) / PF_R8_UINT (3D) 同 LightSamples 光线方向编码 + 追踪优先级标志
DownsampledSceneDepth PF_R32_FLOAT DownsampleFactor 缩放后分辨率 降采样深度, 用于采样距离衰减

5.3 降噪器 Tile 分派

降噪器采用8x8 Tile 间接分派 (Indirect Dispatch) 策略, 只处理有效 Tile, 跳过空白区域:

Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsDenoising.cpp:436-628
DenoiseLighting() — 时序 + 空间降噪管线 
void FMegaLightsViewContext::DenoiseLighting(...) {
    // 1. FastClear: 跳过空 Tile 只清除边界 (Reduce GPU 开销)
    if (UseFastClear(InputType)) {
        FClearTemporalAccumulationCS → 按 Empty Tile 列表分派
    }

    // 2. 时序累积 (Temporal): 最多 12 帧
    FDenoiserTemporalCS(ValidHistory, FastClear, OutputHistoryConfidence)
        → DiffuseLighting + SpecularLighting + LightingMoments (二阶矩)
        → 每像素 (8x8 线程组) 全屏分派

    // 3. 空间滤波 (Spatial): 8px 核半径, 4 次采样
    FDenoiserSpatialCS(SpatialFilter, InputType, FastClear, UseHistoryVariance)
        → 由时序输出的方差引导空间滤波强度
        → SubsurfaceDiffuse (次表面散射写入)
        → 最终写入 SceneColor

    // 4. MegaFilters 管道: 将结果传递给后期处理
    AddMegaFiltersPass(GraphBuilder, View, TEXT("MegaLights"), MegaFiltersInputs);

    // 5. 提取历史资源供下一帧使用
    GraphBuilder.QueueTextureExtraction(DiffuseLighting, &ViewState.DiffuseLightingHistory);
}

与 Lumen / VSM 的互操作机制

  • Lumen 共存: MegaLights 贡献 直接光照 (Direct Lighting), Lumen 负责 间接光照 (Indirect/GI)。两者在同一帧内并行处理, 各自输出到 SceneColor
  • Lumen 集成: MegaLights GBuffer 输入复用 Lumen 的 StochasticLighting 框架 (StochasticLighting::TileClassificationMark, StochasticLighting::FHistoryScreenParameters)
  • VSM 集成: MegaLights 有两条 VSM 路径 — (a) 使用 HWRT 追踪的物理准确阴影, (b) 回退到 VSM 传统阴影贴图采样 (FVirtualShadowMapTraceLightSamplesCS)。VSM 路径可预先标记需要渲染的 Shadow Page (FVirtualShadowMapMarkLightSamplesCS)
  • Nanite 兼容: HWRT 通过 Nanite RayTracing 的 PVS (Procedural Visibility Shader) 实现对 Nanite 几何的精确阴影
  • MegaFilters 管道: MegaLights 的结果在空间滤波后通过 MegaFilters 传递给后期处理, 实现与 TSR 的级联

未来方向

基于源码中的 TODO 注释 (#ml_todo), 可预期的后续工作:

  • VSM 标记对 HairStrands 的支持 (当前仅 GBuffer)
  • 更多光源类型 (当前方向光默认禁用)
  • 进一步优化光线压缩和 WaveOps 利用
  • 与 Lumen Lite 的深度集成
  • 移动端/非 HWRT 平台的 SDF 回退路径优化
