Nanite 与 Lumen

UE5.8.0 核心渲染技术深度解读

源码位置：Engine/Source/Runtime/Renderer/Private/Nanite/ 与 Engine/Source/Runtime/Renderer/Private/Lumen/

1. 概述与定位

1.1 两个革命 UE5 的技术

Nanite 用虚拟几何消灭了 DrawCall 瓶颈。传统的 GPU 渲染中，每个 DrawCall 都需要 CPU 提交几何数据——当场景包含数百万个三角形时，DrawCall 数量爆炸，CPU 成为瓶颈。Nanite 将可见性判定、裁剪和光栅化全部迁移到 GPU，CPU 仅需提交高层级的 BVH 节点，单帧 DrawCall 从数万个降到两位数。配合 Virtual Shadow Maps（VSM），阴影渲染同样受益于 GPU 驱动的几何管线。

Lumen 用实时全局光照替代了预烘焙。传统工作流中，光照贴图（Lightmap）需要数小时甚至数天的离线预计算；动态 GI 方案受限于性能只能在少数场景使用。Lumen 通过 Surface Cache（卡片表面缓存）+ Radiance Cache（世界空间辐射度探针）+ Screen Probe Gather（屏幕探针收集） 三级递进体系，在 4ms 内完成完整 GI 求解，同时支持 Software Ray Tracing（SDF/DF 追踪）与 Hardware Ray Tracing（DXR 加速）双路径。

1.2 Nanite + Lumen 在 UE5 渲染管线中的位置

图 1：Nanite 与 Lumen 在 UE5.8.0 延迟渲染管线中的位置。绿色标注的为 Nanite 参与阶段，紫色标注的为 Lumen 参与阶段。

关键关系：Nanite 和 Lumen 并非简单的先后依赖。Nanite 在第 4 阶段输出的 VisBuffer64（可见性缓冲区） 和 SceneDepth 是后续所有 Pass 的基础——BasePass 读取 VisBuffer 进行材质着色，Lumen 读取 SceneDepth 和 GBuffer 进行 GI 计算。同时，Lumen 的 Surface Cache 会单独捕获 Nanite 几何体的卡片表示（通过 ENaniteMeshPass::LumenCardCapture），使得 GI 系统能够感知 Nanite 的高细节几何。

2. 顶层架构设计（Nanite）

2.1 与传统管线的核心差异

Nanite 最大的变革在于将 可见性判定 从 CPU 完全迁移到 GPU。传统管线中，CPU 逐 Primitive 执行 Frustum Culling 和 Occlusion Culling，然后为每个可见 Mesh 提交 DrawCall——当场景包含数万个网格时，CPU 不堪重负。Nanite 将场景编码为 BVH（Bounding Volume Hierarchy），CPU 仅做实例级预裁剪，具体的 Cluster 裁剪、HZB 遮挡查询、光栅化全部在 GPU 以 Compute Shader 完成。

维度传统渲染Nanite
几何裁剪CPU Frustum/Occlusion CullingGPU Cluster Culling + Two-Pass HZB
中间输出直接写入 GBufferVisibility Buffer（VisBuffer64）
光栅化硬件光栅化（PS）软件+硬件混合光栅化（CS）
材质着色逐像素 PS 调用Compute Shader 屏幕空间分桶着色
DrawCallCPU 提交（FMeshDrawCommand）GPU Indirect Dispatch
LODCPU 离散切换（Pop）GPU 连续微多边形（无 Pop）
过绘制处理依赖 Early-Z / Pre-Z PassTwo-Pass HZB 遮挡剔除 + 深度导出
几何压缩逐顶点属性Cluster 级别量化和流式传输

2.2 四级裁剪流水线

Nanite 的 GPU 裁剪分为四个层级，从粗粒度到细粒度逐级过滤，每一级都剔除不在视野内或被遮挡的候选几何体。

图 2：Nanite 四级裁剪流水线 —— 从实例、BVH 节点、Cluster 到三角形，逐级缩小候选集。两轮 HZB 遮挡剔除是核心优化。

特别注意 Two-Pass HZB 遮挡剔除（代码中通过 CULLING_PASS_OCCLUSION_MAIN 和 CULLING_PASS_OCCLUSION_POST 实现）：第一轮 (Main Pass) 用上一帧的 HZB 剔除当前帧新可见的 Cluster；第二轮 (Post Pass) 在 Main Pass 光栅化完成后，构建新的 HZB，重新测试上一帧被遮挡的对象，将其复活。这种设计避免了单帧遮挡带来的误剔除。

2.3 VisBuffer64 与深度双缓冲策略

Nanite 的核心创新之一是 Visibility Buffer（VisBuffer64）。与传统管线直接在 GBuffer 中写入材质属性不同，Nanite 的光栅化阶段只输出 64-bit 的可见性 ID（包含 Cluster ID + Triangle ID + Material ID），而不计算任何材质。这带来了两个关键优势：

  • 光栅化极简：硬件光栅化阶段不需要执行复杂 PS，仅输出 ID；软件光栅化 Compute Shader 同样只需写入一个 64-bit 值。
  • 材质着色独立：材质计算在后续的 Compute Shader 阶段（ShadeBinning + DispatchBasePass）中完成，每个像素只执行一次材质计算，彻底消除过绘制造成的重复着色。

VisBuffer64 的分辨率与屏幕分辨率一致，同时 Nanite 还维护独立的 SceneDepth 目标。光栅化完成后，EmitDepthTargets()（NaniteComposition.cpp:256）从 VisBuffer 解析深度，写入 SceneDepth 以便后续 Pass 使用。这种双缓冲设计使得深度测试和材质着色完全解耦。

图 3：Nanite 光栅化输出的三个核心纹理 —— VisBuffer64、SceneDepth、ShadingMask

2.4 ShadeBinning 屏幕空间材质分桶

VisBuffer 光栅化完成后，进入 ShadeBinning 阶段（NaniteShading.cpp:1953 ShadeBinning()）。这个 Compute Shader 扫描全屏 VisBuffer，将每个像素归类到其对应材质的分桶（ShadingBin）中。每个 Bin 收集所有使用该材质的像素坐标，形成紧凑的 像素列表。

接着 DispatchBasePass()（NaniteShading.cpp:1665）对每个 Bin 发起一次 Compute Shader Dispatch，该 Shader 读取 Bin 中的像素列表，从 VisBuffer 解码几何信息（顶点位置、法线、UV），并执行完整的材质计算写入 GBuffer。每个像素只计算一次材质，无论场景中有多少层重叠的几何体。

图 4：ShadeBinning 屏幕空间材质分桶与 DispatchBasePass 流程

2.5 光栅调度策略

Nanite 支持三种光栅化调度策略（定义在 NaniteCullRaster.h:25 ERasterScheduling 枚举），通过 r.Nanite.ComputeRasterization 和 r.Nanite.AsyncRasterization 控制选择：

策略枚举值行为适用场景
HardwareOnly HardwareOnly = 0 所有三角形走硬件光栅化（VS/PS 或 Mesh Shader），不启用软件光栅 当 r.Nanite.ComputeRasterization=0 时强制此模式
HardwareThenSoftware HardwareThenSoftware = 1 大三角形硬件光栅化 + 小三角形软件光栅化，顺序执行 默认模式（AsyncCompute 不可用时）
HardwareAndSoftwareOverlap HardwareAndSoftwareOverlap = 2 硬件和软件光栅化异步并行执行，通过 AsyncCompute 重叠 当 r.Nanite.AsyncRasterization=1 且硬件支持时

软件光栅化与硬件光栅化的三角形大小分界线由 r.Nanite.MinPixelsPerEdgeHW（默认 32 像素）控制。当三角形的屏幕空间边长大于此阈值时走硬件路径（Mesh Shader / Vertex+Pixel Shader），小于时走软件 Compute Shader 路径。这是因为硬件光栅化对微小三角形效率极低（Rasterizer 以 2x2 像素块工作），而 Compute Shader 的软件光栅器可以精确到像素级。

3. 顶层架构设计（Lumen）

3.1 三级递进数据流：Surface Cache → Radiance Cache → Screen Probe Gather

Lumen 的 GI 计算遵循一个三级递进模型：从粗略的场景几何缓存到精细的屏幕空间光照。每一级都是上一级的"加速结构"，逐步提高精度和细节。

图 5：Lumen 三级递进数据流 —— 从粗糙的 Surface Cache 到精细的 Screen Probe Gather

三级递进的含义：

  • Surface Cache（第 1 级）：将场景几何投影为 6 方向卡片（Card），每张卡片存储 Albedo、Opacity、Normal、Emissive 等材质属性。卡片以 128x128 物理页（Physical Page）为单位在 Atlas 中分配。远处的物体占用更少的页面，实现自动 LOD。
  • Radiance Cache（第 2 级）：在世界空间中分布辐射度探针，多层 Clipmap（4 级，从 50m 覆盖到 5km）确保近处精度高、远处覆盖广。每个探针追踪半球方向的光线，将结果存储在 FinalRadianceAtlas 中。Radiance Cache 在帧间缓存，仅增量更新移动或变化区域。
  • Screen Probe Gather（第 3 级）：屏幕空间最精细层，每 16x16 像素放置一个探针。追踪短距离光线，然后与 Radiance Cache 融合——近处用精确追踪，远处用缓存插值。通过自适应探针（Adaptive Probes）在几何边缘和光照变化剧烈区域放置额外探针。

3.2 Software RT (SDF/DF) vs Hardware RT (DXR) 双路径

Lumen 支持两种光线追踪后端，根据硬件能力自动选择。

维度Software RT（SDF / Mesh Distance Field）Hardware RT（DXR / Ray Tracing）
追踪介质全局距离场 (Global SDF) + Mesh 距离场 (Mesh SDF)DXR 加速结构（BLAS / TLAS）
精度中：距离场存在细节损失，远距离足够高：精确三角形级相交测试
性能快：SDF 步进追踪，无 BVH 构建开销中等：需维护 TLAS/BLAS，但有硬件加速
硬件要求所有 SM6 GPU需 DXR 支持的 GPU（RTX 系列 / RDNA2+）
降噪需要（追踪结果噪声较大）需要（但样本质量更高）
Surface Cache 光照默认走 SDF 追踪可选启用 HWRT 加速（UseHardwareRayTracedDirectLighting）
反射Screen Probe Gather（SDF）HWRT 反射更精确，尤其镜面反射
Mesh SDF 细节每 Mesh 单独距离场，开销可控N/A（直接追踪三角形）

关键设计：Lumen 的 Software RT 并非"降级模式"，而是 默认模式。它在性能和质量的平衡上经过精心调校——Global SDF 提供粗粒度的世界表示，Mesh SDF 为近距离几何提供更精细的距离场。Hardware RT 作为可选加速，需要项目设置中的显式启用（r.RayTracing.ForceAllRayTracingEffects），提供更高质量的反射和更精确的 GI。

3.3 ReSTIR 在直接光照和反射中的重要性重采样

ReSTIR（Reservoir-based Spatiotemporal Importance Resampling） 是 Lumen 用于高效采样大量光源的核心技术。在场景中存在成百上千个动态光源时，传统方案需要为每个像素追踪多条阴影光线——这在计算上不可行。ReSTIR 的工作原理是：

  • 每像素维护一个 Reservoir（水库）：存储从大量候选光源中重要性采样选出的少数"最佳"光源样本。
  • 时间复用：当前帧的 Reservoir 继承上一帧的采样结果，通过重采样（Resample）将旧样本与新候选合并。
  • 空间复用：相邻像素的 Reservoir 互相共享样本，通过加权重采样选出对当前像素最重要的光源。
  • 结果：每像素只需追踪 1~2 条阴影光线，就能准确渲染数百个动态光源的阴影效果。

在 Lumen 反射中，ReSTIR 也被用于重要性采样反射方向，提高粗糙表面反射的收敛速度。相关源码在 LumenReSTIRGather.cpp 中实现。

3.4 辐射度缓存体系

Radiance Cache 是 Lumen 性能的关键。其设计要点（LumenRadianceCache.cpp）：

  • 多层 Clipmap：默认 4 级 Clipmap，每级分辨率相同但覆盖范围不同（如 50m、200m、800m、3200m），确保近处探针密度高、远处覆盖广。
  • 增量更新：每帧只更新一小部分探针（约 1/32），通过 GLumenDirectLightingUpdateFactor = 32 控制更新速度。
  • 探针偏移：通过 GRadianceCacheIrradianceProbeOffsetMode（1=使用 GBuffer 偏移）将探针"推"到最近的表面上，避免探针落在几何体内部或空中。
  • 跟踪瓦片排序：GRadianceCacheSortTraceTiles 将追踪瓦片按方向排序以提取 GPU Cache 一致性。
  • 空间滤波：CVarRadianceCacheFilterProbes 对相邻探针进行空间滤波以减少噪声。

5. 设计决策与权衡

5.1 软件光栅 vs 硬件光栅策略选择

Nanite 的核心权衡之一：在 Compute Shader 中做软件光栅化，还是走传统硬件光栅化管线？

因素软件光栅（Compute Shader）硬件光栅（HW Rasterizer）
小三角形优秀：精确像素级光栅化，无 Quad 浪费极差：2x2 Quad 开销巨大，仅 1 像素有效
大三角形较差：大量线程处理少数三角形效率低优秀：硬件 Rasterizer 专门优化
阈值三角形边长 < 32 像素（r.Nanite.MinPixelsPerEdgeHW）三角形边长 ≥ 32 像素
可编程性完全灵活，可任意定制受 Rasterizer 固定管线限制
位移/曲面细分天然支持（Dice → Micropolygon）需要 Mesh Shader 支持

r.Nanite.MinPixelsPerEdgeHW = 32.0 的默认值经过了大量性能测试：三角形边长超过 32 像素时，硬件光栅化的 Quad 利用率足够高；小于 32 像素时，软件光栅化（FMicropolyRasterizeCS）的逐像素精确控制更加高效。

5.2 Surface Cache 内存占用 vs 精度权衡

Lumen 的 Surface Cache 使用一张 4096x4096 的 Atlas 纹理（r.LumenScene.SurfaceCache.AtlasSize），所有场景几何的卡片在此 Atlas 中分配空间。这带来了直接的内存 vs 精度的权衡：

  • 分辨率分配：每张卡片页（Card Page）为 128x128 物理像素。近距离物体分配更多页面（高分辨率），远距离物体分配更少页面（低分辨率）。页面的分配和释放通过 SurfaceCacheFeedback 系统驱动。
  • Atlas 碎片化：长时间运行时，Atlas 可能出现碎片化。Lumen 通过 LRU 缓存淘汰和页面重分配处理碎片化问题。
  • Atlas 放大可行：r.LumenScene.SurfaceCache.AtlasSize 可以放大到 8192 或更高，以支持更多几何细节，但会线性增加 VRAM 占用。
  • 分帧更新：r.LumenScene.DirectLighting.UpdateFactor = 32 确保每帧只更新约 3% 的卡片，将开销分摊到多帧。

5.3 Screen Probe 的屏幕空间信息限制

Screen Probe Gather 虽然是 Lumen 最精细的 GI 层，但存在 屏幕空间的固有限制：

  • 屏幕外信息丢失：屏幕外的光源/几何无法直接通过 Screen Probe 感知。这由 Radiance Cache（世界空间探针）和 Surface Cache 来弥补。
  • 镜面反射回退：屏幕空间无法获取镜面反射的另一侧信息。Lumen 反射使用 Radiance Cache 作为 Ray Guiding 的远距离回退方案。
  • 自适应探针：在几何边缘、阴影边界等光照变化剧烈区域，通过 r.Lumen.ScreenProbeGather.NumAdaptiveProbes（默认 8 个/Uniform 探针）放置额外探针来减少漏光。

5.4 SkeletalMesh/Foliage 对 Nanite 的支持限制及原因

Nanite 对骨骼网格（SkeletalMesh）和植被（Foliage）的支持在 UE5.8 中仍在发展中：

  • SkeletalMesh Skinning：源码中通过 NaniteSkinnedMeshesSupported() 检查平台支持（NaniteCullRaster.cpp:1383, 1637），并在 FSkinningDim 着色器排列中编译蒙皮路径。5.8 中 Skinned Mesh 的 Nanite 支持需要 Velocity 导出场景才启用（NaniteComposition.cpp:366），且仅限于支持 WPO 动画类骨骼网格。
  • 植被限制：植被（随风吹动的树叶、草）的主要挑战是顶点动画和非刚性变形——Nanite 的 BVH 是预计算的，运行时骨骼/顶点动画需要更新 BVH 边界或使用动态节点边界。源码中有 CVarNaniteCullingSkinnedNodeBounds（默认开启）来使用皮肤节点边界替代实例边界。
  • 设计根源：Nanite 的核心假设是几何体在静态空间中是连续的、不变的——这使得 BVH 预计算和流式传输可行。骨骼网格需要每帧更新顶点位置，破坏了这一假设。5.8 的进展是引入了有限的蒙皮支持（通过 WPO/Displacement），但完整的 SkeletalMesh 支持仍需要更深入的架构改动。
  • WPO Disable Distance：通过 CVarNaniteCullingWPODisableDistance，远距离的 WPO 可以被禁用，减少更新开销。

6. UE5.8 演进变化

6.1 异步光栅化增强

UE5.8 引入了更细粒度的异步光栅化控制：

  • Shadow Depths 异步：r.Nanite.AsyncRasterization.ShadowDepths（默认 0，实验性）允许 VSM 的深度渲染与主视图异步进行。
  • Custom Pass 异步：r.Nanite.AsyncRasterization.CustomPass（默认 1）允许 Custom Depth/Stencil Pass 异步光栅化。
  • Lumen Mesh Cards 异步：r.Nanite.AsyncRasterization.LumenMeshCards（默认 0，实验性）允许 Lumen Card 捕获异步进行。
  • Shader Bundle：UE5.8 引入了 Shader Bundle 机制（r.Nanite.Bundle.Raster、r.Nanite.Bundle.Shading），允许将多个光栅化/着色 Dispatch 合并为一个 GPU 命令，减少 CPU 开销。
  • 动态分辨率缩放：GDynamicNaniteScalingPrimary 和 GDynamicNaniteScalingShadow 实现基于时间预算的动态 LOD 缩放——当 GPU 超预算时自动降低 MaxPixelsPerEdge 以提高性能。

6.2 Lumen Lite 简化模式（5.8 新增）

UE5.8 中虽然没有直接名为 "LumenLite" 的代码文件，但通过以下机制实现了简化/降级路径：

  • UpdateFactor 调节：提高 r.LumenScene.DirectLighting.UpdateFactor（从 32 到 64 或 128）可降低每帧更新的卡片数量，减少 GPU 开销。
  • Surface Cache 冻结：r.LumenScene.SurfaceCache.Freeze 可冻结 Surface Cache，在静态场景中避免每帧重新捕获。
  • 降级到仅 Radiance Cache：关闭 r.Lumen.ScreenProbeGather 时，Lumen 回退到仅使用世界空间探针（Radiance Cache），省去屏幕探针开销。
  • 远场 Ray Tracing 遮挡模式：r.LumenScene.FarField.OcclusionOnly 将远场追踪限制为仅遮挡查询，跳过着色计算。

6.3 SkeletalMesh/Foliage 的 Nanite 支持进展

UE5.8 中的进展包括：

  • FSkinningDim 排列：Nanite 光栅化和深度导出着色器现在包含蒙皮排列（NaniteCullRaster.cpp:1383），允许在光栅化时执行骨骼变换。
  • Assembly Transforms：NaniteShared.h 中定义了 AssemblyTransforms 缓冲区，用于将蒙皮后的顶点位置传递给后续 Pass。
  • Voxel 和 Curve 支持：FNaniteRasterPipeline 结构中包含 bVoxel 和 bCurve 标志（NaniteShared.h:633-634），材质可见性测试中也区分了 Triangle、Voxel、Curve 三种 Bin 可见性，表明 Nanite 正在扩展到非三角形几何类型。
  • Tessellation Displacement：r.Nanite.Tessellation 默认启用运行时曲面细分/位移，支持雕刻类几何的细节增强。相关参数 r.Nanite.DicingRate 控制微多边形大小。
