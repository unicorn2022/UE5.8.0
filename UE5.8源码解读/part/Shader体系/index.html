Shader 编译体系

从 .usf 文件到 GPU 可执行代码的全流程 — UE 的跨平台 Shader 编译体系

1. 概述与定位

Unreal Engine 的 Shader 编译体系是整个渲染管线的入口基础设施。它负责将开发者编写的 .usf（Unreal Shader File）编译为目标平台 GPU 可执行的字节码，并为运行时提供高效的类型安全的参数绑定机制。

与典型的游戏引擎不同，UE 的 Shader 系统不仅仅是"编译 HLSL"，它提供了一整套编译期反射、变体（排列）管理、DDC 缓存和跨平台支持的框架。

2. 顶层架构设计

2.1 Shader 类型层级

UE 的 Shader 类型系统采用编译期多态 + 运行时动态分发的混合架构。所有 Shader 类都继承自 FShader，但其元类型 (ShaderMetaType) 决定了 Shader 的编译、缓存和查找方式。

关键设计: FShaderType 使用 EShaderTypeForDynamicCast 枚举提供安全的运行时向下转型 (类似 LLVM 的 dyn_cast)。每种 ShaderMetaType 存储不同的函数指针 (构造器、ShouldCompilePermutation 等)，以多态方式支持不同的编译策略。

2.2 SHADER_PARAMETER_STRUCT 宏体系

UE5 的 SHADER_PARAMETER_STRUCT 宏系统是整个参数绑定的核心，它利用 C++ 的编译期反射 + 侵入式单向链表 技术，自动从参数结构体生成元数据和运行时绑定代码。

2.3 Shader 排列系统

UE 使用 TShaderPermutationDomain 在编译期将多维变体参数映射为一维排列 ID。这避免了运行时的字符串查询，将所有排列索引计算完全在编译期完成。

宏维度类型示例PermutationCount

SHADER_SPECIALIZATION_BOOL("D")bool (SpecConst)ENABLE_FEATURE2
SHADER_PERMUTATION_INT("D", N)int32 [0..N-1]QUALITY_LEVELN
SHADER_PERMUTATION_RANGE_INT("D", X, N)int32 [X..X+N-1]SAMPLE_COUNTN
SHADER_PERMUTATION_SPARSE_INT("D", v...)int32 (稀疏)1, 2, 4, 8N
SHADER_PERMUTATION_ENUM_CLASS("D", E)枚举类 (0..MAX-1)ELightingModelMAX

排列域的组合

C++ ShaderPermutation.h 使用示例
 1class FMyDimA : SHADER_PERMUTATION_BOOL("MY_DIM_A");
 2class FMyDimB : SHADER_PERMUTATION_INT("MY_DIM_B", 4);  // 0..3
 3
 4// 编译期: PermutationCount = 2 * 4 = 8
 5using FMyDomain = TShaderPermutationDomain<FMyDimA, FMyDimB>;
 6
 7// 设置排列向量
 8FMyDomain PermutationVector;
 9PermutationVector.Set<FMyDimA>(true);
10PermutationVector.Set<FMyDimB>(3);
11
12// 得到一维排列 ID = 1*1 + 3*2 = 7
13int32 PermutationId = PermutationVector.ToDimensionValueId();
14
15// ID 还原为向量
16FMyDomain Restored(7);
17assert(Restored.Get<FMyDimA>() == true);
18assert(Restored.Get<FMyDimB>() == 3);
排列 ID 计算: 对于 n 维域 [D0, D1, ..., Dn-1]，排列 ID = SUMi (dimi_val * PRODj<i dimj_count)。这是编译期完成的，零运行时开销。

以下是完整的自定义 PixelShader 实现示例，展示从 .usf 源码到 RDG 调度的完整流程。

(1) Shader 源文件: MyCustomShader.usf

HLSL /Engine/Private/MyCustomShader.usf 
 1#include "/Engine/Private/Common.ush"
 2
 3// 与 C++ BEGIN_SHADER_PARAMETER_STRUCT 中的 SHADER_PARAMETER 一一对应
 4float4 TintColor;         // SHADER_PARAMETER(FVector4f, TintColor)
 5float  BlendFactor;       // SHADER_PARAMETER(float, BlendFactor)
 6Texture2D InputTexture;    // SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture)
 7SamplerState InputSampler; // SHADER_PARAMETER_SAMPLER(SamplerState, InputSampler)
 8
 9void MainPS(
10    in float2 UV : TEXCOORD0,
11    out float4 OutColor : SV_Target0)
12{
13    float4 TexColor = InputTexture.Sample(InputSampler, UV);
14    OutColor = lerp(TexColor, TintColor, BlendFactor);
15}

(2) C++ 头文件: MyCustomShader.h

C++ Renderer/Private/MyCustomShader.h 
#pragma once

#include "GlobalShader.h"
#include "RenderGraphUtils.h"

// ===== 定义全局 PixelShader =====
class FMyCustomPS : public FGlobalShader
{
public:
    DECLARE_GLOBAL_SHADER(FMyCustomPS);
    SHADER_USE_PARAMETER_STRUCT(FMyCustomPS, FGlobalShader);

    // ===== 参数结构体 =====
    BEGIN_SHADER_PARAMETER_STRUCT(FParameters, )
        SHADER_PARAMETER(FVector4f, TintColor)
        SHADER_PARAMETER(float, BlendFactor)
        SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture)
        SHADER_PARAMETER_SAMPLER(SamplerState, InputSampler)
        RENDER_TARGET_BINDING_SLOTS()
    END_SHADER_PARAMETER_STRUCT()

    // ===== 排列示例: 一个 bool 维度 =====
    class FUseHDR : SHADER_PERMUTATION_BOOL("USE_HDR");
    using FPermutationDomain = TShaderPermutationDomain<FUseHDR>;

    // 控制哪些排列被编译
    static bool ShouldCompilePermutation(const FGlobalShaderPermutationParameters& Parameters)
    {
        // 只在 SM5+ 平台编译
        return IsFeatureLevelSupported(Parameters.Platform, ERHIFeatureLevel::SM5);
    }
};

// ===== 简易调度器 =====
void AddMyCustomPass(FRDGBuilder& GraphBuilder, FRDGTextureRef InputRT, FRDGTextureRef OutputRT);

(3) C++ 实现文件: MyCustomShader.cpp

C++ Renderer/Private/MyCustomShader.cpp 
#include "MyCustomShader.h"
#include "PixelShaderUtils.h"
#include "Renderer/Private/PostProcess/PostProcessing.h"

// 注册 Shader 类型到全局表 (关键!)
// 参数: ShaderClass, SourceFile, EntryFunction, Frequency
IMPLEMENT_GLOBAL_SHADER(FMyCustomPS, "/Engine/Private/MyCustomShader.usf", "MainPS", SF_Pixel);

void AddMyCustomPass(FRDGBuilder& GraphBuilder, FRDGTextureRef InputRT, FRDGTextureRef OutputRT)
{
    // 1. 获取全局 ShaderMap
    FGlobalShaderMap* ShaderMap = GetGlobalShaderMap(GMaxRHIFeatureLevel);

    // 2. 选择排列
    FMyCustomPS::FPermutationDomain PermutationVector;
    PermutationVector.Set<FMyCustomPS::FUseHDR>(false);
    TShaderMapRef<FMyCustomPS> PixelShader(ShaderMap, PermutationVector);

    // 3. 设置参数
    FMyCustomPS::FParameters* PassParameters =
        GraphBuilder.AllocParameters<FMyCustomPS::FParameters>();
    PassParameters->TintColor      = FVector4f(1.0f, 0.5f, 0.2f, 1.0f);
    PassParameters->BlendFactor    = 0.3f;
    PassParameters->InputTexture   = InputRT;
    PassParameters->InputSampler   = TStaticSamplerState<>::GetRHI();
    PassParameters->RenderTargets[0] = FRenderTargetBinding(OutputRT, ERenderTargetLoadAction::ENoAction);

    // 4. 添加到 RDG
    FPixelShaderUtils::AddFullscreenPass(
        GraphBuilder,
        ShaderMap,
        RDG_EVENT_NAME("MyCustomPass"),
        PixelShader,
        PassParameters,
        FIntRect(0, 0, OutputRT->Desc.Extent.X, OutputRT->Desc.Extent.Y));
}

4. 设计决策与权衡

4.1 编译期反射宏 vs 手动绑定

编译期反射方案 (SHADER_PARAMETER_STRUCT)

方面优势代价
类型安全编译期检查参数名和类型; 运行时 ValidateShaderParameters 验证所有资源已设置宏展开复杂, 调试困难; 错误信息常指向宏内部而非用户代码
自动化自动生成 FShaderParametersMetadata (布局/偏移/类型); 自动绑定到 RHI侵入式单向链表存储每个成员; END_SHADER_PARAMETER_STRUCT 需逆序遍历重建数组
代码量每个 Shader 的参数定义简化为几行宏调用生成的模板代码编译时间长; .usf 中的变量名必须与 C++ 中的宏名精确匹配

核心权衡: 编译期反射牺牲了编译速度和模板复杂度，换取了运行时零开销的类型安全参数绑定。在 UE5.8 中，RDG 参数 (SHADER_PARAMETER_RDG_*) 不仅绑定资源还自动插入资源转换 Barrier，这是手动绑定无法达成的。

4.2 Shader 排列爆炸的缓解策略

排列爆炸问题: 如果 N 个材质功能各有一个 bool 维度 (Quality, Shadow, Fog, Reflection...)，排列总数 = 2N。N=10 时即达 1024 排列。实际项目中 50+ 维度很常见，理论排列数远超合理范围。

UE5.8 的缓解策略:

  • ShouldCompilePermutation 过滤器: 每个 Shader 类可重写此函数，按平台/功能等级过滤，只编译实际使用的排列
  • ShouldPrecachePermutation 预缓存控制: 区分 "Precached" / "NotPrecached" / "NotUsed"，运行时只预缓存必要变体
  • Specialization Constants: 使用 SHADER_SPECIALIZATION_BOOL (而非 SHADER_PERMUTATION_BOOL) 将编译期变体转为运行时 Specialization Constant，减少编译次数但保留运行时特化
  • 延迟编译 (ODSC): On-Demand Shader Compilation 在需要时才编译特定排列，而非预编译所有
  • 减少宏维度: 合并相关功能为枚举维度 (SHADER_PERMUTATION_ENUM_CLASS) 而非多个 bool

4.3 Cook 时预编译 vs 运行时按需编译

策略适用场景优点缺点
Cook 预编译 发行版构建 零运行时编译延迟; 所有 Shader 质量保证 Cook 时间长; 大量排列导致 DDC 膨胀
运行时按需编译 (ODSC) 编辑器/开发 Cook 快速; 只编译实际用到的排列 首次编译卡顿; 用户可能遭遇 Shader 编译等待

4.4 DXC vs FXC 的选择

UE5.8 默认使用 DXC (DirectX Shader Compiler) 编译 Shader Model 6 代码。

特性FXC (旧)DXC (新)
Shader Model最大 SM 5.1SM 6.0+ (6.6/6.7)
Wave Intrinsics不支持WaveActiveSum, WavePrefixSum, WaveReadLaneAt...
16-bit 类型不支持min16float (硬件加速)
Bindless Resources不支持ResourceDescriptorHeap / SamplerDescriptorHeap
Raytracing (DXR)不支持RayQuery, TraceRay, CallShader
网格着色器不支持AmplificationShader + MeshShader

Wave Intrinsics 在 UE5.8 中的关键应用: Nanite 裁剪/光栅化使用 WaveActiveCountBits 做 Cluster 级别压缩; Lumen 使用 WavePrefixSum 做自适应采样分配; MegaLights 使用 Wave 操作做光源分组剔除。

5. 底层实现关联

5.1 SHADER_PARAMETER_STRUCT 宏展开机制

SHADER_PARAMETER_STRUCT 宏系统的核心技巧是 编译期的侵入式单向链表。每个 SHADER_PARAMETER 宏调用都在结构体内生成一个 zzMemberId 嵌套类型，并通过函数指针串联前后成员。

C++ 宏展开伪代码 (简化) 
// SHADER_PARAMETER(float, Intensity) 展开为:
struct zzMemberId##__LINE__ {
    using PrevMemberId = zzLastMemberId;       // 指向前一个成员 ID
    static constexpr int32 MemberOffset = __COUNTER__;
};
using zzLastMemberId = zzMemberId##__LINE__;  // 更新链表尾

// 存储实际数据成员
typename TShaderParameterTypeInfo<float>::Type Intensity;

// 元数据收集器函数 (链表中每个节点的 zzFuncPtr 指向它)
static void* zzAppendMember##__LINE__(
    int32 zzMemberId, TArray<FShaderParametersMetadata::FMember>* Members)
{
    // 填充 FMember: Name, Offset, BaseType, Precision, NumElements, ...
    Members->Emplace(...);
    return reinterpret_cast<void*>(zzFuncPtrPrev); // 返回指向前一个节点
}

// END_SHADER_PARAMETER_STRUCT 中的 zzGetMembers() 逆序遍历链表
// 将函数指针数组反向, 按成员顺序填充 FMember 列表

5.3 Shader 编译 Job 调度

FShaderCompilingManager 管理所有编译任务的调度和执行。核心流程:

  1. ProcessAsyncTasks: 主循环，处理传入和完成的编译任务
  2. ProcessFinishedJob: 处理单个完结 Job，将 MapResults 反射应用到 FShaderMap
  3. ProcessCompiledShaderMaps: 批量处理完成的 Shader Map，触发材质重编译
  4. ProcessAsyncResults: 轮询 SCW 进程的结果

Job 优先级通过 EShaderCompileJobPriority 控制 (Low → Normal → High → ExtraHigh → ForceLocal)。全局 Shader 默认为 High 优先级。

6. UE5.8 演进变化

DXC 成为默认编译器

UE5.8 在所有支持平台上默认使用 DXC (DirectX Shader Compiler)，FXC 仅通过 r.D3D.ForceDXC 0 回退使用。DXC 基于 LLVM 后端，支持 SM 6.0+ 的所有特性。

Shader Model 6.6 / 6.7 Wave Intrinsics

SM 6.6 引入 WaveSize 属性、16-bit 类型和 Enhanced Barriers。SM 6.7 引入 QuadAny/QuadAll、I8Dot 等。UE5.8 的 Nanite/Lumen 大量使用 Wave Intrinsics:

  • Nanite Cluster Culling: WaveActiveCountBits 压缩存活 Cluster 列表
  • Lumen Radiance Cache: WavePrefixSum 自适应探针分配
  • MegaLights: WaveReadLaneAt 分组光源剔除

Nanite/Lumen Compute Shader 专用路径

UE5.8 中，Nanite 的材质着色 (NaniteShading.cpp) 和 Lumen 的 ReSTIR Gather (LumenReSTIRGather.cpp) 完全以 Compute Shader 实现，绕过传统 PS/VS 硬件光栅化管线:

  • Nanite: VisBuffer64 → Screen-Space ShadeBinning → DispatchBasePass (Compute)
  • Lumen: 屏幕探针 → Wave 自适应采样 → 去噪 → 合成 (Compute)

r.D3D.ForceDXC1强制使用 DXC 编译器; 0 = 回退到 FXC
r.ShaderDevelopmentMode01 = 启用详细 Shader 日志; 控制 LogShaders 运行时输出
r.Shaders.Symbols0生成调试符号: 1=启发式, 2=全部, 3=全部+强制生成
r.Shaders.Optimize1优化控制: 0=无优化, 1=默认, 2=最大优化
r.Shaders.FastMath1启用快速数学 (精度换速度)
r.Shaders.ZeroInitialise1零初始化未绑定的 CB 变量
r.Shaders.BoundsChecking0启用运行时边界检查
r.Shaders.WarningsAsErrors0将 Shader 编译警告视为错误
r.Shaders.RemoveDeadCode1移除未使用代码
r.Shaders.SkipCompression0跳过 Oodle 压缩
r.ShaderCompiler.RecompileShadersOnSave0保存 .usf 时自动重编译 (编辑器功能)
r.ShaderCompiler.LogSlowJobThreshold5.0超过秒数的编译任务打印日志
r.ShaderCompiler.DumpDDCKeys0导出 DDC 键到文件用于调试
r.Shaders.Validation1运行时 Shader 验证
r.SceneCulling1GPU 场景裁剪 (Nanite 依赖)
