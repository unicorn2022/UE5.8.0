1. 概述 & 定位

延迟渲染管线的核心思想

延迟渲染管线将几何体属性写入与光照计算解耦为两个阶段：首先在 BasePass 中把所有不透明物体的材质属性（法线、反照率、粗糙度等）写入一组 G-Buffer 纹理；随后在 Lighting Pass 中，基于 G-Buffer 和深度信息对每个像素独立完成光照计算。这种设计使得光照计算复杂度仅与屏幕分辨率相关，而非场景几何复杂度。

2.2 G-Buffer 布局

G-Buffer 是延迟渲染的核心数据结构。BasePass 将所有不透明物体的表面属性写入这组 RenderTarget，后续的 Lighting Pass 依据这些数据独立完成光照计算。

Render Target通道内容格式位深说明
SceneDepthDepth (32-bit float)R32_FLOAT (D32)32 bits深度缓冲，PrePass 写入、BasePass 只读（DDM_AllOpaque 模式下）
GBufferAWorldNormal.xyz + PerObjectGBufferData (AO)R10G10B10A232 bits世界空间法线（编码）+ 逐对象 AO 数据
GBufferBMetallic + Specular + Roughness + ShadingModelIDR8G8B8A832 bits金属度 / 高光度 / 粗糙度 / 着色模型 ID
GBufferCBaseColor.rgb + IndirectIrradianceR8G8B8A832 bits基础颜色（反照率）+ 预计算间接辐照度
GBufferDCustomData (各种用途)R8G8B8A832 bits自定义数据通道（毛发、Subsurface、ClearCoat 等）
GBufferEPrecomputedShadowFactorsR8G8B8A832 bits预计算阴影因子 + 静态光照
SceneVelocityMotionVectors.xyR16G16_FLOAT32 bits运动矢量（可选写入 BasePass 或独立 Pass）
SceneStencilStencil (8-bit)D24S88 bits光照通道标记、Distortion 标记、Subsurface 标记

总 GBuffer 带宽: Depth 4B + A 4B + B 4B + C 4B + D 4B + E 4B + Velocity 4B + Stencil 1B = 约 29 bytes/pixel (@ 1080p: ~60 MB; @ 4K: ~240 MB)

2.3 Clustered Deferred Shading — 聚簇划分策略

UE5 默认使用 Clustered Deferred Shading，通过将视锥体沿深度方向划分为 3D 网格（Froxel），实现高效的光源-像素匹配。

图 3: 3D 视锥体网格划分示意图 — Froxel 是 3D 空间中的最小聚簇单元

聚簇策略对比

策略划分方式内存开销光源分配粒度UE 选择
Tiled Shading屏幕空间 2D 瓦片 (如 16x16)低粗 — 深度范围不可知已弃用
Clustered Shading3D 视锥体网格 (XY 空间 + Z 深度)中精细 — 指数 Z 划分适配近处精细度 默认 (SM5+)
Tiled+Clustered2D 瓦片内进一步按深度分簇高最精细不必要(Clustered 已足够)

UE5.8 的 Frustum Voxel (Froxel) 深度划分策略：Z 轴使用指数分布而非均匀分布。近裁剪面附近分配更多切片以匹配透视投影下的几何密度，远裁剪面附近减少切片数量。这使得在 PrepareForwardLightData() 中每个 Froxel 只需处理少数几个光源，极大降低逐像素光照的计算量。

3.2 逐阶段展开

阶段 1: PrePass + Nanite — 深度预通道

入口: RenderPrePass() DeferredShadingRenderer.cpp:2464

  • DDM_None: 不执行 PrePass，BasePass 直接写深度
  • DDM_AllOpaque: 所有不透明几何 Depth-Only Pass，BasePass 只读深度
  • DDM_AllOpaqueNoVelocity: 同上，Velocity 独立 Pass 写深度（减少 BasePass 带宽）
  • Nanite: GPU 裁剪 VisBuffer 光栅化 EmitDepthTargets 解析深度

输出资源: SceneDepth.Z (D32), VisBuffer64, HZBTexture, PartialDepth

阶段 2: ShadowDepth — 阴影深度渲染

入口: RenderShadowDepthMaps() DeferredShadingRenderer.cpp:3291

  • VSM (Virtual Shadow Maps): 页面化 128x128 虚拟阴影贴图，Nanite 几何单 Pass 覆盖所有光源
  • CSM (Cascaded Shadow Maps): 方向光级联阴影（传统路径）
  • RT Shadows: 光线追踪阴影（可选，r.RayTracing.Shadows）

输出资源: ShadowDepthAtlas, VSM Physical Page Pool, VSM Page Table

阶段 3: BasePass — G-Buffer 写入

入口: RenderBasePass() BasePassRendering.cpp (static 方法) DeferredShadingRenderer.cpp:3059

  • 传统几何: FMeshDrawCommand 硬件光栅化 Pixel Shader 写 G-Buffer (GBufferA~E + Velocity)
  • Nanite 几何: ShadeBinning DispatchBasePass (Compute Shader) 材质着色 写 G-Buffer
  • DBuffer Decal: 延迟 Decal 写入 DBufferA/B/C（在 BasePass 之前读取）

输出资源: GBufferA (Normal+AO), GBufferB (Metallic/Roughness), GBufferC (BaseColor/Indirect), GBufferD (CustomData), GBufferE (PreShadow), SceneVelocity, SceneStencil (LightChannels+Tags)

阶段 4: Deferred Lighting — 延迟光照计算

入口: RenderLights() LightRendering.cpp DeferredShadingRenderer.cpp:3467

  • RenderDiffuseIndirectAndAO: Lumen GI Screen Probe / ReSTIR Gather + DFAO
  • RenderLights: Clustered Deferred Shading — 对每个 Froxel 累加其关联光源的贡献
  • RenderMegaLights (5.8): 大量动态光源 (Sample Shade Denoise Resolve)
  • RenderDeferredReflectionsAndSkyLighting: SSR / Lumen Reflection / Planar / SkyLight
  • RenderTranslucencyLightingVolume: 3D 体积光照注入（供半透明采样）

关键数据流: G-Buffer (读) LightGrid (读) SceneColor (写，累加结果)

阶段 5: Atmospheric — 大气/雾/体积云

入口: RenderLightShaftSkyFogAndCloud() DeferredShadingRenderer.cpp:3723

  • SkyAtmosphere: Rayleigh/Mie 散射 天空光照 LUT 合成到 SceneColor
  • HeightFog + LocalFogVolume: 高度雾 + 局部雾体积 Separate Fog Composition 或直接合成
  • VolumetricCloud: 体积云渲染（Async Compute 可选）
  • LightShaft: 光束遮挡/泛光效果

阶段 6: Translucency — 半透明 Forward 渲染

入口: RenderTranslucency() TranslucentRendering.cpp DeferredShadingRenderer.cpp:4011

  • FrontLayerTranslucency: 前层半透明（Lumen/MegaLights 预处理）
  • Standard Translucency: Forward 渲染，从后到前混合到 SceneColor
  • Separate Translucency: 低分辨率独立渲染（后处理阶段 Upsample 合成）
  • Distortion: 折射/变形效果（屏幕空间 UV 偏移）
  • OIT (Order Independent Transparency): 排序无关半透明（三角形排序 + 累积）

阶段 7: PostProcessing — 最终图像处理

入口: AddPostProcessingPasses() PostProcess/PostProcessing.cpp DeferredShadingRenderer.cpp:4311

  • TSR (Temporal Super Resolution): 时序超采样（默认反走样+超分辨率方案）
  • TAA: 时序反走样（TSR 降级方案）
  • DOF (Depth of Field): Accumulation DOF (5.8 新增) / Diaphragm DOF
  • MotionBlur: 运动模糊
  • Bloom: 泛光效果（FFT 加速可选）
  • Tonemap: 色调映射 (ACES / Filmic) + 曝光 (Auto-Exposure Histogram)
  • EyeAdaptation: 自动曝光 (Histogram + 曲线 LUT)
  • ColorGrading: LUT 颜色分级
  • LensFlares: 镜头光晕
  • Final Output: 输出到 ViewFamilyTexture RenderTarget

4. 设计决策 & 权衡

4.1 G-Buffer 带宽 vs Pixel Shader 复杂度

大量 G-Buffer 的好处

更丰富的材质数据，减少 PS 后续计算

支持复杂光照模型（直接读取预计算值）

减少光照 Pass 中的纹理采样和指令数

大量 G-Buffer 的代价

显存带宽压力大：~29 bytes/pixel x 8M pixels (4K) 232 MB 单帧写入

BasePass 填充率瓶颈（ROP 压力）

非 Nanite 几何的 Overdraw 浪费（传统路径多次写同一像素）

UE5.8 的策略: BasePass 可选输出 Velocity (bBasePassCanOutputVelocity)，在 DDM_AllOpaque 模式下 BasePass 可以输出运动矢量，避免额外的 Velocity Pass。同时 r.SelectiveBasePassOutputs 允许着色器只写到相关 RT，减少不必要带宽。

4.2 Clustered vs Tiled Deferred — 为什么选 Clustered

Tiled Shading 的局限

只能按 2D 屏幕空间划分 Tile，无法区分深度

一个 Tile 可能覆盖从近到远的所有光源

光源列表过大，每像素需遍历过多光源

Clustered Shading 的优势

Z 轴指数划分：近处更多切片匹配透视投影

每个 Froxel 仅关联影响范围内的光源

遮挡剔除(HZB)可进一步减少 Froxel 光照数量

支持 MegaLights 大量光源（每个 Froxel 高效采样）

UE5.8 控制: r.UseClusteredDeferredShading 1 (默认启用)，实现位于 LightRendering.cpp:1684 和 Froxel/Froxel.h。

4.3 Early-Z Pass 的收益与成本

EarlyZPassMode行为GPU 时间带宽适用场景
DDM_None无 PrePass，BasePass 独立写深度最短最低移动端、简单场景
DDM_AllOccluders仅遮罩体写入深度较短较低有大量遮罩体的场景
DDM_AllOpaque全部不透明几何写入深度，BasePass 只读深度 + 无 depth test 开销适中较高 UE5 默认 (Nanite 强制)
DDM_AllOpaqueNoVelocity同 DDM_AllOpaque + 独立 Velocity Pass 补全深度写较长最高需精确运动矢量的场景

关键权衡: DDM_AllOpaque 在 BasePass 中可跳过 depth test，节省 Pixel Shader 的SV_Depth输出带宽。但由于 PrePass 的额外 DrawCall 和 ROP 写，在小三角形或低 Overdraw 场景下可能比 DDM_None 更慢。UE5 因 Nanite 必然输出深度（VisBuffer EmitDepth），所以 check(!UseNanite(ShaderPlatform) || bNeedsPrePass) 确保 Nanite 场景强制走 PrePass。

4.4 为什么半透明用 Forward 而非 Deferred

如果半透明用 Deferred...

每个半透明对象需独立的 G-Buffer Pass

G-Buffer 仅能存单层表面数据（不能存多层）

Blending 需要多次 Sample、计算、写回——违背 G-Buffer 设计初衷

Forward Translucency 的优势

直接通过 Alpha Blending 按排序混合到 SceneColor

支持 OIT (Order Independent Transparency) 降低排序成本

通过 TranslucencyLightingVolume 采样 3D 光照信息

Separate Translucency 可低分辨率渲染后 Upsample

UE5.8 的半透明分类: ETranslucencyPass::Type 包括 TPT_StandardTranslucency、TPT_TranslucencyAfterDOF、TPT_TranslucencyAfterMotionBlur 等，支持在不同后处理阶段合成。

6. UE5.8 演进变化

6.1 Lumen 如何改写传统 Lighting Pass

传统 UE4 延迟光照

BasePass 直接光照 (Clustered) SSR SkyLight 带 Lightmap 的静态 GI

GI 方式 Lightmap 烘焙 (静态) 或 DFAO (动态但简陋)

UE5.8 Lumen 集成

UpdateLumenScene LumenSceneLighting DispatchAsyncLumenIndirectWork RenderDiffuseIndirectAndAO RenderLights Reflection

GI 方式 Lumen GI (Screen Probe / ReSTIR / HWRT) — 实时动态

Lumen 在 Render() 中的插入点:

  • BeginUpdateLumenSceneTasks (1966): 启动 Lumen 场景更新（异步 Task）
  • RenderLumenSceneLighting (3041): 计算 Lumen 场景的直接光照（异步 Compute 可选）
  • DispatchAsyncLumenIndirectLightingWork (3248): 异步分发 Lumen GI 计算
  • RenderDiffuseIndirectAndAmbientOcclusion (3429): 合成 Lumen GI 到 SceneColor
  • RenderLumenMiscVisualizations (4164): 调试可视化

6.2 MegaLights 的管线集成点

MegaLights 是 UE5.8 正式可用的大量动态光源系统，其管线集成完全在现有延迟光照流程中：

MegaLights 四阶段工作流

  1. CreateMegaLightsFrameTemporaries (3025): 创建帧临时资源（光源列表、Volume 纹理、采样缓冲区）
  2. DispatchAsyncMegaLightsVolumePasses (3053): 异步分发 Volume 穿透/光源注入 Pass
  3. GenerateMegaLightsSamples (3229/3279): 生成屏幕空间采样点（Async Compute 可选）
  4. RenderMegaLights (3470): 最终着色输出 ShadeBinning + Denoise + Resolve SceneColor

集成特点: MegaLights 采样在 VSM 页面标记之前（可基于采样结果定制 VSM 分配），在 RenderLights() 之后渲染（避免重复计算标准 Clustered 光源）。

6.3 Accumulation DOF — 后处理的变化

UE5.8 引入了 Accumulation DOF (累积景深) 作为传统 Diaphragm DOF 的替代方案：

  • 通过多帧累积实现高质量散景效果，而非单帧高频采样
  • 减少 DOF Pass 的单帧 GPU 成本，将计算分摊到多帧
  • 集成在 AddPostProcessingPasses() (PostProcessing.cpp:4311) 中，替换/增强原有 DiaphragmDOF 路径
  • 与 TSR 的时序超采样协同工作，共享运动矢量数据
