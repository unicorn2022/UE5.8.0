UE5.8.0 移动端渲染管线FMobileSceneRenderer / TBDR 适配 / Vulkan Mobile 优化

UE5 如何在手机上跑起来

移动端渲染的核心挑战在于极致的功耗、带宽和发热约束。UE5.8 的 FMobileSceneRenderer 完整实现了 ES3.1 特性级别的渲染管线，支持 Mobile Forward（前向渲染）和 Mobile Deferred（延迟渲染）两种模式，并深度针对 TBDR（Tile-Based Deferred Rendering） GPU 架构（如 Mali、Adreno、Apple GPU）做了 Subpass 级优化。

与桌面端 FDeferredShadingSceneRenderer 相比，移动端渲染器砍掉了 Nanite 虚拟几何、Lumen GI（仅 Lite）、体积云等大量高开销特性，但保留了 VSM 阴影、Substrate 材质框架、TSR 超分等核心能力。5.8 版本带来的 Mobile Lumen Lite 支持是移动端 GI 能力的里程碑事件。

2.3 Vulkan Mobile 的 TBDR 适配策略

TBDR (Tile-Based Deferred Rendering) 架构 GPU 通过在片上 Tile Memory 中完成多次渲染再写回内存, 大幅减少带宽消耗。UE5.8 的移动渲染器通过 Vulkan Subpass 机制显式利用这一特性:

LOAD_OP_CLEAR → Subpass链内 LOAD_OP_LOAD (Tile Memory保留 Depth) → STORE_OP_DONT_CARE (无后处理时丢弃) Color: TBDR 只在最终 Subpass 写回 System Memory; Memoryless RT 完全在 Tile Memory 中, 不分配显存 Mobile Deferred — GPU 端延迟着色 Subpass 链 Subpass 0 BasePass → GBuffer Write A/B/C (SceneColor) Subpass 1 GBuffer+Depth Input Read A/B/C + Depth Subpass 2 Deferred Lighting Read GBuffer+Depth → Write SceneColor Subpass 3 Fog + Translucency Blend to SceneColor 无 FBF (Framebuffer Fetch) 支持的设备回退: Multi-Pass 模式: BasePass → 显式 Resolve → Decals独立Pass → Lighting独立Pass → Translucency独立Pass, 每次都需要 Load/Store 往返带宽

图 2-1: Vulkan Mobile TBDR Subpass 链示意图 — Forward (上) 与 Deferred (下) 的 Subpass 组织

关键机制: Framebuffer Fetch & Pixel Local Storage
iOS Metal 通过 GSupportsShaderFramebufferFetch、Android GLES 通过 GSupportsPixelLocalStorage (PLS) 实现单 Pass 内读取当前像素的 Color/Depth。
RequiresMultiPass() (MobileShadingRenderer.cpp:3309) 精确判断是否必须走多 Pass 回退: Vulkan Subpass → 单Pass; iOS FBF → 单Pass; GLES PLS → 单Pass; 其余 → 多Pass。

4.4 TBDR 架构下避免 Pixel Shader 中的 RenderTarget 切换

传统即时模式渲染器 (IMR) 可以在单 Pass 内频繁切换 RenderTarget, 但 TBDR 架构下 RT 切换会触发 Tile Memory 刷新到 System Memory, 极大消耗带宽。UE5.8 的解决方案:

  1. Subpass 链式编排: 在一个 RenderPass 内通过 NextSubpass() 切换逻辑阶段, GPU 保持在同一个 Tile 中处理, 无需写回。
    MobileShadingRenderer.cpp:2254-2265 — Forward Single Pass 内的 Subpass 切换
    RenderMobileBasePass(RHICmdList, View, ...); // Subpass 0: 写入 SceneColor
PostRenderBasePass(RHICmdList, View);
RHICmdList.NextSubpass();                        // 切换 Subpass, 不刷新 Tile Memory
RenderDecals(RHICmdList, View, ...);              // Subpass 1: 读取 Depth, 写入 SceneColor
RenderModulatedShadowProjections(RHICmdList, ...);
RenderFog(RHICmdList, View);
RenderTranslucency(RHICmdList, View, ...);        // Subpass 1 继续: 半透明混合
  2. InputAttachment: Vulkan Deferred Subpass 中, Lighting Pass 通过 subpassLoad 读取 GBuffer, 而非绑定为 Texture。这利用 Tile Memory 的零延迟读取。
    VulkanRenderpass.h:694-738 — Deferred Subpass 2 的 InputAttachment 配置
    // Subpass 2: 从 Subpass 1 的 GBuffer 和 Depth 读取
InputAttachments2[0].attachment = DepthInputAttachment;   // Depth as Input0
InputAttachments2[2].attachment = ColorAttachmentReferences[1].attachment; // GBufferA
InputAttachments2[3].attachment = ColorAttachmentReferences[2].attachment; // GBufferB
InputAttachments2[4].attachment = ColorAttachmentReferences[3].attachment; // GBufferC
  3. Memoryless RenderTarget: bMemorylessMSAA (L745) 控制 MSAA 颜色目标完全在 Tile Memory 中, 不分配系统内存。条件: 单 Pass 渲染 + 非编辑器。
  4. Depth Store 优化: 无需后处理的 Depth 目标标记 STORE_OP_DONT_CARE, bKeepDepthContent (L696) 决定是否保留。

5. 底层实现关联

5.1 MobileShadingRenderer.cpp 主循环

FMobileSceneRenderer::Render() (L1181) 是整个移动端渲染的入口, ~800 行代码编排完整帧:

// MobileShadingRenderer.cpp:1181-1955 — Render() 主入口结构
void FMobileSceneRenderer::Render(FRDGBuilder& GraphBuilder, ...) {
    // 1. 帧前准备
    OnRenderBegin(GraphBuilder, SceneUpdateInputs);
    InitViews(GraphBuilder, SceneTexturesConfig, InstanceCullingManager, ...);

    // 2. 阴影 & 自定义深度
    RenderShadowDepthMaps(GraphBuilder, ...);
    RenderCustomDepthPass(GraphBuilder, ...);

    // 3. 深度预通道 (可选)
    if (bIsFullDepthPrepassEnabled) RenderFullDepthPrepass(...);

    // 4. AO & DFAO & Shadow Projections
    RenderAmbientOcclusion(...);
    RenderDistanceFieldLighting(...);
    RenderMobileShadowProjections(...);

    // 5. 主渲染 — 关键分支
    if (bDeferredShading) {
        if (bRequiresMultiPass) RenderDeferredMultiPass(...);
        else                   RenderDeferredSinglePass(...);
    } else {
        RenderForward(...);
    }

    // 6. 后处理 & 输出
    AddMobilePostProcessingPasses(...);
    CopySceneCaptureComponentToTarget(...);
    OnRenderFinish(...);
}

5.2 MobileBasePass 的简化 GBuffer 格式

移动端延迟渲染的 GBuffer 与桌面端有显著差异 (GetColorTargets_Deferred, L2545):

RT1 (GBufferA)PF_R8G8B8A8PF_R8G8B8A8 + CustomDataNormal.xy (Octahedron) + ShadingModelID + PerObjectData
RT2 (GBufferB)PF_R8G8B8A8PF_R8G8B8A8Specular + Roughness + Metallic + AO

移动端最多 4 个 Color Target + Depth, 远少于桌面端的 8+ 个, 显著减少 TBDR Tile Memory 压力。

6. UE5.8 演进变化

6.1 Mobile Lumen Lite 5.8 新特性

UE5.8 为移动端引入了 Lumen Lite——这是 Lumen GI 在高性能移动设备上的精简版本。与桌面端完整 Lumen 相比, Lumen Lite 裁剪了 Surface Cache 多层更新和 ReSTIR Gather, 仅保留核心的 Radiance Cache + Screen Probe 简化计算。

  • 启用条件: r.Lumen.DiffuseIndirect.Allow=1 + 设备支持 (高端 Adreno/Apple GPU)
  • CLAUDE.md 中列为 ✅ 新增 (5.8) 特性
  • 与 Mobile Deferred 渲染路径深度集成

6.2 Mobile VSM 进展

UE5.8 继续推进 Mobile VSM (Virtual Shadow Maps) 的适配工作。在 MobileShadingRenderer.cpp 中可以观察到 VSM 的初始化代码已完整保留 (L1310): 

// MobileShadingRenderer.cpp:1310
VirtualShadowMapArray.Initialize(GraphBuilder,
    Scene->GetVirtualShadowMapCache(),
    UseVirtualShadowMaps(ShaderPlatform, FeatureLevel),
    ViewFamily.EngineShowFlags);

当前 Mobile VSM 处于实验阶段, 主要通过 CSM 过渡, 未来版本将完整替换。

6.3 SceneColor 压缩优化

5.8 中强化了移动端 SceneColor 格式的自动选择逻辑。在不渲染到 SceneColor (即直接输出到 Swapchain) 时, 使用 PF_R8G8B8A8 或 PF_FloatR11G11B10 格式以节省带宽。手机 OLED 的 bGammaSpace 通道控制 LDR 输出路径。

6.4 Mobile TSR (Temporal Super Resolution)

Mobile TSR 是 UE5.8 向移动端引入的时序超分辨率技术, 对标桌面端 TSR 但做了显著的性能裁剪:

  • 与 Mobile TAA 共享历史帧缓冲 (TemporalAAHistory.RT[0])
  • MobileSSR.cpp:124 — 通过 View.AntiAliasingMethod 判断使用 TSR noise pattern
  • bKeepDepthContent = true 当 AA 为 AAM_TemporalAA 或 AAM_TSR
